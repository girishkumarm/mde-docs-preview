## Market Data Engine — Architecture Diagrams

This page contains all architecture diagrams for the Market Data Engine design document. Each diagram is stored as an `.excalidraw` file (open in [excalidraw.com](https://excalidraw.com) to view/edit) and also rendered as SVG below.

<Info>
**v3 diagrams are the current architecture.** The v2 diagrams at the bottom are kept for historical reference only.
</Info>

### Diagrams Index

| # | Diagram | What it shows |
| --- | --- | --- |
| 1 | **High-Level Architecture (v3)** | Full system: Merged Feed model, dual Redis clusters (Feed + API), Kafka-based broadcast, all data flows |
| 2 | Feed Handler Pipeline | Packet processing: UDP streams → Virtual Thread Receiver → Header Parse → Router → 6 Processors → LocalCache + Kafka + Batch HSET |
| 3 | Temporal Platform | Temporal Server → Dashboard → Worker Pool → Task Queues → Workflows + extensibility |
| 4 | Service Decommission Mapping | 12 current services → 5 new deployment groups |

### Decision Diagrams Index

| # | Decision | What it resolves |
| --- | --- | --- |
| D1 | Replace Redis PubSub with Kafka Pull | Why Broadcast Service uses Kafka consumer groups instead of Redis PubSub fan-out |
| D2 | Two Redis Clusters (Feed + API) | Separation of hot-path feed writes from high-volume API reads |
| D3 | Merged Feed Architecture | Both NSE Direct and Omnenest in same pipeline with cross-source dedup (replaces 85/15 split) |
| D4 | Kafka acks=all + Idempotent Producer | Strongest delivery guarantees for market data despite ~1ms latency |
| D5 | Expiry Day Cleanup Workflow | Temporal workflow for expired F&O contract cleanup |
| D6 | Scrip Master in PostgreSQL Only | Instrument master in PostgreSQL + ConcurrentHashMap, not Redis |

## Decision Diagrams (v3)

These diagrams document the key architectural decisions made in the Market Data Engine v3 design.

### Decision 1: Replace Redis PubSub with Kafka Pull

Why Broadcast Service uses Kafka consumer groups instead of Redis PubSub fan-out.

![mde-decision-1-pubsub-to-kafka.svg](/images/693502030/mde-decision-1-pubsub-to-kafka.svg)

*Source: [mde-decision-1-pubsub-to-kafka.excalidraw](/images/693502030/mde-decision-1-pubsub-to-kafka.excalidraw)*

### Decision 2: Two Redis Clusters (Feed + API)

Separation of hot-path feed writes from high-volume API reads into dedicated clusters.

![mde-decision-2-redis-split.svg](/images/693502030/mde-decision-2-redis-split.svg)

*Source: [mde-decision-2-redis-split.excalidraw](/images/693502030/mde-decision-2-redis-split.excalidraw)*

### Decision 3: Merged Feed Architecture

Why both NSE Direct and Omnenest feed into the same pipeline with cross-source dedup instead of 85/15 traffic split.

![mde-decision-3-feed-merge.svg](/images/693502030/mde-decision-3-feed-merge.svg)

*Source: [mde-decision-3-feed-merge.excalidraw](/images/693502030/mde-decision-3-feed-merge.excalidraw)*

### Decision 4: Kafka acks=all + Idempotent Producer

Why market data requires strongest delivery guarantees despite ~1ms latency increase.

![mde-decision-4-kafka-acks.svg](/images/693502030/mde-decision-4-kafka-acks.svg)
