### 6.12 FNO Data Writer

**What it does:** Consumes F&O tick data from Kafka, computes option analytics + Greeks, stores in Redis Cluster 2. Handles **all F&O instrument types** — futures AND options.

**Input:** Kafka topics (`pml-marketfeed-nse-derivative-ticks`, `pml-marketfeed-bse-derivative-ticks`) **Output:** Redis Cluster 2 (option chain, option insights, futures data)

**FNO data Redis schema (matching current):**

```
Fields per instrument:
    pmlId           Long      Unique record identifier
    exchange        String    Exchange code (NSE, BSE)
    segment         String    Market segment
    securityId      Integer   Security identifier
    name            String    Security name
    instrument      String    Instrument type (FUTSTK, FUTIDX, OPTSTK, OPTIDX)
    symbol          String    Trading symbol
    lotSize         Integer   Contract lot size
    expiryDate      String    Contract expiry date
    strikePrice     Double    Option strike price (NULL for futures)
    optionType      String    CE / PE (NULL for futures)
    underlyingScripCode  Long    Underlying security code
    underlyingSymbol     String    Underlying trading symbol
    ltp             Float     Last traded price
    volume          Long      Traded volume
    oi              Long      Open interest
    ltt             Long      Last trade time (epoch ms)
    tradedContracts Long      Number of contracts traded
    pClose          Float     Previous day close price
    pOi             Long      Previous day open interest
    netChgPrc       Double    Net change in price
    percChgPrc      Double    % change in price
    netChgVol       Long      Net change in volume
    percChgVol      Double    % change in volume
    netChgOi        Long      Net change in open interest
    percChgOi       Double    % change in open interest
    percChgIv       Double    % change in implied volatility
    spotPrice       Float     Underlying spot price
    delta           Double    Delta — price sensitivity
    gamma           BigDecimal  Gamma — delta sensitivity
    vega            Double    Vega — volatility sensitivity
    theta           Double    Theta — time decay
    iv              Double    Implied volatility
    feedReceiveTime Long      Server-side feed receive timestamp
    metadata        Map<String,Object>  Additional key-value pairs
```

**Option Insights Redis schema:**

```
Fields:
    exchange        String    Exchange code
    symbol          String    Trading symbol (underlying)
    instrumentType  String    Instrument type
    expiryDate      Date      Contract expiry date (IST)
    spotPrice       Float     Underlying spot price
    atmStraddle     Double    ATM straddle value
    pcrOI           Double    Put-Call ratio (open interest)
    pcrVolume       Double    Put-Call ratio (volume)
    cprOI           Double    Call-Put ratio (open interest)
    cprVolume       Double    Call-Put ratio (volume)
    maxPain         Double    Max pain strike price
    startTime       Date      Candle start time (IST)
```

**Computed fields & recomputation frequency:**

| Field | Formula | Frequency |
| --- | --- | --- |
| Greeks (Delta, Gamma, Theta, Vega) | Black-Scholes | ATM ±5: every tick. Rest: every 5–30s |
| Implied Volatility (IV) | Black-Scholes inverse | Same as Greeks |
| PCR (OI) | Total Put OI / Total Call OI | Every 5 seconds |
| PCR (Volume) | Total Put Vol / Total Call Vol | Every 5 seconds |
| CPR (OI/Volume) | Inverse of PCR | Every 5 seconds |
| Max Pain | Strike where total buyer loss is maximum | Every 30 seconds |
| ATM Straddle | ATM Call LTP + ATM Put LTP | On every underlying tick |
| Net/% Change fields | Compare with previous close/OI | On every tick |

    Score: ranking metric
    Member: instrumentToken

Query params: movers category + exchange + index
Recomputation: every 5 seconds during market hours
```

### 6.17 Self-Healing Supervisor

**What it does:** Continuously monitors every component and auto-recovers from failures.

**Health checks — differentiated by instrument type:**

| Instrument Type | Stale Threshold | Rationale |
| --- | --- | --- |
| Liquid equity (NIFTY 50 stocks) | No tick for 5 minutes | These trade constantly |
| Mid/small cap equity | No tick for 30 minutes | Lower trading frequency |
| Illiquid scrips | No tick for 2 hours | May not trade at all some days |
| Index instruments | No tick for 30 seconds | Indices update very frequently |
| Active F&O contracts | No tick for 10 minutes | ATM options are heavily traded |
| Deep OTM options | No tick for 2 hours | Very few trades |

<Warning>

### 6.19 Expiry Day Cleanup

**What it does:** Cleans up expired derivative contracts from Redis and recalculates dependent metrics after expiry settlement.

**Temporal Workflow: ExpiryCleanupWorkflow**

```
Trigger: Market close status on expiry days
    - Weekly: Thursday (NIFTY, BANKNIFTY, FINNIFTY weekly options)
    - Monthly last Thursday: stock options + monthly futures
    - Quarterly: index futures (Mar, Jun, Sep, Dec)

Timeout: 5 minutes
Expected completion: < 30 seconds

Step 1: Identify expiring contracts from PostgreSQL
    → SELECT * FROM instruments
      WHERE expiry_date = CURRENT_DATE
        AND scrip_type IN ('OPTION', 'FUTURE')
        AND status = 'ACTIVE'
    → Typically 2,000–5,000 contracts on weekly expiry
    → Up to 20,000 contracts on monthly expiry (all stock options)

Step 2: Mark as EXPIRED in PostgreSQL
    → UPDATE instruments SET status = 'EXPIRED' WHERE id IN (...)
    → Prevents Feed Handler from processing stale ticks for these instruments

Step 3: Clean expired data from Redis Cluster 2
    → Remove expired instruments from FNO option chain hashes
    → Remove expired instruments from Market Movers sorted sets
    → Remove expired instruments from option insights hashes
    → HDEL / ZREM operations, batched

Step 4: Stop FNO Data Writer from computing Greeks for expired contracts
    → Publish expiry event to Kafka (topic: instrument-expiry-events)
    → FNO Data Writer removes expired instruments from its in-memory computation set
    → No more Black-Scholes calculations for expired strikes

Step 5: Recalculate PCR / Max Pain using only live contracts
    → Trigger FNO Data Writer to recompute option insights for affected underlyings
    → PCR (OI/Volume) recalculated excluding expired contracts
    → Max Pain recalculated with remaining strikes only
    → ATM Straddle updated if ATM strike was an expiring contract

Step 6: Propagate settlement prices to Portfolio/Ledger Engine via Kafka
    → Publish settlement event: {instrumentToken, settlementPrice, expiryDate}
    → Topic: instrument-settlement-events
    → Portfolio Engine uses this for P&L calculation
    → Ledger Engine uses this for margin release
```

**Applies to:**

- Weekly NIFTY/BANKNIFTY/FINNIFTY option expiry (every Thursday)
- Monthly stock option expiry (last Thursday of month)
- Monthly futures expiry (last Thursday of month)
- Quarterly index futures expiry (last Thursday of Mar/Jun/Sep/Dec)

### 6.20 Resilience & Zero-Error User Experience

<Info>
**New in v3.2.** This section codifies the platform-wide promise that **users should NEVER see an error**. If a backend component fails, the system degrades silently. The user always sees data — possibly slightly stale, possibly partial — never an error popup, never a red modal, never a "Something went wrong" screen.
</Info>

**Design Principle:** "Users should NEVER see an error. If a backend component fails, the system degrades silently."

This is not a soft target. Every Charts API call, every WebSocket subscription, every Market Movers request, every Option Chain query must produce a useful response within the latency budget — even when the upstream is down, slow, or returning garbage. The mechanisms below describe how each request path enforces this contract.

#### 6.20.1 Layered Fallback Strategy

For every user-facing data path, the response is produced by walking down a 4-layer fallback ladder. The first layer that returns data wins. The user always gets *something*.

| Layer | Source | Freshness | When Used |
| --- | --- | --- | --- |
| **Layer 1 (primary)** | Live data from Kafka / Redis Cluster 1 (Feed) | Real-time (<100ms old) | Normal operation |
| **Layer 2 (cached)** | Last-known-good snapshot from Redis Cluster 2 (API) | Seconds-old | Layer 1 timeout, partial outage |
| **Layer 3 (stale)** | Most recent snapshot from TimescaleDB | Minutes/hours-old, marked "Last updated: Xs ago" | Layer 2 unavailable, cluster outage |
| **Layer 4 (final)** | Empty state with skeleton/loading UI | None | Total backend outage — no error popup, just placeholder |

**How it works in practice:**

- Each API endpoint declares its fallback chain at startup.
- The fallback runtime wraps every downstream call with a deadline (e.g., 30 ms for Layer 1).
- If Layer 1 misses the deadline OR throws, the runtime tries Layer 2 with a fresh deadline.
- If Layer 2 fails, Layer 3 is consulted.
- If Layer 3 also fails, Layer 4 returns an empty-but-structurally-valid response — never a 5xx, never an exception.
- Each downgrade is recorded in the response metadata (`source: "L2_CACHED"`, `staleAgeSec: 12`) so the frontend can render a freshness indicator.

#### 6.20.2 Stale-While-Revalidate Pattern

APIs always return SOMETHING useful within 50ms. They never block on a slow upstream.

- **Synchronous path:** Serve the most recent cached value immediately. Tag the response with `stale: true` if the cache is older than the per-API freshness threshold.
- **Background refresh:** A non-blocking background task fetches the latest from the upstream. When it arrives, the cache is updated. The next request gets fresh data.
- **Result:** The user always sees data immediately. Staleness is exposed as metadata (e.g., "Updated 8s ago") — not as an error.

```
Request arrives at Charts API
    ↓
Check Redis Cluster 2 cache (current-day candles)
    ↓
Cache hit (any age) → respond IMMEDIATELY with cached value
                      ↓
                      If cache age > 5s → kick off background refresh
                                          (non-blocking — does not delay this response)
    ↓
Cache miss → try TimescaleDB synchronously (Layer 3)
           → respond with TimescaleDB data + stale flag
           → kick off background refresh to repopulate cache
```

This pattern guarantees that **the response to the user never waits for an upstream that may be slow or down**. When upstream recovers, the next request silently gets fresh data.

#### 6.20.3 Circuit Breakers (Resilience4j)

Every downstream call is wrapped with a Resilience4j circuit breaker. When too many calls fail or time out, the circuit opens and subsequent calls are short-circuited to the fallback path — protecting the failing service and the calling service alike.

**Circuit-breaker behavior:**

- **CLOSED** (normal): Calls pass through. Failures are tracked in a sliding window.
- **OPEN** (failing): Calls short-circuit immediately to fallback. The downstream is not hit at all.
- **HALF-OPEN** (probing): After cooldown, a small number of test requests are allowed through. If they succeed, circuit returns to CLOSED. If they fail, it returns to OPEN with extended cooldown.

**Per-service thresholds:**

- **Failure rate threshold:** 50% over a rolling window of 20 requests → trip OPEN
- **Cooldown ladder (exponential):** First open → 10s, second open in same hour → 30s, third → 60s
- **Half-open probe count:** 3 requests allowed through during HALF-OPEN

**Wrapped paths (examples):**

- Charts API → Redis Cluster 2 (current-day candles)
- Charts API → TimescaleDB (historical candles)
- FNO Dashboard API → Redis Cluster 2 (option chain)
- Market Movers API → Redis Cluster 2 (sorted sets)
- Broadcast Service → Redis Cluster 1 (snapshot on connect)
- Charts Data Writer → TimescaleDB (writes — circuit-breaker prevents WAL buildup if DB is slow)
- FNO Data Writer → Redis Cluster 2 (writes)
- Any cross-service HTTP call

**Why this matters:** Without circuit breakers, a slow downstream creates a cascading failure — every caller blocks on it, threads exhaust, the calling service itself becomes unavailable. The breaker keeps the calling service responsive even when downstream is in trouble.

#### 6.20.4 Silent Feature Flags

When a backend component is degraded or down, the affected UI widgets are **hidden silently** — not replaced with an error banner, not greyed out with a sad face, just... gone. The user perceives a slightly simpler page, not a broken one.

**Feature-flag triggers (auto-managed by Incident Manager — §6.21):**

| Component Down | Flag Toggled | UI Effect |
| --- | --- | --- |
| FNO Data Writer down | `fno.dashboard.greeks.enabled = false` | Greeks columns hidden from option chain. Rest of chain still visible. |
| Market Movers Writer down | `markets.movers.live.enabled = false` | Market Movers tab shows EOD list with "Updated at market close" caption. No error. |
| Charts Data Writer behind on writes | `charts.intraday.enabled = false` | Today's candle hidden; historical chart still renders. |
| Option Insights computation stalled | `fno.dashboard.insights.enabled = false` | PCR/Max Pain widgets hidden. Option chain unaffected. |
| Alert Engine down | `alerts.creation.disabled = true` | "Create Alert" button hidden. Existing alerts unaffected. |

**Key properties:**

- **Auto-triggered by Incident Manager** — when anomaly detection flags a component, the feature flag flips automatically. No human in the loop.
- **Auto-restored** — when the component recovers, the flag flips back. No manual ops action.
- **Silent at the UI layer** — frontend treats `flag = false` as "render nothing for this widget". No banner, no popup, no toast.
- **Self-hosted feature-flag service** (Unleash) — no external dependency on critical path.

#### 6.20.5 Request Hedging

For latency-critical reads (live price, option chain), the API sends parallel requests to two Redis replicas and returns whichever responds first. This eliminates long-tail latency caused by occasional GC pauses or replica hiccups.

**How it works:**

- The Redis client is configured with two read endpoints (replica A, replica B).
- For each request, both replicas receive the call simultaneously.
- The first valid response wins; the other is discarded.
- Client-side latency is `min(latency_A, latency_B)` — almost always faster than waiting on a single replica.
- Outbound load is doubled, but reads are cheap and Redis is provisioned with headroom.

**Where applied:**

- WebSocket snapshot on connect (Broadcast Service → Redis Cluster 1)
- Charts API current-day candle reads (Charts API → Redis Cluster 2)
- Option Chain reads (FNO Dashboard API → Redis Cluster 2)
- Market Movers reads (Market Movers API → Redis Cluster 2)

**Where NOT applied:**

- Writes (would double-write and cause consistency problems)
- Expensive aggregation reads (would double the work for marginal benefit)

#### 6.20.6 Client-Side Resilience

The frontend (mobile + web) is built to assume backend instability and recover transparently.

- **WebSocket auto-reconnect** with exponential backoff: 1s → 2s → 4s → 8s → 16s → 30s (capped). On reconnect, fetch fresh snapshot from REST endpoint, then resume stream.
- **Mobile app caches last known state on close** — when user reopens the app, the last-seen prices render instantly from local storage; live updates layer in as the WebSocket connects.
- **Progressive data loading** — LTP first (smallest payload, most important), market depth second, Greeks last. If Greeks never arrive, the option chain still renders with LTP and depth.
- **Skeleton loading states** — never a spinner that says "Loading..." indefinitely. After 3 seconds, render skeleton placeholder rows; data fills in as it arrives.
- **No error modals on transient failures** — only on user-action errors (e.g., "Symbol not found"). Backend issues never produce modals.

#### 6.20.7 Per-API Fallback Table

The mandatory fallback chain for each user-facing API:

| API | Primary (Layer 1) | Layer 2 | Layer 3 | Final (Layer 4) |
| --- | --- | --- | --- | --- |
| **Charts API** | Redis Cluster 2 (current-day candles) | TimescaleDB (`candles_1m`) | Previous day's candles from TimescaleDB | Empty chart with "Data updating" message |
| **FNO Dashboard API** | Redis Cluster 2 (live option chain) | Last successful snapshot (in-memory cache) | Previous close values + lot size from Instrument Master PG | Hide widget silently via feature flag |
| **Market Movers API** | Redis Cluster 2 (sorted sets, refreshed every 5s) | Last 5-minute snapshot (in-memory cache) | Last 30-minute snapshot from TimescaleDB | Static "trending today" list from EOD |
| **WebSocket LTP stream** | Kafka live tick (via Broadcast Service) | Redis Cluster 1 snapshot | TimescaleDB last tick (1m candle close) | Mark as stale, freeze price, show "Last updated: Xs ago" |
| **Instrument Search** | PostgreSQL (instruments table) | Local ConcurrentHashMap (in-memory) | Stale list (last sync) | Common tickers from cache (e.g., NIFTY 50 + top 100) |
| **Option Insights** | Redis Cluster 2 (PCR/Max Pain) | Last successful computation (in-memory) | EOD insights from TimescaleDB | Hide widget silently |
| **Periscope live price** | Local in-process cache | Redis Cluster 1 | TimescaleDB last tick | Return previous close from Instrument Master PG |

#### 6.20.8 "Never Show Error" Enforcement

This is the operational rule that makes the design real:

- **All error responses converted to 200 OK.** A failed downstream lookup returns:

```json { "data": null, "stale": true, "source": "FALLBACK_L4", "lastUpdatedMs": 1713168900000, "message": "Updating" } ``` Not a 5xx, not a 4xx for backend issues. The frontend treats `null` data as a loading state, never as an error.

- **HTTP 5xx errors are logged + sent to Incident Manager but NEVER surfaced to the client.** The API gateway has a global error handler that catches any uncaught exception and converts it to the structure above.
- **Frontend rules:**
- `null` data → render skeleton or last-known-good
- `stale: true` → render data with subtle freshness indicator (timestamp footer, no banner)
- HTTP failure → silent retry with backoff, no UI change unless data has been missing for >10s, then show subtle "Reconnecting" footer
- **No exceptions for "edge cases."** Even on cold-start, even during rolling deploys, even during a Redis Cluster failover — the contract holds.

**Audit:** Every Layer 2/3/4 hit is recorded in a `fallback_invocations` Prometheus counter, labeled by API + layer. If any API hits Layer 4 more than 0.1% of requests in a 5-minute window, an Incident Manager alert fires.

### 6.21 AI-Powered Incident Manager

<Info>
**New in v3.2.** **Platform-wide capability — not just Market Data Engine internal tooling.** Detects incidents BEFORE users notice. Uses an LLM to root-cause and suggest fixes. Auto-remediates known patterns. Integrates with Slack, Jira, PagerDuty, and Kubernetes.
</Info>

**Purpose:** Today, most incidents are first reported by users — embarrassing and slow. With Incident Manager, the platform detects anomalies in seconds, gathers context-rich evidence, asks an LLM to root-cause it, and either auto-remediates (if the fix is reversible and high-confidence) or pages a human with a fully-prepared analysis. Mean time to recovery (MTTR) drops from hours to minutes.

#### 6.21.2 Detection Signals

The Incident Manager continuously ingests signals from every service in the platform. Detection is not based on a single metric crossing a threshold — it uses statistical baselines and cross-signal correlation to reduce false positives.

| Signal Source | Examples | Threshold |
| --- | --- | --- |
| **Prometheus** | p99 latency, error rate, queue depth, GC pause frequency, JVM heap | Z-score > 3 sustained for 60 seconds |
| **Application Logs** | ERROR-level log spike, new stack traces, OutOfMemoryError | >10x baseline frequency in 1 minute, OR appearance of any never-before-seen stack trace |
| **Kafka** | Consumer lag growth | >5 seconds sustained for 30 seconds OR >2x normal lag for 2 minutes |
| **Redis** | Slow query log entries, evicted keys, replica lag | >1 slow query per 10 seconds, OR any noeviction-induced write failure |
| **Feed Health** | Heartbeat missing, sequence gaps | >3 seconds of silence on any UDP stream, OR sequence gap > 100 with no fill |
| **Synthetic Probes** | Charts API health endpoint, Option Chain API health endpoint | 3 consecutive failures within 60 seconds |
| **WebSocket** | Disconnect rate spike, snapshot fetch latency | Disconnect rate > 5x baseline OR snapshot fetch p99 > 200 ms |
| **TimescaleDB** | Continuous aggregate refresh lag, write WAL bloat | cagg refresh lag > 10 minutes, OR WAL > 5 GB |

**Correlation engine:** Anomalies that cluster in time and space (same service, same datacenter, same dependency chain) are merged into a single incident. Example: if Charts Data Writer's CPU spikes AND its Kafka consumer lag grows AND TimescaleDB ingest latency rises — that's ONE incident, not three. The correlation prevents alert fatigue.

#### 6.21.3 LLM-Powered Debugging Workflow

When an incident is detected and confirmed by the correlation engine, the LLM debugging workflow runs:

```
1. INCIDENT DETECTED (anomaly + correlation)
       │
       ▼
2. SIGNAL COLLECTOR snapshots:
       - Last 5 minutes of metrics from affected services
       - Last 5 minutes of ERROR-level logs
       - Distributed traces (Tempo/Jaeger) for failing requests
       - Recent deploys (Argo CD / GitHub Actions history)
       - Service topology (which services depend on which)
       │
       ▼
3. CONTEXT ASSEMBLY:
       - Known-good baselines for each metric
       - Recent code changes (git diff for last 3 deploys)
       - Runbook entries (from confluence) for similar past incidents
       │
       ▼
4. LLM ANALYSIS (Claude Sonnet 4.5):
       - Root-cause hypothesis (top 3, ranked by confidence)
       - Evidence supporting each hypothesis
       - Suggested fix steps for each hypothesis
       - Risk assessment for each fix (reversible? data-impacting?)
       - Relevant runbook references
       │
       ▼
5. POST TO SLACK CHANNEL #incident-{id}:
       - Summary (1-line)
       - Top hypothesis with evidence
       - Suggested fix
       - "Approve & execute" / "Investigate further" buttons
       │
       ▼
6. AUTO-EXECUTE OR HUMAN-IN-LOOP:
       If LLM confidence > 90% AND fix is in the auto-remediation allowlist:
           → Execute fix (e.g., kubectl rollout restart)
           → Continue monitoring; if metrics recover within 5 min, close incident
           → If metrics don't recover, escalate to human
       Else:
           → Wait for human approval in Slack
           → Execute approved fix on click
       │
       ▼
7. OUTCOME LOGGING:
       - Whether the LLM's hypothesis was correct
       - Whether the fix worked
       - Time-to-detection, time-to-mitigation
       - Feeds back into the model evaluation pipeline
```

#### 6.21.4 Example LLM Prompt and Response

**Example incident — Charts Data Writer OOM:**

```
INCIDENT #1234
Service: Charts Data Writer
Symptoms: Consumer lag climbing (30s → 90s in 3 min)
Metrics: CPU normal, memory climbing, GC pause frequency up

If the LLM's confidence is above the auto-execute threshold AND the fix (`kubectl rollout undo`) is in the auto-remediation allowlist, the rollback is executed immediately; the incident channel is updated with execution result.

#### 6.21.5 Auto-Remediation Rules

Auto-remediation only fires for actions that are (a) reversible, (b) well-understood, and (c) inside an explicit allowlist. Anything destructive or ambiguous goes to humans.

| Trigger | Auto-Remediation | Confirmation Required? |
| --- | --- | --- |
| Pod memory or CPU over threshold for 2 min | `kubectl rollout restart` (rolling restart) | No (reversible) |
| Service container OOMKilled | `kubectl rollout undo` to previous version | No (reversible) |
| Circuit breaker stuck OPEN > 5 min | Manual escalation only | Yes |
| Feed silence > 5 seconds | Trigger source failover (NSE→Omnenest reroute logged) | No (already automatic at feed layer) |
| Kafka consumer lag > 2x threshold for 5 min | Scale up consumer pods (`kubectl scale --replicas=+2`) | No (reversible) |
| Redis slow queries detected | Log + alert, no auto-action | Yes (humans investigate) |
| Component down → Affected widgets | Toggle feature flag to hide widget | No (reversible) |
| Component recovered → Affected widgets | Toggle feature flag to re-enable | No (reversible) |
| Continuous aggregate refresh lag > 10 min | Manually trigger refresh (`CALL refresh_continuous_aggregate(...)`) | No (idempotent) |

**Safety rails:**

- No remediation may be triggered for the same service more than 3 times in 30 minutes (prevents flap loops).
- Every auto-remediation is logged to an immutable audit table (`incident_actions`).
- All auto-remediations post to `#incidents-audit` Slack channel for human review post-hoc.
- Database writes, deletes, schema changes are NEVER auto-remediated.

#### 6.21.6 Integration Points

The Incident Manager is glue. It depends on existing observability + ops infrastructure:

| System | Direction | Purpose |
| --- | --- | --- |
| Prometheus AlertManager | Incident Manager subscribes | Receive metric-based alerts |
| Loki / Elasticsearch | Incident Manager queries | Pull logs for context assembly |
| Tempo / Jaeger | Incident Manager queries | Pull traces for failing requests |
| Slack webhooks | Incident Manager posts | Per-severity channel routing (`#incidents-p0`, `#incidents-p1`) |
| Jira API | Incident Manager creates tickets | Auto-create with LLM analysis as description; auto-close on metric recovery |
| PagerDuty | Incident Manager pages | P0 only |
| Kubernetes API | Incident Manager executes | Auto-restart, auto-scale, rollout undo |
| GitHub / Bitbucket | Incident Manager reads | Recent commits + diffs for LLM context |
| Unleash (feature flags) | Incident Manager toggles | Silent feature degradation per §6.20 |
| Argo CD | Incident Manager reads | Recent deploy history for LLM context |

#### 6.21.7 Severity Classification

| Severity | Definition | Examples | Notification |
| --- | --- | --- | --- |
| **P0** | User-impacting now or imminently | Both feeds down; Charts API completely failing; WebSocket pods crashlooping | PagerDuty page + Slack `#incidents-p0` + LLM analysis |
| **P1** | Single component degraded; no user impact yet (fallback active) | NSE Direct down (Omnenest sole source); Continuous aggregate refresh lag; FNO Writer slow | Slack `#incidents-p1` + auto-create Jira |
| **P2** | Operational concern, no functional impact | Redis memory at 75%; Kafka topic close to retention boundary; Prometheus scrape latency rising | Jira ticket only |

#### 6.21.8 Why Build This?

**Today's reality (without Incident Manager):**

- Issues are usually first reported by users → embarrassing, slow to respond
- Operators page each other; spend 10–30 minutes gathering context (logs, metrics, traces, recent deploys)
- The same incidents recur because runbook lookup is manual

**With Incident Manager:**

- Detected in seconds via correlation across signals
- Context assembled automatically; LLM produces analysis before a human even opens Slack
- Auto-remediation handles 80% of "obvious" incidents (OOMs, restart loops, feature-flag toggles for component failures) without human intervention
- Humans focus only on novel, complex incidents
- MTTR drops from hours to minutes
- P0s that reach users drop dramatically — most are caught + remediated before user impact

**Build this from Day 1, not as a post-launch addon.** A platform of this size and complexity cannot be operated at the required reliability without it.
