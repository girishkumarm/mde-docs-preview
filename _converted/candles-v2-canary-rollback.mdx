## 3. Canary plan

### Day -1 — Staging warm-up

- Deploy `candle-writer` to staging with both sinks enabled.
- Leave `charts-api` on `MDE_CHARTS_API_DEFAULT_SOURCE=redis` (no read traffic hits ClickHouse yet).
- Observe for **24 hours** continuously during market hours + one overnight window.

**Exit checks (must all pass before Day 0):**

- Redis: `SCAN 0 MATCH CANDLE:RUN:* COUNT 1000` returns one key per active token per interval; key TTLs align with the configured routing windows below.
- ClickHouse: `SELECT toStartOfMinute(now()), count() FROM candles_1m WHERE bucket_start > now() - INTERVAL 10 MINUTE GROUP BY 1` — row-growth rate ≥ expected (≈ active-token count per minute during market hours).
- Grafana board `05-charts-pipeline` shows no red panels for 24h.
- Zero firing alerts in `mde.kafka`, `mde.redis`, `mde.service.availability` groups.

### Day 0 — Single-pod canary

1. **Fleet default stays safe:** confirm `MDE_CHARTS_API_DEFAULT_SOURCE=redis` is set on the `charts-api` Deployment (env at the pod-spec level, not in ConfigMap, so it overrides the container default of `auto`).
2. **Canary pod:** patch one pod (label it `canary=true`) to `MDE_CHARTS_API_DEFAULT_SOURCE=auto`. This is the only pod that will route reads to ClickHouse when the request falls outside the Redis window.
3. Watch the success gates at **30 min**, **2h**, and **24h** checkpoints.

**Redis routing windows (for reference — applied only on the canary pod):**

| Interval | Redis window | Older requests → |
| --- | --- | --- |
| 1m | 3h | ClickHouse `candles_1m` |
| 5m | 6h | ClickHouse `candles_5m` |
| 15m | 1d | ClickHouse `candles_15m` |
| 1h | 7d | ClickHouse `candles_1h` |

### Success gates (at each checkpoint)

| Metric | Threshold |
| --- | --- |
| p99 read latency — Redis path (per interval) | ≤ 50 ms |
| p99 read latency — ClickHouse path (per interval) | ≤ 300 ms |
| 5xx rate on `/v1/candles` (canary pod) | < 0.1 % |
| ClickHouse connection-pool utilisation | < 70 % of `CLICKHOUSE_POOL_MAX` |
| `ClickHouseQueryFailed` alerts firing | 0 |
| Data parity (sampled): canary-pod response equals Redis-only baseline for requests inside the Redis window | 100 % match |

If **any** gate fails at any checkpoint, execute **Tier 1 rollback** immediately. Do not wait for the next checkpoint.

### Ramp

Each ramp step requires **all success gates green for ≥ 1 hour** before proceeding.

| Step | Fleet on `auto` | Minimum soak |
| --- | --- | --- |
| 1 | 1 pod (canary) | 24 h |
| 2 | 10 % of fleet | 1 h |
| 3 | 50 % of fleet | 1 h |
| 4 | 100 % of fleet (flip the Deployment default, remove the `redis` override) | — |

After Step 4 holds green for 24h, merge `chore/drop-legacy-candle-writers` to decommission `charts-writer` and `charts-historical-writer`.

## 4. Rollback plan

Three tiers, ordered by speed and blast-radius. Always start at Tier 1; escalate only if the prior tier does not restore SLOs.

### Tier 1 — Seconds (flip one env var)

- On every `charts-api` pod currently running `MDE_CHARTS_API_DEFAULT_SOURCE=auto`, set it back to `redis`:

```
kubectl set env deployment/charts-api MDE_CHARTS_API_DEFAULT_SOURCE=redis -n mde
kubectl rollout restart deployment/charts-api -n mde
```
- Rolling restart takes the standard pod-disruption-budget time (< 60 s for a single-pod canary; a few minutes for a full fleet).
- Outcome: all read traffic returns to Redis-only; ClickHouse reads stop. `candle-writer` continues to write to both sinks (safe — ReplacingMergeTree is idempotent).

### Tier 2 — Minutes (roll back the API image)

- Use this if Tier 1 does not restore SLOs (e.g. routing bug that ignores the env var, or a charts-api code regression).
- In ArgoCD, roll `charts-api` back to the commit immediately **before** `35560c0` (i.e. the Deployment manifest pinned to the image built from `a9863de`).
- This removes the entire `MDE_CHARTS_API_DEFAULT_SOURCE` wiring; the service reverts to its pre-env-switch behaviour (Redis-only by construction).
- Outcome: same as Tier 1 at the read layer, plus any regression shipped in `35560c0` is removed.

### Tier 3 — Hour (drop the new writer)

- Use this only if the new dual-sink write path itself is producing bad data (e.g. incorrect OHLCV aggregation or corrupt Redis keys).
- Scale `candle-writer` to 0 replicas in ArgoCD.
- Revert the staged cleanup by re-applying the old `charts-writer` + `charts-historical-writer` manifests. The legacy manifests are preserved on the tip of `main`; they are only *removed* by `chore/drop-legacy-candle-writers`, which has not been merged yet — so re-enabling is a standard ArgoCD sync of the prior Deployment YAML.
- Confirm `charts-writer` / `charts-historical-writer` resume Kafka consumption (check lag on `mde-candles-1m`).
- Outcome: write path returns to legacy; read path must already be on Tier 1 or Tier 2.

## 5. What rollback does NOT cover

- **ClickHouse rows written during the canary stay.** `candles_1m` is `ReplacingMergeTree` — duplicate `(token, bucket_start)` writes during a later cutover are deduplicated on merge. No cleanup required.
- **Materialized-view rollups (**`candles_5m/15m/1h/1d`) also stay. They are derived from `candles_1m` and recompute correctly on further writes.
- **Redis running-candle state can be left to expire naturally.** Each `CANDLE:RUN:{interval}:{token}` key has a TTL bounded by the next bucket boundary; letting them expire is safer than a flush.
- **Manual Redis flush is available but discouraged.** `DEL CANDLE:*` (or `SCAN + UNLINK` in production to avoid a blocking operation) will drop every running candle key, including active ones. Only do this if a corrupted Redis state is actively causing incorrect reads; accept the brief gap while keys repopulate from the next tick.

Rollback does **not** revert ClickHouse schema migrations (V2_1, V2_2, V3). Schema is forward-compatible with the legacy writer path.

## 8. Appendix — env-var quick reference

All values are sourced from Kubernetes Deployment env or ConfigMap. No secrets are listed here; see the parent checklist for secret-handling policy.

| Env var | Values / range | Default | Purpose |
| --- | --- | --- | --- |
| `MDE_CHARTS_API_DEFAULT_SOURCE` | `auto` / `redis` / `clickhouse` | `auto` (container default) | Canary knob. Prod default should be `redis` until Step 4 of the ramp. |
| `MDE_CHARTS_API_AUTO_REDIS_WINDOW_1M` | Duration (e.g. `3h`) | `3h` | Requests within this window served from Redis for the `1m` interval. |
| `MDE_CHARTS_API_AUTO_REDIS_WINDOW_5M` | Duration | `6h` | Same, for `5m`. |
| `MDE_CHARTS_API_AUTO_REDIS_WINDOW_15M` | Duration | `1d` | Same, for `15m`. |
| `MDE_CHARTS_API_AUTO_REDIS_WINDOW_1H` | Duration | `7d` | Same, for `1h`. |
| `CLICKHOUSE_SOCKET_TIMEOUT_MS` | Integer ms | (service default) | Per-query socket timeout for the JDBC driver. |
| `CLICKHOUSE_QUERY_TIMEOUT_SECONDS` | Integer s | (service default) | Server-side query timeout. |
| `CLICKHOUSE_POOL_MAX` | Integer | (service default) | Max JDBC connections. Utilisation monitored in success gates. |

Override order during canary: pod-spec env overrides Deployment env overrides ConfigMap overrides container default.
