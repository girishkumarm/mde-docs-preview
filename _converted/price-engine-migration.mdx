## 1. Executive Summary

The Price Engine migration replaces **25 legacy PML services** spread across 118 repositories with the **Market Data Engine (MDE)** — a unified, 14-service Gradle monorepo built on Java 21, Spring Boot 3.2, Kafka, Redis, TimescaleDB/ClickHouse, and Temporal.

**Current state:** All 14 MDE stories (Epic PM3-140083) are code-complete with 200+ passing tests. The system has been proven end-to-end in local/integration environments.

**Migration approach:** Strangler fig pattern — 4 phases over 7 weeks, zero-downtime, with feature-flag-controlled rollout.

**Additionally:** This document plans 4 new features not in the original scope: Market Depth, Peer Groups, User Price Alerts, and Service-Level Alerts (~31 engineering days).

**Key numbers:**

| Metric | Current (PML) | Target (MDE) |
| --- | --- | --- |
| Services | 25+ standalone JARs | 14 modules in 1 monorepo |
| Languages | Java 8-17, mixed | Java 21 (unified) |
| Data stores | ES + MySQL + Redis (7 DBs) + S3 | PostgreSQL + TimescaleDB + ClickHouse + Redis (2 clusters) + ES + S3 |
| Feed source | CMOTS API + NSE multicast | NSE Direct UDP (colocation) |
| Tick latency | 50-200ms (via CMOTS relay) | <2ms P50, <10ms P99 (direct feed) |
| Candle storage | ES indices (limited history) | ClickHouse 10yr + TimescaleDB 90d intraday |
| Deployment | 25 separate CI/CD pipelines | 5 deployment groups, 1 pipeline |

#### 4.1.1 Intraday Candles (ES → TimescaleDB)

| Detail | Value |
| --- | --- |
| Source | ES `intraday_candle_*` indices |
| Target | TimescaleDB `candles_1m` hypertable |
| Data volume | ~50GB (90 days × ~5K instruments × 375 candles/day) |
| Migration strategy | Bulk export from ES → CSV → TimescaleDB `COPY` |
| Timeline | 1 day (export) + 1 day (import) + 1 day (validation) = **3 days** |
| Validation | Row count match, random sample price comparison (1% of rows) |

```sql
-- TimescaleDB target table (already exists from Story 7)
-- candles_1m(timestamp, instrument_token, open, high, low, close, volume, vwap)
-- Hypertable with 1-day chunks, compression after 7 days
```

**Migration script outline:**

1. ES scroll query: `GET /intraday_candle_*/_search?scroll=5m` with batch size 10,000
2. Transform ES `_source` → CSV with columns matching TimescaleDB schema
3. `\COPY candles_1m FROM '/tmp/candles_batch_N.csv' WITH CSV HEADER`
4. Validate: `SELECT COUNT(*), MIN(timestamp), MAX(timestamp) FROM candles_1m`

### 5.2 WebSocket Protocol Changes

| Aspect | Current (PML) | New (MDE) |
| --- | --- | --- |
| Endpoint | `wss://*/ws/tick` | `wss://*/ws/v1/feed` |
| Auth | Cookie-based session | API key (`x-api-key` header) |
| Frame format | JSON text frames | Protobuf binary frames |
| Subscribe | `{"action":"subscribe","tokens":[123,456]}` | JSON control frame: `{"type":"SUBSCRIBE","tokens":["NSE:RELIANCE:EQ"]}` |
| Unsubscribe | `{"action":"unsubscribe","tokens":[123]}` | JSON control frame: `{"type":"UNSUBSCRIBE","tokens":["NSE:RELIANCE:EQ"]}` |
| Tick data | JSON: `{"t":123,"ltp":2450.30,"v":5200000}` | Protobuf binary: `Tick { token, ltp, open, high, low, close, volume, ... }` |
| Throttle | Variable (server-side) | 1 tick/sec per instrument (configurable) |
| Client pattern | SharedWorker + multiple tabs | Same pattern supported (binary decode in worker) |
| Reconnection | Manual (app handles) | Server sends heartbeat, client reconnects on miss |

## 7. New Feature Plans

These features were identified as missing from the original Price Engine design and are planned (not implemented).

### 7.1 Market Depth (Level 2 Order Book)

**What it does:** Shows the top 5 bid/ask price levels with quantities for any instrument. Gives traders insight into supply/demand at each price level.

**Current state:** PML does NOT expose market depth to the UI. The data exists in the exchange feed but is discarded after best bid/ask extraction.

**MDE state:** Feed Handler already decodes `BCastHeader` with full MBP (Market By Price) depth data. Publisher currently writes only best bid/ask to Redis. The depth data is available in the `Tick` protobuf but not stored.

**Design:**

```text
Feed Handler -> Tick proto (already includes depth[5]) -> Publisher
                                                          |
                                                   Redis C2: DEPTH_{seg}:{pmlId}
                                                   Value: protobuf DepthSnapshot {
                                                       bid[5] { price, qty, orders }
                                                       ask[5] { price, qty, orders }
                                                       timestamp
                                                   }
                                                   TTL: none (overwritten each tick)
                                                          |
                                                   REST: GET /v1/depth/{token}
                                                   WebSocket: included in tick frame (optional subscribe flag)
```

**Implementation plan:**

1. **Publisher change** (1 day): Add `DepthWriter` component — extracts depth from `Tick` proto, writes to Redis C2 as `DEPTH_{seg}:{pmlId}` key with protobuf-serialized `DepthSnapshot` value
2. **REST endpoint** (0.5 day): `GET /v1/depth/{token}` — reads from Redis C2, returns JSON
3. **WebSocket extension** (1 day): New subscribe mode `SUBSCRIBE_WITH_DEPTH` — tick frames include depth levels
4. **Frontend** (2 days): Depth ladder widget (bid/ask visualization with color-coded quantity bars)

**Effort: 4.5 days** | **Dependencies:** Publisher (done), Broadcast Service (done)

### 7.2 Peer Groups (Sector Comparison)

**What it does:** For any stock, shows a comparison table of key metrics (P/E, ROE, market cap, revenue growth, etc.) against its industry/sector peers. Essential for investment decision-making.

**Current state:** PML has no peer comparison feature. Users must manually search for companies in the same sector.

**Design:**

```text
Instrument Master (sector, industry data)
                              |
                   PostgreSQL: peer_groups table
                   +---------------------------------------------+
                   | isin | sector | industry | indices | ...     |
                   +---------------------------------------------+
                              |
                   Peer Resolution Service
                   1. Find all instruments in same industry
                   2. Filter by market cap tier (large/mid/small)
                   3. Enrich with fundamentals from company_fundamentals
                   4. Enrich with live price data from Redis snapshot
                              |
                   REST: GET /v1/peers/{isin}?limit=10
                   Response: {
                       target: { isin, symbol, metrics... },
                       peers: [{ isin, symbol, metrics... }, ...],
                       industryAvg: { pe, roe, debtEquity, ... }
                   }
```

**Implementation plan:**

1. **Schema migration** (0.5 day): Add `sector`, `industry`, `market_cap_tier` columns to `instrument_master` if not present; Flyway V3 migration
2. **CMOTS integration** (2 days): Temporal workflow to fetch sector/industry classification from CMOTS API, populate `instrument_master`
3. **Peer resolution logic** (2 days): Service that finds peers (same industry + similar market cap), computes industry averages
4. **REST endpoint** (1 day): `GET /v1/peers/{isin}` with Caffeine cache (5-min TTL, refreshed on fundamentals update)
5. **Frontend** (2.5 days): Peer comparison table component, bar charts for metric comparison

**Effort: 8 days** | **Dependencies:** Company Fundamentals Service (not yet built), Instrument Master (done)

### 7.3 User Price Alerts

**What it does:** Users set price conditions (e.g., "alert me when RELIANCE crosses ₹2500") and receive push notifications when triggered.

**Current state:** PML has no price alert feature. Users must manually check prices.

**Design:**

```text
User API
                   POST /v1/alerts { instrumentToken, type, value }
                   GET  /v1/alerts (list user's alerts)
                   DELETE /v1/alerts/{id}
                              |
                   PostgreSQL: alert_rules (already in schema)
                   Redis C2: alerts:instrument:{token} -> Set<AlertRule>
                   Redis C2: alerts:user:{userId} -> Set<alertId>
                              |
                   Alert Evaluation Engine (runs inside Publisher pipeline)
                   On each tick:
                   1. Check if instrument has active alerts (Redis lookup, O(1))
                   2. If yes, evaluate each alert condition
                   3. If triggered: update status, publish to Kafka, send push
                              |
                   Kafka: mde-alerts-triggered
                              |
                   Push Notification Service (external -- FCM/APNs)
```

**Alert types supported:**

- `PRICE_ABOVE` / `PRICE_BELOW` — absolute price crossing
- `PERCENT_CHANGE` — % change from previous close exceeds threshold
- `VOLUME_SPIKE` — volume exceeds N× average
- `CIRCUIT_HIT` — upper or lower circuit hit
- `52W_HIGH` / `52W_LOW` — new 52-week high or low

**Scale considerations:**

- Target: 1M active alerts across all users
- Redis `alerts:instrument:{token}` keeps alerts indexed by instrument for O(1) lookup during tick processing
- Most instruments will have 0 alerts; popular stocks (RELIANCE, TCS, NIFTY) might have 10K+ alerts each
- Alert evaluation per tick: ~0.1ms even for 10K alerts (simple numeric comparison)

**Implementation plan:**

1. **Alert CRUD API** (2 days): REST endpoints, PostgreSQL persistence, Redis index sync
2. **Alert evaluation engine** (3 days): Plugs into Publisher tick pipeline, evaluates all alert types
3. **Kafka publisher** (1 day): Publishes triggered alerts to `mde-alerts-triggered`
4. **Push notification integration** (2 days): Consumes from Kafka, calls FCM/APNs
5. **Rate limiting** (1 day): Max 50 alerts per user, max 10 alerts per instrument per user, cooldown period
6. **Frontend** (3 days): Alert creation form (on stock detail page), alerts list page, notification center

**Effort: 12 days** | **Dependencies:** Publisher (done), PostgreSQL schema (done)

### 7.4 Service-Level Alerts (System Monitoring)

**What it does:** Automatic, system-wide alerts for market anomalies that ops/traders should know about — NOT user-configured.

**Current state:** PML has basic Grafana alerting on metrics but no semantic market event alerts.

**Design:**

```text
Publisher (tick pipeline)
                              |
                   Service Alert Detectors (run inside Publisher)
                   +-- CircuitHitDetector -- instrument hits upper/lower circuit
                   +-- VolumeAnomalyDetector -- volume > 5x 20-day avg in 1 min
                   +-- IndexLevelBreachDetector -- NIFTY/SENSEX crosses round numbers
                   +-- FeedGapDetector -- no ticks for instrument > 30s during market hours
                   +-- FlashCrashDetector -- >5% drop in <60s across 10+ instruments
                              |
                   Kafka: mde-feed-anomalies (already exists, Story 14)
                              |
                   Incident Manager (already built, Story 14)
                   -> Classifies severity (INFO/WARNING/CRITICAL)
                   -> Stores in PostgreSQL incidents table
                   -> Routes to Slack/PagerDuty based on severity
```

**Implementation plan:**

1. **Detector implementations** (3 days): 5 detector classes plugged into Publisher’s tick dispatch pipeline
2. **Anomaly event schema** (0.5 day): Protobuf `AnomalyEvent { type, instrument, severity, details, timestamp }`
3. **Incident Manager integration** (1 day): Consume anomaly events, classify, store, route
4. **Grafana dashboard** (1 day): Anomaly frequency, severity distribution, response time
5. **Slack/PagerDuty routing** (1 day): Webhook integration for CRITICAL alerts

**Effort: 6.5 days** | **Dependencies:** Publisher (done), Incident Manager (done)

## Appendix A: Instrument Token Mapping

The biggest breaking change is the token format:

| Current (PML) | New (MDE) | Example |
| --- | --- | --- |
| Numeric scripId (`256265`) | String token (`NSE:RELIANCE:EQ`) | RELIANCE equity |
| Numeric scripId (`11536386`) | String token (`NSE:RELIANCE:FUT:2026-04-24`) | RELIANCE futures |
| Numeric scripId (`11536642`) | String token (`NSE:RELIANCE:CE:2026-04-24:2500`) | RELIANCE call option |

**Migration approach:** Build a bidirectional mapping service:

- `GET /v1/internal/token-map?scripId=256265` → `NSE:RELIANCE:EQ`
- `GET /v1/internal/token-map?token=NSE:RELIANCE:EQ` → `256265`

This mapping service is used during the transition period (Phase 2-4). After full migration, the scripId mapping is deprecated.
