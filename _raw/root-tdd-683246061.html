1. Overview

What is the Market Data Engine?

The Market Data Engine is the single, authoritative source of all market data in the Paytm Money Pro platform. Every price a user sees — equity LTP, option chain, chart candle, market mover ranking, index value — originates from this engine.

No other system connects directly to exchange feeds. This means:

  • One place to fix a bad price
  • One place to monitor feed health
  • One place for corporate action adjustments
  • One audit trail for price corrections

Design Philosophy

  1. Correctness first — A wrong price delivered fast is worse than a correct price slightly slower.
  2. Self-healing over manual intervention — Auto-reconnect, auto-recover, notify operators (not page them).
  3. Composable — Independently deployable. Trade Engine, Portfolio Engine, Ledger Engine depend on it, not the other way around.
  4. Observable — Every tick, correction, feed switch, reconnection is logged, metriced, and traceable.
  5. API-compatible — All read APIs match the current contract since the UI is not being replaced.
  6. Zero-Error UX (new in v3.2) — Users must never see an error popup. The system degrades silently through layered fallbacks; the AI Incident Manager handles internal noise.

4. Architecture — High Level

5. Feed Sources & Failover

Merged Feed Architecture

The Market Data Engine receives data from two independent sources simultaneously and merges them into a single, best-of-both stream:

SourceProtocolLocationTypical Latency
NSE/BSE DirectUDP Multicast, dedicated cable, 6 streamsExchange colocationBaseline (fastest)
OmnenestOmnenest RS protocol (proprietary)Third-party DC3–8 ms slower than NSE Direct

How the merge works:

Both NSE Direct and Omnenest feed into the same Feed Handler. The Feed Handler deduplicates by (instrumentId, timestamp) — whichever tick arrives first wins. Since NSE Direct is typically 3–8 ms faster, NSE ticks usually arrive first and are used. Omnenest ticks that arrive second are dropped as duplicates. But if NSE misses a UDP packet (packet loss), the Omnenest copy fills the gap automatically — and vice versa.

Key properties of the merged architecture:

  • All users get the best-of-both sources — always the fastest, most complete data
  • No user is ever on a "slower" feed — unlike an 85/15 traffic split
  • Omnenest is continuously validated — every Omnenest tick is compared against its NSE counterpart. Divergence beyond threshold triggers an alert.
  • Automatic gap filling — if either source drops a packet, the other fills in seamlessly
  • Zero-switchover failover — if NSE dies entirely (heartbeat timeout), Omnenest is already active and delivering data. No switchover delay.
  • Input ~4K TPS (2K from each source), output ~2K TPS after dedup

Open point: Verify whether NSE_FNO_0 / NSE_FNO_1 represent A/B redundancy (same instruments, two copies) or separate instrument sets (different instruments per stream). This affects dedup logic.

Failover Detection

Critical: Exchanges only send ticks when values change. Not receiving a tick for a few seconds is normal — it means no trades happened. We cannot use "no ticks received" as the sole detection signal.

Failover actions:

External Circuit Breaker

A circuit breaker sits between the exchange and our feed handler. This is NOT our service — it's managed externally. When the exchange halts trading on an instrument (circuit limit hit), the halt message reaches us through this external component.
