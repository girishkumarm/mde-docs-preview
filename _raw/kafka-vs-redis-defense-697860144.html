Audience: engineers / managers questioning why MDE uses Kafka for broadcast fanout instead of Redis Pub/Sub.
Format: 5-minute read. Every claim has a citation.
Last updated: 2026-04-20, from 10 parallel research + code-audit agents.

TL;DR — The case in 5 bullets

  1. Redis Pub/Sub explicitly loses messages. Redis's own docs: "messages … are forever lost" on disconnect. Not a scaling quirk — a documented design. No replay, no persistence, no offsets. (Redis docs)

  2. Redis Pub/Sub scales NEGATIVELY in a cluster. Redis maintainer, GitHub #2672: "PubSub scales linearly … but in the negative direction." 1 node → 12.5K RPS; 10 nodes → worse. That is the opposite of what we need. (GitHub issue)

  3. Kafka is the peer-group choice. Zerodha open-sources kaf-relay (a production Kafka cross-cluster replicator). Robinhood calls Kafka "mission-critical … in every line of business, including market data." Confluent runs a tier-1 bank trading desk at 1.6 M msgs/s with p99 < 5 ms. (zerodha/kaf-relay · Robinhood Kafka Summit · Confluent tier-1 bank)

  4. The math is ruthless. At MDE peak (5k ticks/s × ~300 B × 20 broadcast pods), Redis must push 100k deliveries/s out of a single-threaded node — Discord documented the Redis ~100k/s cliff in exactly this scenario. Kafka (model A, unique group per pod) = 10 MB/s per broker = 1.3% of a 10 GbE NIC. Linear scaling by adding brokers.

  5. "Why not 90,000 topics or partitions?" — it is a non-question. Kafka scales by throughput not entity count. Jun Rao (Confluent, Kafka co-creator): partition count = max(target_throughput / 10 MB·s⁻¹). MDE needs 6 topics × ~280 partitions total, 0.14% of a standard cluster's capacity, keyed by instrumentToken so all 90k instruments hash onto those 280 partitions with strict per-instrument ordering. (Jun Rao / Confluent)

The five killer citations (bring these verbatim)

1. Redis itself says Pub/Sub loses messages

"Redis' Pub/Sub exhibits at-most-once message delivery semantics. As the name suggests, it means that a message will be delivered once if at all. Once the message is sent by the Redis server, there's no chance of it being sent again. If the subscriber is unable to handle the message (for example, due to an error or a network disconnect) the message is forever lost."

— https://redis.io/docs/latest/develop/pubsub/

2. Redis Pub/Sub has NEGATIVE cluster scaling

"PubSub works by broadcasting every publish to every other Redis Cluster node… With 1KB messages, 10 nodes, 1 GBit/s: 12.5K RPS. With 5KB messages, 50 nodes: only 500 RPS (vs >100K RPS on a single instance). PubSub scales linearly with respect to the cluster size, but in the negative direction."

— https://github.com/redis/redis/issues/2672 (Redis maintainer)

3. Redis in cluster mode silently drops messages on replicas, "by design"

GitHub #10862 — maintainers confirmed Pub/Sub messages are silently lost when subscribers connect to replica nodes. Closed as working-as-intended.

— https://github.com/redis/redis/issues/10862

4. AWS itself routes high-throughput fanout away from Redis

"Kafka generally has higher throughput than Redis OSS pub/sub… Use Kafka for applications that stream large datasets and require high recoverability. Choose Redis OSS when… tolerate small data loss."

— https://aws.amazon.com/compare/the-difference-between-kafka-and-redis/

5. Kafka's fanout semantics are in the docs in one sentence

"If all the consumer instances have the same consumer group, records will effectively be load balanced over the consumer instances. If all the consumer instances have different consumer groups, then each record will be broadcast to all the consumer processes."

— https://kafka.apache.org/documentation/#intro_consumers

That last quote is the entire answer to "but won't every pod reading from Kafka get duplicate deliveries?" — it is literally the semantics we want.

"But we already have Redis. Why another system?"

Redis still plays two roles in MDE. Neither is on the tick broadcast path.

  1. Redis Cluster 1 (Publisher writes, everyone reads): point-in-time snapshots — last tick, OHLC, circuit limits, previous close, 52-week high/low. Used by charts-api, broadcast-service (cold-start snapshot only), instrument-master. Read once per subscription, not per tick. This is correct, defensible, and we keep it.

  2. Redis Cluster 2 (API cache): leaderboards (most-active, top-gainers), session caches. Never fanout.

Redis is kept in its lane. Kafka is the fanout bus. The two are orthogonal.

"Why not Redis Streams then? Same vendor, same box."

  • Redis Streams cannot shard a single stream across cluster nodes: "the partitioning granularity is the key, so it is not possible to shard a dataset with a single huge key." → App-level sharding on top, then you have Kafka with worse tooling. (Redis cluster spec)

  • Default persistence is fsync-every-1-second: "you can only lose one second worth of writes." At 5,000 tps that is 5,000 ticks on every crash. (Redis persistence)

  • Even Redis-friendly hosts recommend Kafka for this profile: Instaclustr: "Use Kafka if you need unbounded throughput and reliable horizontally scalable clusters, unbounded persisted message retention and replaying." (Instaclustr blog)

  • Redis 7.0 Sharded Pub/Sub (SPUBLISH/SSUBSCRIBE): routing optimization only. Still no persistence, still no consumer groups, still no replay. Not a solution, just a smaller patch on the tower.

  • Replication: RF=3, min.insync.replicas=2, acks=all → no committed loss while ≥1 ISR alive.

    • market-movers-writer: mde-movers (shared)

How skeptics will attack — and how you answer

They say

You answer

"Redis is simpler, we already run it."

Redis Pub/Sub is at-most-once by design (Redis docs). For a regulated market-data pipeline, that is a compliance blocker. Kafka is not "another system" — it is the bus every peer Indian broker already uses (Zerodha/kaf-relay is public).

"If 20 pods read from Kafka, every tick is sent 20 times."

Yes — and that's what we want. Kafka docs: "If all the consumer instances have different consumer groups, then each record will be broadcast to all the consumer processes." Egress math: 10 MB/s per broker. 1.3% of a 10 GbE NIC. Cheap.

"Can Kafka hold 90,000 topics or partitions?"

We don't need that. 6 topics, 280 partitions, all 90k instruments hashed onto them by instrumentToken. Jun Rao's formula: partitions = target_throughput / 10 MB·s⁻¹ → 280 is 4× peak need.

"Why not Redis Streams?"

A single stream cannot shard across Redis Cluster nodes (Redis cluster spec). Fsync-every-1-second = 5,000 ticks lost per crash. Instaclustr (Redis-friendly vendor) recommends Kafka for our profile.

"Kafka rebalance storms will kill us."

Fixed by static membership (KIP-345, group.instance.id) + cooperative-sticky assignor (KIP-429). Pods get cached assignments on restart — no rebalance triggered.

"Kafka is too high-latency for market data."

Confluent benchmark: tier-1 bank trading desk 1.6 M msg/s at p99 < 5 ms. MDE needs far less than that. LinkedIn runs 7 trillion msgs/day on Kafka.

"What if a broadcast pod crashes?"

Kafka: resumes from committed offset. Zero ticks lost. Redis Pub/Sub equivalent: 81% of active instruments miss a tick in a 10-second restart window.

"What about the radio-tower problem you described in Redis — doesn't Kafka have the same single-broker issue?"

No. Kafka splits partitions across brokers. One topic → 60 partitions → leaders balanced across 3 brokers → 20 MB/s per broker. Redis Pub/Sub is single-threaded single-node regardless of Redis Cluster size (GitHub #2672).

"What's the exit strategy if Kafka doesn't work out?"

Kafka wire protocol has 3 drop-in replacements: Amazon MSK (managed OSS Kafka), Redpanda (C++ rewrite), Confluent Cloud. All speak the same protocol. Our code requires zero changes to swap vendors.

"Are we sure this is industry standard?"

Zerodha kaf-relay — Kafka cross-cluster replicator, open-source, production. Robinhood — Kafka "in every line of business, including market data." Upstox SDE-II JD requires Kafka. No Indian broker publishes an architecture where Redis Pub/Sub is the market-data bus.
