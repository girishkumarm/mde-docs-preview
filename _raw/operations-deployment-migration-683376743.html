13. Migration Strategy

Strangler fig pattern — Phase 0 + 4 phases, roughly 8 weeks total. Phase 0 is new in v3.1 and must complete before the rest of the migration can rely on TimescaleDB.

Phase 0 — Historical data migration via Temporal (prerequisite)

Runs before any production traffic hits TimescaleDB.

  1. Stand up the separate TimescaleDB PostgreSQL instance, create hypertables, compression policies, retention policy, and all continuous aggregates (see §6.10).
  2. Launch HistoricalDataMigrationWorkflow on Temporal:
  • Activity A: read MongoDB candles collection in date-partitioned batches → write candles_1m with ON CONFLICT ... DO UPDATE.
  • Activity B: read Elasticsearch historical indices → write candles_1m (same upsert semantics).
  • Activity C: refresh all continuous aggregates for the full historical window.
  • Activity D: validate — per-day row counts TimescaleDB vs MongoDB/ES, spot-check sample instruments (liquid + illiquid) across sample dates.
  1. Keep MongoDB + Elasticsearch in read-only mode for 30 days as a safety net. Charts API is configured to fall back to the old stores if a TimescaleDB data defect is found.
  2. Cut Charts API reads over to TimescaleDB after validation passes.
  3. After 30 days of clean operation, decommission MongoDB and Elasticsearch.

15. Business Rules Checklist

An engineer building this system must implement all of the following:

Feed Handling

  • ☐ One virtual thread per UDP stream (12 total — 6 NSE Direct + 6 Omnenest)
  • ☐ Buffer size: 8192 bytes per datagram
  • ☐ Drop malformed packets silently (log warn), never crash receiver thread
  • ☐ Parse BCastHeaderBroadcast (64 bytes) first, route by packetCode
  • ☐ Set uniqueSequenceId as MDC traceId for every packet

Cross-Source Merge & Dedup

  • ☐ Both NSE Direct and Omnenest feed into the same Feed Handler
  • ☐ Dedup by (instrumentId, timestamp) — first arrival wins
  • ☐ Track per-source tick counts for continuous validation
  • ☐ Alert on divergence: if sources disagree on LTP by > threshold for > 5 seconds
  • ☐ Metrics: cross_source_dedup_count per source

Duplicate Detection (per-source)

  • ☐ Skip if ltp + volume + avgTradePrice all unchanged from previous tick
  • ☐ Index: skip if LTP unchanged
  • ☐ OI: skip if OI unchanged, or if volume/OHLC are all zero

MBP Processing

  • ☐ Valid trade gate: volume > 0 AND market OPEN or POST_CLOSE_START
  • ☐ OHLC: Lua atomic updates (3_O, 3_H, 3_L, 3_C)
  • ☐ 52-week: Lua atomic updates (36_H, 36_L)

Redis Batching

  • ☐ One RedisBatchProcessor per CacheType
  • ☐ Flush on: queue > max batch size OR 10ms timeout
  • ☐ Single HSET per flush
  • ☐ @Retryable(maxAttempts=6, exponential 500ms→60s)

Redis Topology

  • ☐ Two separate Redis clusters: Cluster 1 (Feed) and Cluster 2 (API)
  • ☐ maxmemory-policy noeviction on both clusters
  • ☐ Alert at 80% memory usage
  • ☐ Key prefixes only, DB 0 only (no DB numbers) — required for future Cluster mode
  • ☐ Dedicated connection pools per service per cluster

Kafka Producer

  • ☐ enable.idempotence=true
  • ☐ acks=all
  • ☐ retries=2147483647 (infinite retries within delivery.timeout.ms)
  • ☐ max.in.flight.requests.per.connection=5
  • ☐ delivery.timeout.ms=120000
  • ☐ Avro serialization with schema registry
  • ☐ Partition key: instrumentToken (ensures ordering per instrument)

Market Status

  • ☐ Track per (ExchangeSegment, marketType) pair, not globally
  • ☐ POST_CLOSE_START (5.0) is valid trading state
  • ☐ CLOSE (4.0) stops all price updates

Feed Failover

  • ☐ Heartbeat-based detection: 3 seconds timeout (confirm heartbeat interval with exchange team)
  • ☐ Staleness indicator: 1 second no heartbeat → UNCERTAIN flag → propagate to Broadcast Service
  • ☐ Both sources merged — no traffic routing needed
  • ☐ Both down: serve stale from Redis, mark as stale, CRITICAL alert
  • ☐ Staleness UI: amber (>3s), red (>10s) coloring on "Last updated: Xs ago"

Pre-Market Loading

  • ☐ All schedulers skip non-trading days (@TradeDay)
  • ☐ If TRD_DAY absent → P0 alert
  • ☐ Execution order: clean → scrip master → ckt/OHLC/pclose/52wk → trade/index

Instrument Master

  • ☐ Scrip master in a dedicated PostgreSQL instance only — no Redis copy
  • ☐ Instrument master PG is separate from the TimescaleDB time-series instance
  • ☐ Services load instrument map into ConcurrentHashMap on startup
  • ☐ Reload on Kafka event (instrument-master-updated topic)

TimescaleDB (Time-Series Store)

  • ☐ Deployed as a SEPARATE PostgreSQL instance from Instrument Master (workload isolation)
  • ☐ ticks defined as a hypertable, chunk interval 1 day
  • ☐ ticks retention policy: drop chunks older than 7 days (hot retention)
  • ☐ candles_1m defined as a hypertable, chunk interval 7 days
  • ☐ candles_1m compression policy: compress chunks older than 7 days (~10x size reduction)
  • ☐ candles_1m retention: 10 years
  • ☐ Continuous aggregates defined for 5m, 15m, 1h, 1d, 1w, 1M on top of candles_1m
  • ☐ Continuous aggregate refresh policies: 5m/15m/1h refresh every minute with 2-minute lag; 1d refresh every 5 min intraday + once post-close; 1w/1M refresh at EOD
  • ☐ No application code performs rollups — all higher intervals are produced by continuous aggregates
  • ☐ Charts Data Writer writes ONLY ticks and candles_1m; no 5m/15m/1h writes
  • ☐ Charts API routes interval parameter to the correct view (candles_1m / 5m / 15m / 1h / 1d / 1w / 1M)
  • ☐ Monitor timescaledb.cagg.refresh.lag per view — alert at P1 if any view falls behind threshold
  • ☐ Separate backup/PITR policy from Instrument Master PG

Historical Data Migration (Phase 0, one-time)

  • ☐ Migration implemented as a Temporal workflow (HistoricalDataMigrationWorkflow)
  • ☐ Activity reads MongoDB historical candles and writes candles_1m idempotently (ON CONFLICT ... DO UPDATE)
  • ☐ Activity reads Elasticsearch historical data and writes candles_1m idempotently
  • ☐ Activity refreshes continuous aggregates for the full backfilled window
  • ☐ Validation activity verifies row counts per day TimescaleDB vs source
  • ☐ Validation activity spot-checks sample instruments (liquid + illiquid) across sample dates
  • ☐ MongoDB and Elasticsearch kept in read-only mode for 30 days as safety net
  • ☐ Charts API has a runtime-toggleable fallback to legacy stores during the safety window
  • ☐ Cutover to TimescaleDB only after validation passes
  • ☐ Flyway-style versioned migrations for schema changes; all migration activities are idempotent and safely re-runnable
  • ☐ Decommission MongoDB + Elasticsearch after 30-day clean-operation window

Reconciliation

  • ☐ Only pclose is reconciled
  • ☐ Compare Redis vs exchange bhav copy (from S3)
  • ☐ Upload CSV to S3 regardless of mismatch count
  • ☐ Critical mismatch → P0 Slack

Expiry Day Cleanup

  • ☐ Triggered by market close on expiry days (weekly Thu + monthly last Thu)
  • ☐ Mark expired contracts as EXPIRED in PostgreSQL
  • ☐ Clean expired data from Redis Cluster 2 (option chain, movers, insights)
  • ☐ Stop Greeks computation for expired contracts
  • ☐ Recalculate PCR/Max Pain using only live contracts
  • ☐ Propagate settlement prices to Portfolio/Ledger Engine via Kafka
  • ☐ Completion target: < 30 seconds

Resilience (new in v3.2)

  • ☐ Every downstream call wrapped with a Resilience4j circuit breaker (50% failure threshold over 20 requests, exponential cooldown 10s/30s/60s)
  • ☐ Every user-facing API has a declared 4-layer fallback chain (live → cached → stale → empty); failover between layers is automatic
  • ☐ Stale-while-revalidate enabled on every cache read — synchronous response served from cache, background refresh kicks off if cache age exceeds threshold
  • ☐ Request hedging enabled for latency-critical reads (live price, option chain, snapshot-on-connect) — parallel reads to two Redis replicas, first response wins
  • ☐ Per-API fallback metrics emitted (fallback.invocations) — Layer 4 invocation rate > 0.1% over 5 min triggers P1 alert

AI-Powered Incident Manager (new in v3.2)

  • ☐ Signal Collectors subscribed to Prometheus, Loki/Elasticsearch logs, Kafka consumer lag, Redis/PG slow query log, synthetic probes
  • ☐ Anomaly detection uses statistical baselines (z-score > 3 sustained 60s; EWMA; Prophet for seasonality), not single-threshold alerting
  • ☐ Cross-service correlation engine merges related anomalies into a single incident
  • ☐ LLM debugger (Claude Sonnet 4.5 or configured provider) receives full context — last 5 min metrics + ERROR logs + traces + recent deploys + service topology + runbook references
  • ☐ Auto-remediation gated by an explicit allowlist (rollout undo, pod restart, scale up, feature-flag toggle, refresh continuous aggregate). Destructive actions NEVER auto-execute.
  • ☐ Auto-remediation rate-limited: same service may not be auto-remediated more than 3 times in 30 minutes (anti-flap)

Zero-Error UX (new in v3.2)

  • ☐ All HTTP 5xx errors converted to 200 OK with { "data": null, "stale": true, "source": "FALLBACK_L4", "lastUpdatedMs": ..., "message": "Updating" } before returning to client
  • ☐ Frontend treats null data as a loading state — never as an error; renders skeleton or last-known-good
  • ☐ Silent feature flags hide widgets when their backing component is degraded — no error banners, no greyed-out states, just fewer widgets
  • ☐ WebSocket auto-reconnect with exponential backoff (1s/2s/4s/8s/16s/30s capped) — fetches fresh snapshot on reconnect, then resumes stream
  • ☐ Mobile + web cache last known state on close; render instantly from cache on next open while WebSocket reconnects in background
