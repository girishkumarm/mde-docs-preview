## 6. Component Deep Dives — Part 2 (FNO, Movers, Alerts, Scheduler, Corporate Actions, Ops, Resilience)

### 6.12 FNO Data Writer

**What it does:** Consumes F&O tick data from Kafka, computes option analytics (Greeks, IV, Max Pain, ATM Straddle, PCR, day-over-day changes), stores enriched data in Redis Cluster 2. Handles **all F&O instrument types** across NSE and BSE -- futures AND options.

**Input:** Kafka topics (`mde-ticks-nse-derivative`, `mde-ticks-bse-derivative`) **Output:** Redis Cluster 2 (option chain, option insights, futures data)

#### Greeks Pricing Models

Two distinct pricing models are used, selected based on the underlying instrument type:

<Tabs>
  <Tab title="Black-Scholes-Merton (BSM)">
    **Used for:** EQUITY stock options (RELIANCE, TCS, INFY, etc. -- `OPTSTK` instrument type)

    **Inputs:**
    - S = spot price of the underlying stock
    - K = option strike price
    - T = time to expiry (in years, computed from market close timestamp)
    - sigma = implied volatility (solved iteratively)
    - r = risk-free rate (91-day T-bill rate, refreshed daily from RBI data)
    - q = continuous dividend yield (annualized, from corporate action data)

    **Outputs (per contract):** delta, gamma, theta, vega, rho for both CE and PE

    **Why BSM:** Stock options settle against the spot price of the underlying equity. The dividend yield `q` adjusts for expected cash flows during the option's life.
  </Tab>
  <Tab title="Black-76">
    **Used for:** INDEX options (NIFTY, BANKNIFTY, FINNIFTY, MIDCPNIFTY, SENSEX -- `OPTIDX` instrument type)

    **Inputs:**
    - F = futures price of the underlying index (from the nearest-expiry futures contract)
    - K = option strike price
    - T = time to expiry (in years)
    - sigma = implied volatility (solved iteratively)
    - r = risk-free rate (91-day T-bill rate)

    **Key difference from BSM:** Uses futures price F instead of spot S, and has no dividend yield q parameter.

    **Why Black-76:** Index options on NSE/BSE settle against the futures price, not the spot index value. Using the futures price directly eliminates the need to model the cost-of-carry and dividend expectations that are already embedded in the futures price.
  </Tab>
  <Tab title="IV Solver">
    **Newton-Raphson (primary):** Fast quadratic convergence for near-ATM options. Typically converges in 3-5 iterations. Uses vega (the partial derivative of option price with respect to volatility) as the Jacobian.

    **Brent method (fallback):** Robust bisection-secant hybrid for deep ITM/OTM options where Newton-Raphson may diverge due to near-zero vega. Guaranteed convergence within the bounded interval.

    **Configuration:**
    - IV bounds: [0.01, 5.0] (1% to 500% annualized volatility)
    - Max iterations: 100
    - Convergence tolerance: 1e-8
    - Fallback trigger: Newton-Raphson fails to converge within 20 iterations or steps outside bounds
  </Tab>
</Tabs>

#### Computed Fields and Recomputation Frequency

| Field | Formula / Source | Frequency |
| --- | --- | --- |
| Greeks (Delta, Gamma, Theta, Vega, Rho) | BSM for equity options, Black-76 for index options | ATM +/-5 strikes: every tick. Rest: every 5-30s adaptive |
| Implied Volatility (IV) | Newton-Raphson with Brent fallback | Same as Greeks |
| Max Pain | For each strike K: totalPain = sum(call_OI x max(0, K - strike_i)) + sum(put_OI x max(0, strike_i - K)). Max Pain = K that minimizes totalPain | Throttled to once per 5 seconds per underlying |
| ATM Straddle | CE price + PE price at strike closest to spot. ATM IV = avg(CE IV, PE IV) | On every underlying spot tick |
| PCR (OI) | Total Put OI / Total Call OI (guard: if denominator = 0, PCR = 0) | Every 5 seconds |
| PCR (Volume) | Total Put Volume / Total Call Volume (guard: if denominator = 0, PCR = 0) | Every 5 seconds |
| Day-over-day OI change | oiChange = current OI - previous day OI (from scheduler-service F&O bhavcopy BOD data) | On every tick |
| Day-over-day price change | priceChange = current LTP - previous day close; priceChangePct = (priceChange / pClose) x 100 | On every tick |
| Day-over-day volume change | volumeChange = current volume - previous day volume | On every tick |

<Note>
Day-over-day change fields (TASK-28) depend on TASK-04 (NSE F&O bhavcopy parsing by scheduler-service) for previous day reference data. The `pOi`, `pClose`, and previous day volume are loaded into Redis by scheduler-service during BOD (06:00 IST) and read by fno-data-writer at startup.
</Note>

#### Max Pain Calculation (TASK-25)

For each underlying (NIFTY, BANKNIFTY, RELIANCE, etc.) and expiry date:

<Steps>
  <Step title="Enumerate all strikes">
    Load the full option chain for the underlying + expiry from Redis.
  </Step>
  <Step title="Compute total pain at each strike K">
    For each candidate strike K in the chain:
    ```
    totalPain(K) = SUM(call_OI_i * max(0, K - strike_i))    // calls expire worthless below K
                 + SUM(put_OI_i  * max(0, strike_i - K))     // puts expire worthless above K
    ```
    where the sum runs over all strikes `i` in the chain.
  </Step>
  <Step title="Select Max Pain strike">
    Max Pain strike = K that **minimizes** totalPain (the strike where option writers collectively lose the least money).
  </Step>
  <Step title="Write to Redis">
    Key: `FNO:MAX_PAIN:{underlying}:{expiry}`
    Value: `{"strike": 22500, "totalPain": 1234567890, "updatedAt": "..."}`
  </Step>
</Steps>

**Throttling:** Recomputed on every OI update, throttled to once per 5 seconds per underlying to avoid excessive computation.

**REST API:** `GET /v1/max-pain/{underlying}?expiry={date}`

#### ATM Straddle Computation (TASK-26)

<Steps>
  <Step title="Find ATM strike">
    ATM strike = strike closest to the current spot price of the underlying.
  </Step>
  <Step title="Compute straddle premium">
    ATM Straddle premium = CE price at ATM strike + PE price at ATM strike.
  </Step>
  <Step title="Compute straddle IV">
    ATM Straddle IV = average of CE IV and PE IV at ATM strike.
  </Step>
  <Step title="Write to Redis">
    Key: `FNO:ATM_STRADDLE:{underlying}:{expiry}`
    Value: `{"strike": 22500, "cePrice": 150.5, "pePrice": 120.3, "premium": 270.8, "ceIV": 0.15, "peIV": 0.14, "avgIV": 0.145}`
  </Step>
</Steps>

**Trigger:** Recomputed on every tick update for the underlying spot price.

**REST API:** `GET /v1/atm-straddle/{underlying}?expiry={date}`

#### Put-Call Ratio (TASK-27)

For each underlying + expiry combination:

```
PCR_OI  = total_put_OI  / total_call_OI    (all strikes summed)
PCR_VOL = total_put_volume / total_call_volume
```

**Guard:** If denominator is 0, set PCR to 0 (prevents division-by-zero during pre-open or for illiquid expiries).

**Redis key:** `FNO:PCR:{underlying}:{expiry}` with value:
```json
{
  "pcrOI": 1.23,
  "pcrVol": 0.95,
  "totalCallOI": 12500000,
  "totalPutOI": 15375000,
  "totalCallVol": 890000,
  "totalPutVol": 845500,
  "updatedAt": "2026-04-22T10:30:00+05:30"
}
```

**REST API:** `GET /v1/pcr/{underlying}?expiry={date}`

#### BSE F&O Support (TASK-29)

BSE has an F&O segment with options on SENSEX and other BSE indices.

- **instrument-master:** Extended to serve BSE F&O tokens (parsed from BSE F&O contract files during daily sync).
- **feed-handler:** Extended to decode BSE F&O MBP packets (BSE binary protocol, same LZO decompression pipeline as BSE equity).
- **fno-data-writer:** Processes BSE F&O ticks identically to NSE F&O. The computation pipeline is exchange-agnostic -- it routes to the correct pricing model based on instrument type, not exchange.
- **Pricing model:** Black-76 for SENSEX index options (same as NIFTY/BANKNIFTY -- all are index options).
- **Redis keys:** Same pattern with BSE tokens: `FNO:CHAIN:{bse_underlying}:{expiry}`, `FNO:MAX_PAIN:{bse_underlying}:{expiry}`, etc.

### 6.14 Market Movers Writer

<Info>
**Architecture change (TASK-24):** Market movers computation was redesigned from raw ticks to closed 1-minute candles. The old architecture consumed raw ticks from `mde-ticks-*` topics (~50K ticks/sec) and used a CoalescingDispatcher to throttle. The new architecture consumes `mde-candles-1m` from charts-service (~120 candles/min for ~7K equity instruments), dramatically reducing processing load and eliminating CoalescingDispatcher entirely.
</Info>

**What it does:** Consumes closed 1-minute candles from Kafka, computes real-time market mover rankings for **equity instruments only** (~7K), stores in Redis Cluster 2 Sorted Sets with index bucketing.

**Input:** Kafka topic `mde-candles-1m` (from charts-service) **Output:** Redis Cluster 2 Sorted Sets

#### Why Candles Instead of Ticks

| Aspect | Old (raw ticks) | New (1-minute candles) |
| --- | --- | --- |
| Input volume | ~50K ticks/sec across 6 topics | ~120 candles/min for ~7K equity instruments |
| Instrument scope | All 90K+ instruments (equity, F&O, indices) | Equity only (~7K NSE_CASH + BSE_CASH) |
| Throttling | CoalescingDispatcher (complex, memory-intensive) | Not needed (candle cadence is naturally throttled) |
| Accuracy | % change computed from latest tick vs prevClose (noisy) | % change computed from candle.close vs prevClose (stable) |
| Processing overhead | High (deduplicate, coalesce, compute per tick) | Low (one computation per instrument per minute) |

#### Mover Categories

| Category | Signal Computation | Score (ZADD) |
| --- | --- | --- |
| `TOP_GAINERS` | `pctChange = ((candle.close - prevClose) / prevClose) x 100` | pctChange (positive, descending) |
| `TOP_LOSERS` | Same formula as gainers | pctChange (negative, ascending) |
| `MOST_ACTIVE` | `candle.volume` | volume (descending) |
| `VOLUME_SPIKE` | `candle.volume > 2x EMA(20)` rolling average | volume / EMA(20) ratio |
| `BREAKOUT_HIGH` | `candle.high > 52-week high` (from Redis `52WK:{token}`) | pctChange above 52W high |
| `BREAKOUT_LOW` | `candle.low < 52-week low` (from Redis `52WK:{token}`) | pctChange below 52W low |
| `UPPER_CIRCUIT` | `candle.high >= upper circuit limit` (from Redis `LC_UC:{token}`) | pctChange |
| `LOWER_CIRCUIT` | `candle.low <= lower circuit limit` (from Redis `LC_UC:{token}`) | pctChange |

- `prevClose` is loaded from Redis C1 BOD data (written by scheduler-service).
- `52WK` and `LC_UC` keys are maintained by publisher (intraday) and scheduler-service (BOD).

#### Index Bucketing

Each signal is computed separately for multiple index groups:

| Index Bucket | Source |
| --- | --- |
| `NIFTY_50` | Constituents from instrument-master `/v1/indices/NIFTY_50/constituents` |
| `NIFTY_NEXT_50` | Constituents from instrument-master |
| `NIFTY_100` | Constituents from instrument-master |
| `NIFTY_200` | Constituents from instrument-master |
| `NIFTY_500` | Constituents from instrument-master |
| `NIFTY_MIDCAP_50` | Constituents from instrument-master |
| `BANKNIFTY` | Constituents from instrument-master |
| `ALL` | All equity instruments (~7K) |

Constituent lists are loaded from instrument-master API at startup and refreshed periodically (every 6 hours).

#### Redis Writes

```
ZADD MOVERS:{category}:{index} {score} {instrumentToken}
ZREMRANGEBYRANK MOVERS:{category}:{index} 0 -{N+1}      // keep top N (default 20)
EXPIRE MOVERS:{category}:{index} 86400                   // 1-day TTL
```

**Flush cycle:** Every 60 seconds, aligned with candle production from charts-service.

**Instrument filter:** Only instruments where segment = `NSE_CASH` or `BSE_CASH` are processed. F&O instruments (82K+), indices (100+), and all other segment types are skipped.

### 6.16 Alert Engine

**What it does:** Monitors ticks against user-defined alert conditions. Runs asynchronously -- never delays the tick pipeline.

**Alert types:** Price cross (above/below), % change, volume spike, IV change, circuit hit, 52-week H/L, price band, spread alert.

Alerts are one-shot by default (trigger once, deactivate). User can set "recurring" to re-arm.

#### Redis Index Reconciliation on Restart (TASK-30)

<Warning>
**P0 blocker.** If alert-engine pod restarts, the in-memory Redis index (`token -> Set<alertId>`) is empty. But alerts still exist in PostgreSQL, and Redis C2 may have stale index keys. Without reconciliation, no alerts fire after a restart.
</Warning>

**Solution:** `AlertIndexService` implements `SmartLifecycle` to ensure correct startup ordering:

<Steps>
  <Step title="Wait for dependencies">
    SmartLifecycle ensures this runs AFTER Redis and PostgreSQL connections are established but BEFORE Kafka consumers start processing ticks.
  </Step>
  <Step title="Load all ACTIVE alerts from PostgreSQL">
    ```sql
    SELECT id, instrument_token, condition_type, threshold, user_id
    FROM alerts
    WHERE status = 'ACTIVE'
    ```
  </Step>
  <Step title="Rebuild in-memory index">
    Construct `Map<instrumentToken, Set<alertId>>` from the loaded alerts.
  </Step>
  <Step title="Reconcile with Redis C2">
    - Scan Redis for existing `ALERT_IDX:*` keys.
    - Delete orphaned Redis keys (alerts that no longer exist in PostgreSQL).
    - Add missing Redis keys (alerts that exist in PostgreSQL but are absent from Redis).
    - Update stale entries (alert definitions that changed while the pod was down).
  </Step>
  <Step title="Start Kafka consumers">
    Only after reconciliation completes do the `@KafkaListener` methods begin processing ticks. This ensures no alerts are evaluated against a stale or incomplete index.
  </Step>
</Steps>

#### Fix 52-Week and Circuit Alert Conditions Using BOD Reference Data (TASK-31)

**Problem:** `52W_HIGH`, `52W_LOW`, and `CIRCUIT_HIT` alert conditions previously relied on tick proto fields which may be stale or absent (e.g., the first tick of the day has no 52-week data embedded).

**Solution:** Maintain a BOD (Beginning-of-Day) reference data store in-memory:

- **On startup + daily refresh at 06:30 IST:** Load from Redis keys:
  - `52WK:{segment}:{instrumentToken}` -- 52-week high and low
  - `LC_UC:{segment}:{instrumentToken}` -- upper and lower circuit limits
- **Store in:** `ConcurrentHashMap<instrumentToken, BodReference>` with fields:
  - `high52w` -- 52-week high price
  - `low52w` -- 52-week low price
  - `upperCircuit` -- upper circuit limit for the day
  - `lowerCircuit` -- lower circuit limit for the day
- **Alert evaluation:** Uses this in-memory store instead of tick fields for 52W and circuit conditions.

| Condition | Evaluation |
| --- | --- |
| `52W_HIGH` | `tick.ltp >= bodRef.high52w` |
| `52W_LOW` | `tick.ltp <= bodRef.low52w` |
| `CIRCUIT_HIT` | `tick.ltp >= bodRef.upperCircuit` OR `tick.ltp <= bodRef.lowerCircuit` |

### 6.17 Self-Healing Supervisor

**What it does:** Continuously monitors every component and auto-recovers from failures.

**Health checks -- differentiated by instrument type:**

| Instrument Type | Stale Threshold | Rationale |
| --- | --- | --- |
| Liquid equity (NIFTY 50 stocks) | No tick for 5 minutes | These trade constantly |
| Mid/small cap equity | No tick for 30 minutes | Lower trading frequency |
| Illiquid scrips | No tick for 2 hours | May not trade at all some days |
| Index instruments | No tick for 30 seconds | Indices update very frequently |
| Active F&O contracts | No tick for 10 minutes | ATM options are heavily traded |
| Deep OTM options | No tick for 2 hours | Very few trades |

<Warning>
**Important:** These thresholds apply ONLY during `OPEN` market status. Outside market hours, stale detection is disabled.
</Warning>

### 6.18 Scheduler-Service (BOD & Corporate Actions)

**What it does:** Temporal workflow worker operating on two task queues. Runs all scheduled jobs: beginning-of-day data preparation and corporate action processing. Merges the former premarket-scheduler and corporate-action-processor.

#### BOD Task Queue: `premarket-scheduler-jobs` (06:00 IST)

<AccordionGroup>
  <Accordion title="TASK-01: Direct Download from Exchange URLs">
    Instead of downloading bhavcopies from S3 (legacy PML dependency), download directly from exchange websites:

    | Source | URL Pattern |
    | --- | --- |
    | NSE Equity Bhavcopy | `https://www.nseindia.com/content/historical/EQUITIES/{DD}{MON}{YYYY}/cm{DD}{MON}{YYYY}bhav.csv.zip` |
    | BSE Equity Bhavcopy | `https://www.bseindia.com/download/BhsavCopy/Equity/EQ_ISINCODE_{DDMMYY}.zip` |

    **Retry:** Exponential backoff (1s, 2s, 4s, 8s, 16s) -- exchange sites can be slow during early morning hours.

    **Fallback:** If direct download fails after 5 retries, fall back to S3 mirror (legacy path, still available as safety net).
  </Accordion>

  <Accordion title="TASK-02: Fix instrumentToken Key Identifier (P0 blocker)">
    Redis keys currently use `pmlId` which is a legacy PML concept not present in MDE's instrument-master.

    **Fix:** Replace `pmlId` with `instrumentToken` (exchange-assigned, e.g., `NSE:RELIANCE:EQ`) as the key identifier.

    **Affected Redis keys:** ALL keys written during BOD:
    - `FEED_PP:{segment}:{instrumentToken}` (was `FEED_PP_{segment}:{pmlId}`)
    - `OHLC:{segment}:{instrumentToken}` (was `MGR_{exchange}_{segment}_OHLC:{pmlId}`)
    - `52WK:{segment}:{instrumentToken}` (was `MGR_{exchange}_{segment}_52_WK_HL:{pmlId}`)
    - `LC_UC:{segment}:{instrumentToken}` (was `MGR_{exchange}_{segment}_LC_UC:{pmlId}`)
    - `PC:{segment}:{instrumentToken}` (was `MGR_{exchange}_{segment}_PC:{pmlId}`)
  </Accordion>

  <Accordion title="TASK-03: BSE Bhavcopy Support">
    Parse BSE bhavcopy CSV format (different column layout from NSE):

    | BSE Column | Mapped Field |
    | --- | --- |
    | `SC_CODE` | Security code (maps to instrumentToken via instrument-master) |
    | `PREVCLOSE` | Previous close price |
    | `OPEN` | Open price |
    | `HIGH` | Day high |
    | `LOW` | Day low |
    | `CLOSE` | Closing price |
    | `NET_TURNOV` | Volume/turnover |

    Writes BOD data to Redis for all BSE instruments: `PC`, `OHLC`, `52WK` keys.
  </Accordion>

  <Accordion title="TASK-04: NSE F&O Bhavcopy Support">
    Parse NSE F&O bhavcopy CSV format. Critical dependency for TASK-28 (day-over-day changes in FNO chain).

    **Extracted fields per contract:**
    - Previous day OI (`OPEN_INT` column)
    - Previous day close price (`CLOSE` column)
    - Previous day volume (`CONTRACTS` column)

    **Written to Redis C2:** Keyed by F&O instrument token, consumed by fno-data-writer for day-over-day change calculations.
  </Accordion>

  <Accordion title="TASK-05: F&O Security Ban List">
    NSE publishes a daily ban list for securities in the F&O segment that have breached the market-wide position limit (MWPL).

    **Workflow:**
    1. Download ban list from NSE at 06:15 IST
    2. Parse CSV: extract symbol, underlying, ban reason
    3. Write to Redis: `FNO:BAN:{instrumentToken}` with value `{"banned": true, "reason": "MWPL breach", "date": "2026-04-22"}`
    4. fno-data-writer and broadcast-service read this flag to annotate banned securities in the option chain and WebSocket stream

    **TTL:** 1 day (auto-expires, refreshed next BOD).
  </Accordion>

  <Accordion title="TASK-06: BSE Circuit Limits from DP File">
    BSE publishes a Daily Price (DP) file with circuit limits for each instrument.

    **Workflow:**
    1. Download BSE DP file during BOD (06:00 IST)
    2. Parse: extract upper/lower circuit limit for each BSE instrument
    3. Write to Redis C1: `LC_UC:{BSE_CASH}:{instrumentToken}` with `{upper, lower}`
    4. Complements intraday circuit limit updates from BSE MBP packets (extracted by publisher)

    <Note>
    NSE circuit limits arrive via the MBP packet during market hours (publisher extracts them). BSE requires this separate DP file download because BSE MBP does not always carry circuit limits for all instruments.
    </Note>
  </Accordion>
</AccordionGroup>

#### BOD Workflow Execution Order

```
06:00 IST — BOD workflow starts
    |
    +-- Step 1: Set TRD_DAY flag (check holiday calendar)
    +-- Step 2: Download bhavcopies in parallel:
    |       NSE Equity bhavcopy (TASK-01)
    |       BSE Equity bhavcopy (TASK-01, TASK-03)
    |       NSE F&O bhavcopy (TASK-04)
    |       BSE DP file (TASK-06)
    |       NSE F&O ban list (TASK-05)
    +-- Step 3: Parse all files, write BOD Redis keys:
    |       PC:{segment}:{instrumentToken}     (previous close)
    |       OHLC:{segment}:{instrumentToken}   (seed with prev close as open)
    |       52WK:{segment}:{instrumentToken}   (52-week high/low)
    |       LC_UC:{segment}:{instrumentToken}  (circuit limits)
    |       FNO:BAN:{instrumentToken}          (F&O ban list)
    +-- Step 4: Verify key counts match expected instrument count
    |
06:30 IST — BOD workflow complete
```

### 6.18.1 Corporate Action Processing (TASK-07)

<Info>
**Full pre-computation design.** Corporate actions (splits, bonus, consolidation, rights, dividends) require historical price adjustment. MDE uses a T-1 pre-computation + T-0 atomic flip strategy with ClickHouse shadow tables, ensuring zero-downtime adjustments.
</Info>

#### T-1 Pre-Computation (Evening Before Ex-Date)

**Trigger:** Temporal cron workflow `PreComputeAdjustmentWorkflow` at 19:30 IST daily (same task queue as existing CA workflow: `corporate-action-jobs`).

<Steps>
  <Step title="Query upcoming corporate actions">
    ```sql
    SELECT * FROM corporate_actions
    WHERE ex_date = CURRENT_DATE + INTERVAL '1 day'
      AND status = 'PENDING'
    ```
  </Step>
  <Step title="Compute adjustment factor">
    | Action Type | Example | Factor | Applied To |
    | --- | --- | --- | --- |
    | Stock Split | 1:5 (1 share becomes 5) | 0.2 | Multiply all historical OHLC prices by 0.2 |
    | Bonus Issue | 1:2 (1 bonus for 2 held) | 0.667 (= 2/3) | Multiply all historical OHLC prices by 0.667 |
    | Consolidation / Reverse Split | 5:1 (5 shares become 1) | 5.0 | Multiply all historical OHLC prices by 5.0 |
    | Rights Issue | Complex | Manual | Requires manual approval (price depends on rights price, ratio, market conditions) |
    | Dividend | Cash payout | (price - dividend) / price | Applied only for special dividends > 5% of price |
  </Step>
  <Step title="Read historical candles from ClickHouse">
    Read from `candles_1m` and all rollup tables (`candles_5m_mv`, `candles_15m_mv`, `candles_1h_mv`, `candles_1d_mv`) for the affected instrument token.
  </Step>
  <Step title="Apply adjustment factor to all OHLC prices">
    ```sql
    INSERT INTO candles_1m_shadow
    SELECT
        instrument_token,
        timestamp,
        open * {factor} AS open,
        high * {factor} AS high,
        low * {factor} AS low,
        close * {factor} AS close,
        volume * (1 / {factor}) AS volume,  -- volume adjusts inversely
        updated_at
    FROM candles_1m
    WHERE instrument_token = '{token}'
    ```
    Shadow tables (`candles_1m_shadow`, etc.) are structurally identical to production tables.
  </Step>
  <Step title="Verify shadow table integrity">
    - Row count in shadow matches original.
    - Spot-check: random sample of 100 candles verified against expected adjusted values.
    - No NULL or negative prices after adjustment.
  </Step>
  <Step title="Mark status: PRE_COMPUTED">
    ```sql
    UPDATE corporate_actions SET status = 'PRE_COMPUTED' WHERE id = {actionId}
    ```
  </Step>
</Steps>

#### T-0 Atomic Flip (Midnight on Ex-Date)

**Trigger:** `ApplyAdjustmentWorkflow` at 00:01 IST on ex-date.

<Steps>
  <Step title="Verify pre-computation completed">
    Check that all corporate actions for today's ex-date have `status = 'PRE_COMPUTED'`.
  </Step>
  <Step title="Atomic partition swap in ClickHouse">
    ```sql
    ALTER TABLE candles_1m REPLACE PARTITION '{partition_id}'
        FROM candles_1m_shadow
    ```
    ClickHouse partition replacement is atomic and zero-downtime. Readers see either old or new data, never a mix.
  </Step>
  <Step title="Update BOD Redis keys">
    - `PC:{segment}:{instrumentToken}` -- adjust previous close by factor
    - `52WK:{segment}:{instrumentToken}` -- adjust 52-week high and low by factor
    - `OHLC:{segment}:{instrumentToken}` -- seed new day's OHLC with adjusted previous close
  </Step>
  <Step title="Mark status: APPLIED">
    ```sql
    UPDATE corporate_actions SET status = 'APPLIED', applied_at = NOW() WHERE id = {actionId}
    ```
  </Step>
</Steps>

#### T-0 Verification (Morning)

**Trigger:** `VerifyAdjustmentWorkflow` at 06:00 IST (before market open at 09:15 IST).

<Steps>
  <Step title="Sample and verify">
    Select 100 random candles for each adjusted instrument. Verify that adjusted price = original price x factor within floating-point tolerance (1e-4).
  </Step>
  <Step title="Cross-check with exchange data">
    Compare adjusted previous close against the exchange's adjusted close from bhavcopy (if available).
  </Step>
  <Step title="Decision">
    - **Verification passes:** Mark status `VERIFIED`. Corporate action complete.
    - **Verification fails:** Auto-rollback -- swap partition back from backup. Mark status `ROLLBACK`. Alert ops team.
  </Step>
</Steps>

#### Auto-Approve Matrix

| Action Type | Auto-Approve? | Rationale |
| --- | --- | --- |
| Stock split | Yes | Simple ratio, high confidence, fully reversible |
| Bonus issue | Yes | Simple ratio, fully reversible |
| Consolidation / reverse split | Yes, if volume impact < 5% | Simple ratio, verify volume adjustment is clean |
| Rights issue | No -- manual approval required | Complex pricing depends on rights price, ratio, market conditions |
| Dividend (regular) | No adjustment needed | Regular dividends do not require historical price adjustment |
| Dividend (special > 5% of price) | No -- manual approval required | Special dividends affect historical comparability |

#### Fallback: Morning Batch Mode

If T-1 pre-computation fails (e.g., ClickHouse maintenance window, network issue):

- Fall back to T-0 morning batch (legacy mode, similar to how Zerodha handles corporate actions).
- Morning batch triggers at 06:00 IST, before market open at 09:15 IST.
- Takes approximately 30 minutes for full history adjustment -- still completes well before market open.
- Ops team is alerted that T-1 pre-computation failed and morning batch is running.

### 6.19 Expiry Day Cleanup

**What it does:** Cleans up expired derivative contracts from Redis and recalculates dependent metrics after expiry settlement.

**Temporal Workflow: ExpiryCleanupWorkflow**

```
Trigger: Market close status on expiry days
    - Weekly: Thursday (NIFTY, BANKNIFTY, FINNIFTY weekly options)
    - Monthly last Thursday: stock options + monthly futures
    - Quarterly: index futures (Mar, Jun, Sep, Dec)

Timeout: 5 minutes
Expected completion: < 30 seconds

Step 1: Identify expiring contracts from PostgreSQL
    -> SELECT * FROM instruments
      WHERE expiry_date = CURRENT_DATE
        AND scrip_type IN ('OPTION', 'FUTURE')
        AND status = 'ACTIVE'
    -> Typically 2,000-5,000 contracts on weekly expiry
    -> Up to 20,000 contracts on monthly expiry (all stock options)

Step 2: Mark as EXPIRED in PostgreSQL
    -> UPDATE instruments SET status = 'EXPIRED' WHERE id IN (...)
    -> Prevents Feed Handler from processing stale ticks for these instruments

Step 3: Clean expired data from Redis Cluster 2
    -> Remove expired instruments from FNO option chain hashes
    -> Remove expired instruments from Market Movers sorted sets
    -> Remove expired instruments from option insights hashes
    -> Remove from FNO:MAX_PAIN, FNO:ATM_STRADDLE, FNO:PCR keys for affected expiries
    -> HDEL / ZREM operations, batched

Step 4: Stop FNO Data Writer from computing Greeks for expired contracts
    -> Publish expiry event to Kafka (topic: instrument-expiry-events)
    -> FNO Data Writer removes expired instruments from its in-memory computation set
    -> No more Black-Scholes/Black-76 calculations for expired strikes

Step 5: Recalculate PCR / Max Pain using only live contracts
    -> Trigger FNO Data Writer to recompute option insights for affected underlyings
    -> PCR (OI/Volume) recalculated excluding expired contracts
    -> Max Pain recalculated with remaining strikes only
    -> ATM Straddle updated if ATM strike was an expiring contract

Step 6: Propagate settlement prices to Portfolio/Ledger Engine via Kafka
    -> Publish settlement event: {instrumentToken, settlementPrice, expiryDate}
    -> Topic: instrument-settlement-events
    -> Portfolio Engine uses this for P&L calculation
    -> Ledger Engine uses this for margin release
```

**Applies to:**

- Weekly NIFTY/BANKNIFTY/FINNIFTY option expiry (every Thursday)
- Monthly stock option expiry (last Thursday of month)
- Monthly futures expiry (last Thursday of month)
- Quarterly index futures expiry (last Thursday of Mar/Jun/Sep/Dec)

### 6.20 Broadcast Service Gaps (TASK-16 through TASK-20)

<Info>
The following tasks address gaps in broadcast-service's WebSocket streaming capabilities.
</Info>

<AccordionGroup>
  <Accordion title="TASK-16: Market Depth (L5) Streaming">
    broadcast-service currently streams only LTP/OHLC snapshots. Market depth (top 5 bid/ask levels) must be added to the WebSocket stream.

    - **Source:** Publisher writes `DEPTH:{segment}:{instrumentToken}` to Redis C1 on every MBP tick.
    - **Stream:** broadcast-service includes depth data in the Protobuf WebSocket frame when the client subscribes with `depth=true`.
    - **Throttle:** Depth updates are sent at most once per 500ms per instrument to limit bandwidth.
    - **Payload:** 5 bid levels + 5 ask levels, each with price, quantity, and order count.
  </Accordion>

  <Accordion title="TASK-17: Index Tick Streaming">
    Index ticks (NIFTY, SENSEX, etc.) must be streamed via WebSocket alongside equity ticks.

    - **Source:** Kafka topics `mde-ticks-nse-index`, `mde-ticks-bse-index`.
    - broadcast-service subscribes to index tick topics in addition to equity/derivative topics.
    - Client subscription: `subscribe({tokens: ["NSE:NIFTY_50:INDEX", ...], depth: false})`.
    - Cold-start snapshot from Redis C1 `FEED_PP:{segment}:{instrumentToken}` for indices.
  </Accordion>

  <Accordion title="TASK-18: F&O Ban List Annotation">
    When an instrument is in the F&O security ban list (TASK-05), broadcast-service annotates the WebSocket tick frame with `banned: true`.

    - On startup: load all `FNO:BAN:*` keys from Redis C1 into a local `Set<instrumentToken>`.
    - On each F&O tick: check if instrument is in the ban set. If yes, set `banned = true` in the Protobuf frame.
    - Refresh ban set at 06:30 IST daily (after scheduler-service BOD completes).
  </Accordion>

  <Accordion title="TASK-19: Market Status Broadcast">
    Broadcast market status changes (PRE_OPEN, OPEN, CLOSE, POST_CLOSE) to all connected WebSocket clients.

    - Publisher writes `MARKET_STATUS` to Redis C1 on every status transition.
    - broadcast-service polls `MARKET_STATUS` every 5 seconds and broadcasts to all clients on change.
    - Client receives: `{"type": "MARKET_STATUS", "status": "OPEN", "timestamp": "..."}`
  </Accordion>

  <Accordion title="TASK-20: Heartbeat and Idle Connection Cleanup">
    Server-to-client heartbeat for detecting and cleaning up idle/zombie connections.

    - Server sends `PING` frame every 30 seconds to each connected client.
    - If no `PONG` response within 10 seconds, connection is marked stale.
    - After 3 consecutive missed PONGs (90 seconds total), connection is force-closed.
    - Metrics: `ws_connections_active`, `ws_connections_closed_idle` Prometheus counters.
  </Accordion>
</AccordionGroup>

### 6.22 Resilience & Zero-Error User Experience

<Info>
This section codifies the platform-wide promise that **users should NEVER see an error**. If a backend component fails, the system degrades silently. The user always sees data -- possibly slightly stale, possibly partial -- never an error popup, never a red modal, never a "Something went wrong" screen.
</Info>

**Design Principle:** "Users should NEVER see an error. If a backend component fails, the system degrades silently."

This is not a soft target. Every charts-service call, every WebSocket subscription, every Market Movers request, every Option Chain query must produce a useful response within the latency budget -- even when the upstream is down, slow, or returning garbage. The mechanisms below describe how each request path enforces this contract.

#### 6.22.1 Layered Fallback Strategy

For every user-facing data path, the response is produced by walking down a 4-layer fallback ladder. The first layer that returns data wins. The user always gets *something*.

| Layer | Source | Freshness | When Used |
| --- | --- | --- | --- |
| **Layer 1 (primary)** | Live data from Kafka / Redis Cluster 1 (Feed) | Real-time (&lt;100ms old) | Normal operation |
| **Layer 2 (cached)** | Last-known-good snapshot from Redis Cluster 2 (API) | Seconds-old | Layer 1 timeout, partial outage |
| **Layer 3 (stale)** | Most recent snapshot from ClickHouse | Minutes/hours-old, marked "Last updated: Xs ago" | Layer 2 unavailable, cluster outage |
| **Layer 4 (final)** | Empty state with skeleton/loading UI | None | Total backend outage -- no error popup, just placeholder |

**How it works in practice:**

- Each API endpoint declares its fallback chain at startup.
- The fallback runtime wraps every downstream call with a deadline (e.g., 30ms for Layer 1).
- If Layer 1 misses the deadline OR throws, the runtime tries Layer 2 with a fresh deadline.
- If Layer 2 fails, Layer 3 is consulted.
- If Layer 3 also fails, Layer 4 returns an empty-but-structurally-valid response -- never a 5xx, never an exception.
- Each downgrade is recorded in the response metadata (`source: "L2_CACHED"`, `staleAgeSec: 12`) so the frontend can render a freshness indicator.

#### 6.22.2 Stale-While-Revalidate Pattern

APIs always return SOMETHING useful within 50ms. They never block on a slow upstream.

- **Synchronous path:** Serve the most recent cached value immediately. Tag the response with `stale: true` if the cache is older than the per-API freshness threshold.
- **Background refresh:** A non-blocking background task fetches the latest from the upstream. When it arrives, the cache is updated. The next request gets fresh data.
- **Result:** The user always sees data immediately. Staleness is exposed as metadata (e.g., "Updated 8s ago") -- not as an error.

```
Request arrives at charts-service
    |
Check Redis Cluster 2 cache (current-day candles)
    |
Cache hit (any age) -> respond IMMEDIATELY with cached value
                      |
                      If cache age > 5s -> kick off background refresh
                                          (non-blocking -- does not delay this response)
    |
Cache miss -> try ClickHouse synchronously (Layer 3)
           -> respond with ClickHouse data + stale flag
           -> kick off background refresh to repopulate cache
```

This pattern guarantees that **the response to the user never waits for an upstream that may be slow or down**. When upstream recovers, the next request silently gets fresh data.

#### 6.22.3 Circuit Breakers (Resilience4j)

Every downstream call is wrapped with a Resilience4j circuit breaker. When too many calls fail or time out, the circuit opens and subsequent calls are short-circuited to the fallback path -- protecting the failing service and the calling service alike.

**Circuit-breaker behavior:**

- **CLOSED** (normal): Calls pass through. Failures are tracked in a sliding window.
- **OPEN** (failing): Calls short-circuit immediately to fallback. The downstream is not hit at all.
- **HALF-OPEN** (probing): After cooldown, a small number of test requests are allowed through. If they succeed, circuit returns to CLOSED. If they fail, it returns to OPEN with extended cooldown.

**Per-service thresholds:**

- **Failure rate threshold:** 50% over a rolling window of 20 requests -> trip OPEN
- **Cooldown ladder (exponential):** First open -> 10s, second open in same hour -> 30s, third -> 60s
- **Half-open probe count:** 3 requests allowed through during HALF-OPEN

**Wrapped paths (examples):**

- charts-service -> Redis Cluster 2 (current-day candles)
- charts-service -> ClickHouse (historical candles)
- fno-data-writer -> Redis Cluster 2 (option chain)
- market-movers-writer -> Redis Cluster 2 (sorted sets)
- broadcast-service -> Redis Cluster 1 (snapshot on connect)
- charts-service -> ClickHouse (writes -- circuit-breaker prevents WAL buildup if DB is slow)
- fno-data-writer -> Redis Cluster 2 (writes)
- Any cross-service HTTP call

**Why this matters:** Without circuit breakers, a slow downstream creates a cascading failure -- every caller blocks on it, threads exhaust, the calling service itself becomes unavailable. The breaker keeps the calling service responsive even when downstream is in trouble.

#### 6.22.4 Silent Feature Flags

When a backend component is degraded or down, the affected UI widgets are **hidden silently** -- not replaced with an error banner, not greyed out with a sad face, just gone. The user perceives a slightly simpler page, not a broken one.

**Feature-flag triggers (auto-managed by ops-service):**

| Component Down | Flag Toggled | UI Effect |
| --- | --- | --- |
| fno-data-writer down | `fno.dashboard.greeks.enabled = false` | Greeks columns hidden from option chain. Rest of chain still visible. |
| market-movers-writer down | `markets.movers.live.enabled = false` | Market Movers tab shows EOD list with "Updated at market close" caption. No error. |
| charts-service writer thread behind on writes | `charts.intraday.enabled = false` | Today's candle hidden; historical chart still renders. |
| Option Insights computation stalled | `fno.dashboard.insights.enabled = false` | PCR/Max Pain widgets hidden. Option chain unaffected. |
| alert-engine down | `alerts.creation.disabled = true` | "Create Alert" button hidden. Existing alerts unaffected. |

**Key properties:**

- **Auto-triggered by ops-service** -- when anomaly detection flags a component, the feature flag flips automatically. No human in the loop.
- **Auto-restored** -- when the component recovers, the flag flips back. No manual ops action.
- **Silent at the UI layer** -- frontend treats `flag = false` as "render nothing for this widget". No banner, no popup, no toast.
- **Self-hosted feature-flag service** (Unleash) -- no external dependency on critical path.

#### 6.22.5 Request Hedging

For latency-critical reads (live price, option chain), the API sends parallel requests to two Redis replicas and returns whichever responds first. This eliminates long-tail latency caused by occasional GC pauses or replica hiccups.

**How it works:**

- The Redis client is configured with two read endpoints (replica A, replica B).
- For each request, both replicas receive the call simultaneously.
- The first valid response wins; the other is discarded.
- Client-side latency is `min(latency_A, latency_B)` -- almost always faster than waiting on a single replica.
- Outbound load is doubled, but reads are cheap and Redis is provisioned with headroom.

**Where applied:**

- WebSocket snapshot on connect (broadcast-service -> Redis Cluster 1)
- charts-service current-day candle reads (charts-service -> Redis Cluster 2)
- Option Chain reads (fno-data-writer -> Redis Cluster 2)
- Market Movers reads (market-movers-writer -> Redis Cluster 2)

**Where NOT applied:**

- Writes (would double-write and cause consistency problems)
- Expensive aggregation reads (would double the work for marginal benefit)

#### 6.22.6 Client-Side Resilience

The frontend (mobile + web) is built to assume backend instability and recover transparently.

- **WebSocket auto-reconnect** with exponential backoff: 1s -> 2s -> 4s -> 8s -> 16s -> 30s (capped). On reconnect, fetch fresh snapshot from REST endpoint, then resume stream.
- **Mobile app caches last known state on close** -- when user reopens the app, the last-seen prices render instantly from local storage; live updates layer in as the WebSocket connects.
- **Progressive data loading** -- LTP first (smallest payload, most important), market depth second, Greeks last. If Greeks never arrive, the option chain still renders with LTP and depth.
- **Skeleton loading states** -- never a spinner that says "Loading..." indefinitely. After 3 seconds, render skeleton placeholder rows; data fills in as it arrives.
- **No error modals on transient failures** -- only on user-action errors (e.g., "Symbol not found"). Backend issues never produce modals.

#### 6.22.7 Per-API Fallback Table

The mandatory fallback chain for each user-facing API:

| API | Primary (Layer 1) | Layer 2 | Layer 3 | Final (Layer 4) |
| --- | --- | --- | --- | --- |
| **charts-service** | Redis Cluster 2 (current-day candles) | ClickHouse (`candles_1m`) | Previous day's candles from ClickHouse | Empty chart with "Data updating" message |
| **fno-data-writer** | Redis Cluster 2 (live option chain) | Last successful snapshot (in-memory cache) | Previous close values + lot size from Instrument Master PG | Hide widget silently via feature flag |
| **market-movers-writer** | Redis Cluster 2 (sorted sets, refreshed every 60s) | Last snapshot (in-memory cache) | Last 30-minute snapshot from ClickHouse | Static "trending today" list from EOD |
| **WebSocket LTP stream** | Kafka live tick (via broadcast-service) | Redis Cluster 1 snapshot | ClickHouse last tick (1m candle close) | Mark as stale, freeze price, show "Last updated: Xs ago" |
| **Instrument Search** | PostgreSQL (instruments table) | Local ConcurrentHashMap (in-memory) | Stale list (last sync) | Common tickers from cache (e.g., NIFTY 50 + top 100) |
| **Option Insights** | Redis Cluster 2 (PCR/Max Pain/ATM Straddle) | Last successful computation (in-memory) | EOD insights from ClickHouse | Hide widget silently |

#### 6.22.8 "Never Show Error" Enforcement

This is the operational rule that makes the design real:

- **All error responses converted to 200 OK.** A failed downstream lookup returns:

```json
{
  "data": null,
  "stale": true,
  "source": "FALLBACK_L4",
  "lastUpdatedMs": 1713168900000,
  "message": "Updating"
}
```

Not a 5xx, not a 4xx for backend issues. The frontend treats `null` data as a loading state, never as an error.

- **HTTP 5xx errors are logged + sent to ops-service but NEVER surfaced to the client.** The API gateway has a global error handler that catches any uncaught exception and converts it to the structure above.
- **Frontend rules:**
  - `null` data -> render skeleton or last-known-good
  - `stale: true` -> render data with subtle freshness indicator (timestamp footer, no banner)
  - HTTP failure -> silent retry with backoff, no UI change unless data has been missing for >10s, then show subtle "Reconnecting" footer
- **No exceptions for "edge cases."** Even on cold-start, even during rolling deploys, even during a Redis Cluster failover -- the contract holds.

**Audit:** Every Layer 2/3/4 hit is recorded in a `fallback_invocations` Prometheus counter, labeled by API + layer. If any API hits Layer 4 more than 0.1% of requests in a 5-minute window, an ops-service alert fires.
