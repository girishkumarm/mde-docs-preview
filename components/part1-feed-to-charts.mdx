## 6. Component Deep Dives — Part 1 (Instrument Master to Charts)

### 6.1 Instrument Master Service (Scrip Master)

**What it does:** Maintains the authoritative list of all tradeable instruments across ALL segments (equity, ETF, REIT, InvIT, bond, index, futures, options, currency). Exposes a REST API for reading instrument data. Triggers a Temporal workflow for daily sync.

**Why it matters:** Without an up-to-date instrument master, the Feed Handler doesn't know what token `26009` means. The scrip master tells us it's `NSE:RELIANCE:EQ`.

**Data sources:**

- **NSE/BSE CSV files** (7-8 files per day): security master CSVs per segment
- NSE CM bhavcopy (equity master)
- NSE FO contract file (futures + options)
- NSE CDS contract file (currency derivatives)
- BSE equity master
- BSE FO contract file
- BSE index master
- ETF/REIT/InvIT/Bond master
- **CMOTS API** (`paytmapi.cmots.com/api`): company-level information for each listing

**Database: PostgreSQL (dedicated instance — separate from the ClickHouse time-series instance)**

Two tables:

```sql
-- Table 1: Instruments (scrips) — covers ALL ScripTypes
CREATE TABLE instruments (
    id                  BIGSERIAL PRIMARY KEY,
```

**ScripMasterDto (current contract — matching existing schema):**

```java

String exchange
String segment

String isin

```

**In-memory instrument map (no Redis copy):**

Scrip master data lives in PostgreSQL only. Services load instrument map from PostgreSQL into a local in-memory `ConcurrentHashMap` on startup. No Redis copy needed — this eliminates the legacy `MGR_SM_*` JSON blob approach. The in-memory map is keyed by `(exchange, securityId)` and provides O(1) lookup during tick processing.

**Temporal Workflow: InstrumentMasterSyncWorkflow**

```
Trigger: Daily at 6:00 PM IST (after market close)
Timeout: 1 hour

Step 1: Download CSV files from NSE + BSE (7-8 files from S3)
    S3 buckets: stg-scrip-master, nse-cm-bhavcopy-{date}, bse-cm-bhavcopy-{date},
                nse-fo-bhavcopy-{date}, bse-fo-bhavcopy-{date},
                nse-cm-security-{date}, bse-cm-dp-{date},
                nse-fo-contract-{date}, bse-fo-eqd-dp-{date}

Step 2: Parse all CSVs → unified instrument records (covering ALL ScripTypes)

Step 3: For each instrument with a company mapping, call CMOTS API
    → GET /api/company/{co_code}
    → Fetches: company name, sector, industry, financials
    → Rate-limited: batch requests, 100/sec
    → Store in companies table (UPSERT by co_code)

Step 4: Diff against current instruments table
    → NEW instruments (new F&O contracts, IPOs) → INSERT
    → REMOVED instruments (expired options, delisted) → status = INACTIVE
    → CHANGED instruments (lot size, circuit limit) → UPDATE

Step 5: Log summary
    → "Added 500, removed 300, updated 50, companies synced: 5,200"

Step 6: Notify downstream services to reload in-memory instrument maps
    → Publish event to Kafka topic (instrument-master-updated)
    → Services listening: Feed Handler, Publisher, all APIs
```

```

GET /v1/instruments?symbol=RELIANCE                → search by symbol
```

### 6.2 Feed Handler

**What it does:** Receives raw binary market data from both NSE Direct UDP multicast and Omnenest RS protocol simultaneously, decodes the proprietary PML binary format into structured objects, deduplicates across sources, and routes to the correct processor. Handles LZO-compressed packets, OI injection for F&O trade ticks, pre-open LTP fallback, cross-minute dedup re-sends, index previous-close enrichment, and post-close grace windows.

**Implementation:**

- One Java 21 virtual thread per UDP stream (6 streams for NSE Direct + 6 streams for Omnenest = 12 threads)
- `MulticastSocket.receive()` → `DatagramPacket` (8192 byte buffer)
- LZO decompression layer (detects and decompresses NSE compressed packets before parsing)
- Parse `BCastHeaderBroadcast` (64 bytes) first from every datagram
- Cross-source dedup: `ConcurrentHashMap<(instrumentId, timestamp), sourceId>` — first arrival wins
- Route by `packetCode` to the correct processor
- Set `uniqueSequenceId` as MDC `traceId` for every packet

<AccordionGroup>

<Accordion title="LZO Decompression Layer (TASK-08 — P0 Blocker)">

NSE sends LZO-compressed packets for large MBP updates (multi-level depth snapshots that exceed the standard UDP MTU). Without LZO decompression, approximately 15% of NSE packets are silently dropped, causing missing ticks for actively traded instruments.

**Detection and flow:**

```
DatagramPacket received (8192 byte buffer)
    │
    ├─ Check first 4 bytes for LZO magic header: 0x894C5A4F
    │
    ├─ If magic matches → LzoDecompressor.decompress(payload)
    │     └─ Uses lzo-java library (pure Java, no native dependencies)
    │     └─ Output: raw binary bytes (same format as uncompressed packets)
    │
    └─ If no magic → pass through unchanged (already uncompressed)
    │
    ▼
Parse BCastHeaderBroadcast (64 bytes) from decompressed bytes
```

**Key implementation details:**

- `LzoDecompressor` sits in the decoder layer, before protobuf deserialization and before header parsing
- Uses the `lzo-java` library (pure Java, zero native deps — important for containerized deployment)
- Magic header check is a single 4-byte integer comparison — zero overhead for uncompressed packets
- Decompressed output is identical in format to uncompressed packets — no downstream changes required

**Metric:**

```
mde_feed_lzo_decompress_duration_seconds (histogram)
    labels: exchange={NSE,BSE}
```

</Accordion>

<Accordion title="OI Injection into MBP Trade Emit (TASK-09)">

When an MBP packet has `tradePrice > 0` AND comes from an F&O segment (NSE_D or BSE_D), the Feed Handler injects the current Open Interest value from `OpenInterestProcessor`'s in-memory map into the tick before publishing.

**Flow:**

```
MbpProcessor receives MBP packet
    │
    ├─ Check: tradePrice > 0 AND segment is F&O?
    │
    ├─ YES → Look up latest OI for same token from OpenInterestProcessor's
    │        in-memory ConcurrentHashMap<Integer, Long> (keyed by securityId)
    │     └─ Set openInterest field on Tick proto before publishing
    │
    └─ NO → Leave openInterest as-is (0 for equity, or whatever packet contains)
    │
    ▼
Publish tick with OI populated
```

**Why this matters:** This is how legacy PML populates OI on trade ticks. Without this injection, fno-data-writer receives trade ticks with zero OI, breaking option chain displays and OI change calculations.

</Accordion>

<Accordion title="Zero-LTP Fallback for Pre-Open Session (TASK-10)">

During pre-open session (07:00-09:15 IST), the LTP field in MBP packets is 0 because no trades have occurred yet. This causes downstream services to see LTP=0 and compute -100% change values.

**Fallback logic:**

```
if (tick.ltp == 0 AND marketStatus == PRE_OPEN):
    previousClose = redis.hget("FEED_PP:{token}", "pc")  // Redis C1
    tick.ltp = previousClose
    tick.isPreOpenFallback = true  // metadata flag
```

**Source of previous close:** Redis Cluster 1, key `FEED_PP:{token}`, field `pc` (previous close) — written by the scheduler-service during BOD (Beginning of Day) data load.

</Accordion>

<Accordion title="Cross-Minute Re-send for Deduplicated Ticks (TASK-11)">

The `SourceDeduper` drops duplicate sequence numbers within the same minute. Problem: at a minute boundary, a tick deduped at 09:15:59 needs to be re-sent at 09:16:00 so the charts-service can open a new 1-minute candle.

**Solution:**

```
Per-token last-seen tick tracking:
    lastSeenTick: ConcurrentHashMap<Integer, TickSnapshot>  // securityId → last tick

On every tick after dedup:
    lastSeenTick.put(securityId, currentTick)

Minute rollover detection (wall clock):
    currentMinute = System.currentTimeMillis() / 60_000
    if (currentMinute != previousMinute):
        for each (securityId, tick) in lastSeenTick:
            re-emit tick with timestamp = minuteBoundary
        previousMinute = currentMinute
```

This ensures charts-service always has at least one tick per minute per active instrument, preventing gaps in the candle timeline.

</Accordion>

<Accordion title="Previous-Close Enrichment for Index Ticks (TASK-12)">

Index ticks from NSE do not carry previous close in the packet. Without enrichment, percentage change calculations for indices (NIFTY 50, BANK NIFTY, FINNIFTY, etc.) are impossible.

```
IndexProcessor receives index tick:
    │
    ├─ Look up previous close from Redis C1:
    │     key: FEED_PP:{indexToken}
    │     field: pc (previous close)
    │
    ├─ Set tick.previousClose = redisValue
    │
    └─ Publish enriched index tick
```

**Dependency:** Scheduler-service must write index previous close to Redis during BOD. If Redis lookup fails, the tick is published without previous close (graceful degradation — % change shows as N/A on frontend).

</Accordion>

<Accordion title="Post-Close Timing Window for Trade Capture (TASK-13)">

After market close at 15:30 IST, closing auction trades arrive for approximately 5 minutes. If `MarketStatusState` transitions to CLOSED immediately at 15:30, these closing auction trades are lost.

**Grace window:**

```
Market status state machine:
    OPEN (09:15 - 15:30)
        │
        ├─ 15:30 IST → transition to OPEN_GRACE (NOT CLOSED)
        │   └─ Grace window: 5 minutes (configurable via mde.market.close-grace-minutes)
        │   └─ Continue accepting and processing all ticks normally
        │
        ├─ 15:35 IST → transition to POST_CLOSE
        │   └─ Only volume/trade updates accepted, no new price levels
        │
        └─ 15:40 IST → transition to CLOSED
            └─ All tick processing stops
```

This prevents missing closing auction trades which are critical for accurate closing prices and EOD candle calculations.

</Accordion>

</AccordionGroup>

| 4 | 4 | headerLength | int | Always 64 |
| 8 | 4 | packetLength | int | Total packet size |

| 16 | 4 | exScripId | int | Exchange security ID |
| 44 | 8 | pmlId | long | PML internal instrument ID |

**Inner Header (BCastHeader, 11 bytes):**

| Offset | Length | Field | Type |
| --- | --- | --- | --- |
| 0 | 1 | exchSeg | byte |
| 1 | 4 | scripId | int |
| 5 | 4 | scripId2 | int |
| 9 | 1 | msgLength | byte |
| 10 | 1 | msgCode | byte |

**18 transaction codes route to 6 processors:**

| Processor | Transaction Codes | What it handles |
| --- | --- | --- |
| MbpProcessor | 111002, 112002, 211002, 212002 | Market depth + trade (equity + derivatives). Also handles BSE circuit limits at byte offsets 264/268. Injects OI for F&O trade ticks (TASK-09). |
| CircuitLimitProcessor | 111006, 112006 | NSE circuit limits |
| MarketStatusProcessor | 111001, 112001, 211001, 212001 | Market open/close/halt. Implements post-close grace window (TASK-13). |
| IndexProcessor | 111003, 211003 | Index values. Enriches with previous close from Redis (TASK-12). |
| OpenInterestProcessor | 112004, 212004 | Derivative open interest. Maintains in-memory OI map consumed by MbpProcessor (TASK-09). |
| HeartbeatProcessor | 111007, 112007, 211007, 212007 | Keep-alive (no Redis write, no publish) |

<Note>
**Note:** BSE circuit limits (211006, 212006) are embedded in MBP packets at byte offsets 264/268 — handled by MbpProcessor, not CircuitLimitProcessor. See TASK-15 for how BSE circuit limits are written to Redis.
</Note>

### 6.3 Duplicate Detector

**What it does:** Filters out unchanged/redundant ticks. Exchanges send periodic MBP refreshes even when nothing changed. Also deduplicates across NSE Direct and Omnenest sources (cross-source dedup). Handles cross-minute re-sends (TASK-11) to ensure downstream candle services always have at least one tick per minute.

```
Cross-source dedup (first stage — in Feed Handler):
    key = (instrumentId, timestamp)
    if localCache.putIfAbsent(key, sourceId) != null:
        → SKIP (duplicate from second source)
        → metric: cross_source_dedup_count++

Per-source dedup (second stage — per processor):
    For each decoded packet:
        prev = localCache.get(cacheType, key)
        if prev != null
           AND tick.ltp == prev.ltp
           AND tick.volume == prev.volume
           AND tick.avgTradePrice == prev.avgTradePrice:
            → SKIP (no further processing)
            → metric: duplicate_ticks_skipped++
        else:
            → Update localCache
            → Track as lastSeenTick for cross-minute re-send (TASK-11)
            → Pass to Publisher + Producer

Cross-minute re-send (TASK-11):
    On minute boundary rollover (wall clock check):
        For each token in lastSeenTick map:
            → Re-emit the last tick with updated timestamp
            → Ensures charts-service always has >= 1 tick per minute per instrument
            → metric: cross_minute_resend_count++

Additional dedup rules:
    Index packets: Skip if new LTP == cached LTP
    OI packets: Skip if OI unchanged, or if volume/OHLC are all zero (stale)
```

Impact: Cross-source dedup eliminates ~50% of incoming ticks (duplicates from second source). Per-source dedup eliminates ~30-40% of remaining ticks. Combined: ~65-70% reduction.

### 6.4 Bad Tick Detection — Future Consideration

<Note>
**Note:** The current production system does not implement bad tick detection, and the existing team has confirmed they don't have this capability. This section documents a **potential future enhancement**, not a launch requirement.
</Note>

**If implemented in a future phase, detection would work as:**

- Compare incoming LTP against a rolling EWMA (exponentially weighted moving average)
- If deviation exceeds a threshold (e.g., 10% for equity, 15% for F&O), flag as suspicious
- Hold the tick, wait for confirmation from subsequent ticks
- Release if confirmed real (e.g., pre-open auction gap), discard if anomaly

**Recommendation:** Defer to Phase 2. The current system operates without it successfully. Focus launch effort on feed reliability and data accuracy through reconciliation instead.

### 6.5 Publisher

**What it does:** After ticks pass through the processing pipeline, the Publisher writes to Redis Cluster 1 (Feed) as the live snapshot store. Also writes market status keys (TASK-14) and BSE circuit limits extracted from MBP packets (TASK-15).

**Processing pattern (every processor follows this):**

```java
// 1. Write bytes to local in-memory cache (ConcurrentHashMap)
localCache.getFeedPpCache(cacheType).put(key, serializedBytes);

// 2. Enqueue for async batched Redis HSET
redisBatchFactory.getRedisQueueProcessor(cacheType)
    .addMessageToQueue(new QueueMessage(key, timestamp));
```

**Output: Redis Hash (snapshot store — current state) — written to Redis Cluster 1 (Feed)**

| Redis Hash Key Pattern | Hash Field | Value Type | Model |
| --- | --- | --- | --- |
| `FEED_PP:{token}` | `{scripId}_{exchSegCode}_{msgCode}` | binary byte[] | Raw processed broadcast packet (MbpBroadcast, IndexBroadcast, etc.) |
| `LC_UC:{token}` | `uc`, `lc` | String | Circuit limits (NSE: from circuit limit packets; BSE: extracted from MBP at byte offsets 264/268 — TASK-15) |
| `52WK:{token}` | `high`, `low` | JSON byte[] | `YearlyHighLowDto {"52_wk_high": float, "52_wk_low": float}` |
| `OHLC:{token}` | `o`, `c`, `h`, `l` | JSON byte[] | `OhlcDto {"o": float, "c": float, "h": float, "l": float}` |
| `PC:{token}` | `pc`, `poi` | JSON byte[] | `PcloseDto {"pc": float, "poi": float}` |
| `TRD_DAY` | varies | String | Trading day status |
| `MKT_STATUS:{exchange}` | — | JSON String | Market status (TASK-14) |

Exchange segment codes: `NSE_E` (1), `NSE_D` (2), `NSE_I` (0), `BSE_E` (4), `BSE_D` (8), `BSE_I` (0).

<Accordion title="BSE Circuit Limits from MBP Packet (TASK-15)">

BSE does not have a separate DP file for intraday circuit limit updates (unlike NSE which sends dedicated circuit limit packets). Instead, BSE embeds circuit limits in MBP packets at byte offsets 264 (upper) and 268 (lower).

**Write to Redis C1:**

```
Key:    LC_UC:{token}
Fields: uc (upper circuit limit), lc (lower circuit limit)

Example:
    HSET LC_UC:532540 uc 1550.00 lc 1250.00
```

**Optimization:** Only write to Redis if the circuit limit values have actually changed compared to the in-memory cache. This avoids unnecessary Redis writes for every BSE MBP packet (which arrive at high frequency).

```java
CktLimitDto cached = localCache.getBseCircuitLimit(securityId);
if (cached == null || cached.uc != newUc || cached.lc != newLc) {
    localCache.putBseCircuitLimit(securityId, new CktLimitDto(newUc, newLc));
    redisDao.hset("LC_UC:" + bseToken, Map.of("uc", newUc, "lc", newLc));
    metrics.counter("mde_bse_circuit_limit_updates_total").increment();
}
```

</Accordion>

</AccordionGroup>

**Batched Redis writes (RedisBatchProcessor):**

- One `RedisBatchProcessor` per `CacheType` (NSE_E, NSE_D, NSE_I, BSE_E, BSE_D, BSE_I)
- Holds a `LinkedBlockingQueue<QueueMessage>`
- Flush triggers: queue size > `redis.max.batch.size` OR 10ms poll timeout
- Single `HSET` for the whole batch via `redisDao.putAll(Map<byte[], byte[]>, hashKey)`
- `@Retryable(maxAttempts=6, backoff=exponential 500ms→60s)`
- Bulk insert page size: `redis.bulk.insert.size=1000`

**Lua Script — Atomic Compare-and-Set (`feed_broadcast_scrip_data.lua`):**

For 52-week H/L and OHLC, we need conditional updates (only write if new value beats existing):

| Request Code | Field | Byte Offset | Write Condition |
| --- | --- | --- | --- |
| `36_H` | 52-wk high | 11 | new > existing |
| `36_L` | 52-wk low | 15 | new < existing |
| `3_O` | OHLC open | 8 | only if changed |
| `3_H` | OHLC high | 16 | new > existing |
| `3_L` | OHLC low | 24 | new < existing |

**MBP Processing — Valid Trade Gate:**

- Volume > 0 AND market status is `OPEN` (3.0), `OPEN_GRACE`, or `POST_CLOSE_START` (5.0)
- After `CLOSE` (4.0): stop all price updates
- After-market-close (status 6): trade volume cache updates skipped entirely

<Note>
**Headless publisher:** The new publisher has no REST API. It is a headless background process — it reads from the feed processor pipeline and writes directly to Redis C1. The legacy `/periscope/v1` and `/s2s/v1` API surfaces belong to the old legacy publisher and are not present in the new architecture. Operational controls (manual overrides, cache reloads) are handled by the ops-service.
</Note>

### 6.5.1 Redis Topology

Two separate Redis clusters, each dedicated to a specific access pattern:

**Redis Cluster 1 — Feed (hot path, low-latency writes)**

| Hash Key Pattern | Writers | Readers |
| --- | --- | --- |
| `FEED_PP:{token}` | Publisher | Broadcast Service (snapshot on connect), charts-service |
| `OHLC:{token}` | Publisher | Broadcast Service (snapshot on connect), charts-service |
| `PC:{token}` | Publisher | Broadcast Service (snapshot on connect), scheduler-service |
| `LC_UC:{token}` | Publisher (NSE via circuit-limit packets; BSE via MBP byte offsets 264/268 — TASK-15) | Broadcast Service (snapshot on connect), charts-service |
| `52WK:{token}` | Publisher | Broadcast Service (snapshot on connect), charts-service |
| `TRD_DAY` | Publisher | Temporal jobs, all services |
| `MKT_STATUS:{exchange}` | Publisher | Broadcast Service (TASK-18), ops-service |

**Redis Cluster 2 — API (derived data, higher-volume reads)**

| Key Pattern | Writers | Readers |
| --- | --- | --- |
| `candle:1m:{token}:{epochMinute}` (1m candle hash) | charts-service | charts-service (REST API) |
| `candle:idx:{token}` (candle index sorted set) | charts-service | charts-service (REST API) |
| FNO option chain hashes | FNO Data Writer | FNO Dashboard API |
| FNO option insights hashes | FNO Data Writer | FNO Dashboard API |
| `mkt_mvs_{category}_{exchange}_{index_id}` (sorted sets) | Market Movers Writer | Market Movers API |

**Configuration rules for both clusters:**

- `maxmemory-policy noeviction` — never silently drop data; fail loudly if memory is full
- Alert at **80% memory usage** — trigger capacity review
- Use **key prefixes only** (no Redis DB numbers). All data in DB 0 — required for future Redis Cluster mode migration
- Connection pooling: dedicated pool per service, sized to expected concurrency

**Why idempotent + acks=all:** Market data is financial-grade. Idempotent producer ensures exactly-once delivery even during retries. The ~1ms latency increase (vs `acks=1`) is acceptable — downstream consumers are not on the hot WebSocket path (that path uses Redis + Kafka pull via Broadcast Service).
</Info>

### 6.7 Broadcast Service (WebSocket to Users)

**What it does:** Delivers real-time market data to end users over WebSocket connections. Reads from Kafka topics (pull model) for streaming and Redis Cluster 1 (Feed) for instant snapshots on user connect. Broadcasts market status changes, maintains server-to-client heartbeats, and enforces per-pod connection limits.

**Data source strategy:**

```
Normal operation:
    → Each Broadcast Service pod consumes from Kafka topic partitions
    → Consumer group = broadcast-service-${POD_NAME} (unique per pod — TASK-16)
    → Each pod receives ALL ticks (full partition set)
    → Receive Protobuf-encoded tick messages
    → Deserialize Protobuf tick
    → Look up subscribers via inverted index (TASK-17)
    → Push Protobuf frames over WebSocket to subscribed users

On user connect (instant backfill):
    → Check per-pod connection limit (TASK-20) — reject with 503 if at capacity
    → Read current snapshot from Redis Cluster 1 (Feed) HGET for subscribed instruments
    → Send snapshot as initial Protobuf frame (user sees current price immediately)
    → Then stream incremental updates from Kafka

On restart / reconnection:
    → Kafka consumer group rebalances partitions
    → New pod picks up from last committed offset — no data loss
    → Users reconnect and get fresh snapshot from Redis + resume Kafka stream
```

<AccordionGroup>

<Accordion title="Unique Kafka Consumer Group Per Pod (TASK-16 — P2)">

**Problem:** If all broadcast-service pods share one Kafka consumer group, each pod gets only a subset of partitions. But subscription-based fan-out means every pod needs ALL ticks — a client subscribed to RELIANCE could connect to any pod.

**Solution:**

```yaml
# application.yml
spring:
  kafka:
    consumer:
      group-id: broadcast-service-${POD_NAME}
```

`POD_NAME` comes from the Kubernetes downward API:

```yaml
# K8s deployment spec
env:
  - name: POD_NAME
    valueFrom:
      fieldRef:
        fieldPath: metadata.name
```

**Effect:** Each pod has its own consumer group and therefore receives ALL Kafka partitions — the full tick stream. This is the correct pattern for fan-out services where every instance needs every message.

</Accordion>

<Accordion title="Inverted Subscription Index (TASK-17 — P2)">

**Problem:** On tick arrival, iterating all connections to find subscribers is O(connections x instruments) — unacceptable at 10K+ connections.

**Solution:** Maintain an inverted index: `Map<Integer, Set<WebSocketSession>>` keyed by instrument token.

```java
// On subscribe
subscriptionIndex.computeIfAbsent(instrumentToken, k -> ConcurrentHashMap.newKeySet())
    .add(session);

// On unsubscribe
Set<WebSocketSession> sessions = subscriptionIndex.get(instrumentToken);
if (sessions != null) sessions.remove(session);

// On disconnect — remove session from ALL token sets
subscribedTokens.get(session).forEach(token -> {
    Set<WebSocketSession> s = subscriptionIndex.get(token);
    if (s != null) s.remove(session);
});

// On tick arrival — O(1) fan-out
Set<WebSocketSession> subscribers = subscriptionIndex.get(instrumentToken);
if (subscribers != null) {
    for (WebSocketSession session : subscribers) {
        session.getMessageQueue().offer(protobufBytes);
    }
}
```

**Scaling impact:** Reduces per-tick fan-out from O(N) to O(subscribers_for_this_token). Critical for scaling beyond 10K concurrent connections.

</Accordion>

<Accordion title="Market Status Broadcasting (TASK-18)">

When market status changes (detected via `MarketStatusProcessor` events from Kafka or by polling the `MKT_STATUS:{exchange}` Redis key), broadcast to ALL connected clients regardless of their instrument subscriptions.

**System message format:**

```json
{
    "type": "MARKET_STATUS",
    "exchange": "NSE",
    "status": "OPEN",
    "session": "NORMAL"
}
```

**Delivery:** Sent as a special Protobuf frame with `message_type = 100` (reserved for system messages). Repeated `market.status.duplicate.count` times (default: 3) for delivery assurance — WebSocket is unreliable and clients may miss a single frame.

**Client behavior:** Frontend uses this to show market open/close indicators, enable/disable order placement, and display "Market Closed" banners.

</Accordion>

<Accordion title="Server-to-Client Heartbeat (TASK-19)">

Prevents zombie connections where the client has crashed but the TCP connection remains open (common on mobile devices that lose network silently).

```
Every 30 seconds per connected session:
    1. Send WebSocket PING frame
    2. Start 10-second timer
    3. If PONG received → connection is alive, reset timer
    4. If no PONG within 10s → mark connection as dead
        → Close WebSocket with GOING_AWAY code
        → Remove from subscription index (TASK-17)
        → Decrement connection counter (TASK-20)
        → metric: mde_broadcast_heartbeat_timeout_disconnections_total++
```

**Configuration:**

```yaml
mde:
  broadcast:
    heartbeat:
      interval-seconds: 30
      pong-timeout-seconds: 10
```

</Accordion>

<Accordion title="Per-Pod Connection Limit (TASK-20)">

Each broadcast-service pod has a maximum connection limit to prevent OOM from unbounded connection growth.

```
On new WebSocket connection:
    currentConnections = connectionCounter.get()
    if (currentConnections >= MAX_CONNECTIONS):
        reject with HTTP 503
        set Retry-After header = 5 (seconds)
        metric: mde_broadcast_connection_rejected_total++
        return

    connectionCounter.incrementAndGet()
    proceed with handshake
```

**Configuration:**

```yaml
mde:
  broadcast:
    max-connections-per-pod: 5000  # default, configurable
```

**HPA integration:** Kubernetes HPA scales broadcast-service pods based on the `mde_broadcast_active_connections` metric. When average connections per pod exceeds 80% of `max-connections-per-pod`, HPA adds a new pod.

</Accordion>

</AccordionGroup>

**Why Kafka pull instead of Redis PubSub:**

- **No subscription management complexity** — Kafka consumer groups handle partition assignment automatically
- **No reference counting** — no need to track per-channel subscriber counts
- **Built-in recovery** — on pod restart, resume from last committed offset (no missed messages)
- **Backpressure** — if a pod falls behind, it catches up at its own pace (pull model)
- **Scalability** — add more pods, Kafka rebalances partitions. No PSUBSCRIBE pattern explosion.

**Market data processing pipeline:**

1. Consume Protobuf message from Kafka topic partition
2. Deserialize Protobuf → extract securityId, exchangeSegmentCode, messageCode, payload
3. Apply validation (reject trade with LTP=0, unless pre-open fallback applied)
4. Apply business rules (market status checks, including grace window)
5. Check staleness flag — if feed is UNCERTAIN, attach staleness metadata
6. Encode as Protobuf message
7. Store in BroadcastCache under key `{securityId}_{exchSegCode}_{msgCode}`
8. Look up subscribers via inverted subscription index (TASK-17) — O(1) per token
9. Enqueue Protobuf bytes into each session's message queue
10. Publisher threads drain queues and send frames over WebSocket

**Subscription limits:**

| Limit | Value |
| --- | --- |
| Max scrips per subscribe message | 50 |
| Max concurrent connections per pod | 5,000 (configurable, TASK-20) |
| Max concurrent connections total | 100,000+ (new target, across 20+ pods) |
| Per-session subscription limit | No enforced cap |

**WebSocket Protobuf contract:**

```text
message FeedMessage {
    int32 message_type = 1;      // PacketType code: 1=Trade, 2=MBP, 3=OHLC, 5=Index, 100=System
    int32 exchange_segment = 2;  // 0=Index, 1=NSE_E, 2=NSE_D, 4=BSE_E, 8=BSE_D
    int32 security_id = 3;
    bytes payload = 5;           // typed message bytes
    bool is_stale = 6;           // true if feed source is UNCERTAIN/DOWN
    int64 last_updated_ms = 7;   // epoch ms of last confirmed tick
}
```

**Authentication:** SSO token validated during WebSocket handshake via auth service. Invalid token → reject, no connection. Unauthenticated sessions closed after 30 seconds.

**Thread model:**

- Kafka consumer threads: one per partition assignment (managed by consumer group). With per-pod consumer groups (TASK-16), each pod consumes all partitions.
- Publisher thread pool: configurable 12-25 threads (`ThreadPoolExecutor`, queue size 1,000)
- Distributor: Java 21 virtual threads via `Thread.ofVirtual()` for per-message distribution
- Per-session queue: `ConcurrentLinkedQueue<byte[]>` (max 10,000 entries)
- Heartbeat scheduler: `ScheduledExecutorService` with single thread for PING dispatch (TASK-19)

**Graceful shutdown (rolling deployment):**

1. Set shutdown flag (`AtomicBoolean`)
2. Stop accepting new connections (return 503 immediately)
3. Stop Kafka consumers (commit offsets, leave consumer group)
4. Close all WebSocket sessions with `GOING_AWAY` close code
5. `awaitTermination(30, TimeUnit.SECONDS)`
6. Messages not delivered within 30s are lost (acceptable for real-time data)

### 6.8 Charts Service — Redis-First 1-Minute Candles (TASK-21)

<Warning>
**Architecture change:** The earlier design used TimescaleDB hypertables and continuous aggregates for candle storage. This has been **completely replaced** with a Redis-first + ClickHouse architecture. Redis handles hot intraday candles (2-3 day TTL), ClickHouse handles historical storage and multi-interval rollups via materialized views.
</Warning>

**What it does:** The charts-service is a single unified service that consumes ticks from Kafka, aggregates them into 1-minute candles using in-memory state, flushes completed candles to Redis C2 for hot reads, batch-inserts to ClickHouse for historical storage, and serves the Charts REST API — all in one process with three thread pools.

**Thread Pool 1 — Tick Aggregation (hot path):**

```
Kafka tick topics (mde-ticks-{exchange})
    │
    ▼
charts-service consumer threads (one per assigned partition)
    │
    ├─ Deserialize Protobuf tick → extract token, ltp, volume, oi, timestamp
    │
    ├─ Aggregate into in-memory 1-minute candle:
    │     ConcurrentHashMap<String, CandleAggregator> activeCandles
    │     key = "{token}:{epochMinute}"
    │
    │     CandleAggregator:
    │       open  = first tick's ltp (if new candle)
    │       high  = max(high, ltp)
    │       low   = min(low, ltp)
    │       close = ltp (always latest)
    │       volume = cumulative
    │       oi    = latest OI value
    │       trades = tick count
    │
    ├─ On minute boundary (epochMinute changes for this token):
    │     1. Finalize previous candle
    │     2. Write to Redis C2 (atomic MULTI/EXEC)
    │     3. Publish completed candle to Kafka topic mde-candles-1m
    │     4. Start new candle with current tick
    │
    └─ Start new CandleAggregator for next minute
```

**Redis C2 candle schema:**

<CodeGroup>

```redis Hash Key (one per candle)
# Key pattern: candle:1m:{token}:{epochMinute}
# Example:     candle:1m:NSE:RELIANCE:EQ:28622415

HSET candle:1m:NSE:RELIANCE:EQ:28622415
    open    2450.00
    high    2458.50
    low     2438.00
    close   2450.50
    volume  523400
    oi      0
    trades  1847

EXPIRE candle:1m:NSE:RELIANCE:EQ:28622415 259200  # 3-day TTL
```

```redis Sorted Set Index (one per token)
# Key pattern: candle:idx:{token}
# Example:     candle:idx:NSE:RELIANCE:EQ

ZADD candle:idx:NSE:RELIANCE:EQ 28622415 28622415
# member = epochMinute, score = epochMinute (for ZRANGEBYSCORE range queries)

EXPIRE candle:idx:NSE:RELIANCE:EQ 259200  # 3-day TTL
```

</CodeGroup>

**Redis write atomicity (per candle flush):**

```
MULTI
    HSET candle:1m:{token}:{epochMinute} open {o} high {h} low {l} close {c} volume {v} oi {oi} trades {t}
    EXPIRE candle:1m:{token}:{epochMinute} 259200
    ZADD candle:idx:{token} {epochMinute} {epochMinute}
    EXPIRE candle:idx:{token} 259200
EXEC
```

**TTL:** 2-3 days on both Hash and ZSET keys (configurable via `mde.charts.redis.candle-ttl-seconds`). Redis AOF persistence enabled on C2 for crash recovery — if the pod restarts mid-day, completed candles survive in Redis.

### 6.9 Charts Service — ClickHouse Historical Storage (TASK-22)

**Thread Pool 2 — ClickHouse Batch Writer:**

The second thread pool in charts-service consumes completed 1-minute candles from the `mde-candles-1m` Kafka topic and batch-inserts them into ClickHouse for permanent historical storage.

```
Kafka topic: mde-candles-1m
    │
    ▼
ClickHouse writer threads (configurable pool, default 4)
    │
    ├─ Accumulate candles in batch buffer
    │     Flush trigger: 500 candles OR 5-second interval (whichever first)
    │
    ├─ Batch INSERT into ClickHouse candles_1m table
    │
    ├─ On success: commit Kafka offsets
    │
    └─ On failure:
          ├─ Pause Kafka consumer for this partition
          ├─ Retry with exponential backoff (1s, 2s, 4s, 8s... max 60s)
          ├─ Alert ops-service after 3 consecutive failures
          └─ Resume consumer after successful retry
```

### 6.10 Charts Service — Partition-Based Horizontal Scaling (TASK-23)

The charts-service scales horizontally via Kafka consumer groups. Each pod in the consumer group handles a subset of Kafka partitions (and therefore a subset of instrument tokens).

**Kafka topic configuration:**

```
Topic: mde-ticks-{exchange} (e.g., mde-ticks-nse, mde-ticks-bse)
Partitions: N (default 12 per exchange topic)
Partitioning key: instrumentToken (consistent hashing)
    → All ticks for a given instrument always go to the same partition
    → Therefore always processed by the same charts-service pod
```

**ConsumerRebalanceListener — graceful partition handoff:**

```java
public class CandleRebalanceListener implements ConsumerRebalanceListener {

    @Override
    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
        // Flush all in-progress candles for tokens owned by revoked partitions
        for (TopicPartition tp : partitions) {
            Set<String> tokens = partitionToTokens.get(tp);
            if (tokens != null) {
                for (String token : tokens) {
                    CandleAggregator candle = activeCandles.remove(token + ":" + currentMinute);
                    if (candle != null) {
                        flushCandleToRedis(token, currentMinute, candle);
                        publishCandleToKafka(token, currentMinute, candle);
                    }
                }
                partitionToTokens.remove(tp);
            }
        }
    }

    @Override
    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {
        // Start fresh aggregation for newly assigned partitions
        // No state to recover — Redis has completed candles, in-progress state starts clean
        for (TopicPartition tp : partitions) {
            partitionToTokens.put(tp, ConcurrentHashMap.newKeySet());
        }
    }
}
```

<Warning>
**Critical:** On partition revocation, ALL in-progress candles for that partition's tokens MUST be flushed before the rebalance completes. Failure to do so causes missing candles (the new owner starts a fresh candle, losing partial aggregation from the old owner).
</Warning>

**Scale characteristics:**

| Pods | Ticks/sec capacity | Partitions per pod | Notes |
| --- | --- | --- | --- |
| 1 | ~20K | 12 | Development / low-traffic |
| 3 | ~50K | 4 each | Production baseline |
| 6 | ~100K | 2 each | Peak trading day (budget day, IPO listing) |

**No cross-pod coordination required.** Because Kafka partitioning ensures all ticks for a given instrument go to the same pod, there is no need for distributed locks or shared state. Each pod independently manages its own `ConcurrentHashMap` of active candles.

### 6.11 Charts REST API

**Thread Pool 3 — Undertow Worker Threads (REST API):**

The charts-service also serves the Charts REST API for frontend chart rendering. The same JVM process handles tick aggregation, ClickHouse writes, and REST serving — eliminating inter-service latency for the hot Redis read path.

**API Endpoint:**

```
GET /v1/candles?token={token}&interval={interval}&from={epoch}&to={epoch}&limit={n}
```

**Parameters:**

| Parameter | Type | Required | Description |
| --- | --- | --- | --- |
| `token` | String | Yes | Instrument token (e.g., `NSE:RELIANCE:EQ`) |
| `interval` | Enum | Yes | `1m`, `5m`, `15m`, `1h`, `1d`, `1w`, `1M` |
| `from` | Long | Yes | Start epoch seconds |
| `to` | Long | Yes | End epoch seconds |
| `limit` | Integer | No | Max candles to return (default: 500) |

**Query routing logic:**

<Steps>

<Step title="Determine data source">

```
if interval == "1m" AND (now - from) < 3 days:
    → Redis C2 (hot path)
elif interval == "1m" AND (now - from) >= 3 days:
    → ClickHouse candles_1m table
elif interval in ("5m", "15m", "1h", "1d"):
    → ClickHouse materialized view table
elif interval in ("1w", "1M"):
    → ClickHouse candles_1d with date function aggregation
```

</Step>

<Step title="Execute query">

**Redis path (intraday 1m):**

```
1. ZRANGEBYSCORE candle:idx:{token} {fromEpochMinute} {toEpochMinute}
   → returns list of epochMinute values that have candles

2. Pipeline HGETALL for each: candle:1m:{token}:{epochMinute}
   → returns open, high, low, close, volume, oi, trades for each candle

3. Assemble into response array
```

**ClickHouse path (historical):**

```sql
-- Example: 5-minute candles
SELECT
    timestamp,
    argMinMerge(open)  AS open,
    maxMerge(high)     AS high,
    minMerge(low)      AS low,
    argMaxMerge(close) AS close,
    sumMerge(volume)   AS volume,
    argMaxMerge(oi)    AS oi
FROM candles_5m
WHERE token = {token:String}
  AND timestamp BETWEEN {from:DateTime64} AND {to:DateTime64}
GROUP BY token, timestamp
ORDER BY timestamp
LIMIT {limit:UInt32};
```

</Step>

<Step title="Apply caching">

**Two-layer cache:**

- **L1 — Caffeine (local, per-pod):**
  - Intraday intervals (1m, 5m, 15m): 5-second TTL (data changes rapidly)
  - Historical intervals (1h, 1d, 1w, 1M): 1-hour TTL (data is immutable after market close)
  - Max entries: 10,000 per interval type

- **L2 — Redis C2 (shared across pods):**
  - Only for ClickHouse-sourced queries (historical)
  - Key: `cache:candles:{token}:{interval}:{fromEpoch}:{toEpoch}`
  - TTL: 5 minutes (intraday), 24 hours (historical)
  - Prevents redundant ClickHouse queries from multiple pods

**Three-layer durability guarantee:**

<CardGroup cols={3}>

<Card title="Layer 1 — Redis AOF" icon="bolt">
Redis C2 with AOF persistence. Survives pod restarts. 2-3 day TTL covers all intraday needs. Sub-millisecond reads.
</Card>

<Card title="Layer 2 — ClickHouse" icon="database">
Permanent columnar storage. ReplacingMergeTree with automatic dedup. Materialized views for all intervals. 10-year retention.
</Card>

<Card title="Layer 3 — Kafka Replay" icon="rotate">

</Card>

<Tip>
**Why not TimescaleDB?** The earlier design used TimescaleDB for both raw ticks and candle storage. ClickHouse was chosen instead because: (1) columnar storage provides 10-15x better compression for time-series data, (2) materialized views update synchronously on INSERT (no background refresh lag), (3) ClickHouse handles the append-heavy write pattern more efficiently than PostgreSQL-based TimescaleDB, and (4) Redis-first for intraday reads eliminates the need for a hot-path database query entirely.
</Tip>
