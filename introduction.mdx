<Info>

- **Production observability** — Prometheus metrics, distributed tracing, 8 Grafana dashboards.

## 1. Overview

### What is the Market Data Engine?

The Market Data Engine is the **single, authoritative source of all market data** in the Paytm Money Pro platform. Every price a user sees — equity LTP, option chain, chart candle, market mover ranking, index value — originates from this engine.

**No other system connects directly to exchange feeds.** This means:

- One place to fix a bad price
- One place to monitor feed health
- One place for corporate action adjustments
- One audit trail for price corrections

### What It Covers

| Data Domain | Examples |
| --- | --- |
| **Equity** | LTP, OHLC, volume, previous close, 52-week H/L, circuit limits for all NSE/BSE stocks |
| **Futures** | LTP, OI, volume, OHLC for equity/index/currency futures |
| **Options** | LTP, OI, Greeks (delta, gamma, theta, vega), IV for all option contracts |
| **Indices** | NIFTY 50, SENSEX, BANK NIFTY — real-time values |
| **ETFs, REITs, InvITs, Bonds** | Same as equity — LTP, OHLC, volume |
| **Option Insights** | PCR (OI/Volume), Max Pain, ATM Straddle, CPR |
| **Charts** | 1m, 5m, 15m, 1h, 1d, 1w, 1M candles — current day + historical |
| **Market Movers** | Gainers, losers, volume gainers, value gainers, 52-week H/L hitters |
| **Instrument Master** | Scrip definitions, company info, segment mappings |
| **Corporate Actions** | Splits, bonus, dividends, rights — with historical price adjustments |

### Design Philosophy

1. **Correctness first** — A wrong price delivered fast is worse than a correct price slightly slower.
2. **Self-healing over manual intervention** — Auto-reconnect, auto-recover, notify operators (not page them).
3. **Composable** — Independently deployable. Trade Engine, Portfolio Engine, Ledger Engine depend on it, not the other way around.
4. **Observable** — Every tick, correction, feed switch, reconnection is logged, metriced, and traceable.
5. **API-compatible** — All read APIs match the current contract since the UI is not being replaced.
6. **Zero-Error UX** — Users must never see an error popup. The system degrades silently through layered fallbacks; the AI Incident Manager handles internal noise.

## 5. Feed Sources & Failover

### Merged Feed Architecture

The Market Data Engine receives data from **two independent sources simultaneously** and merges them into a single, best-of-both stream:

| Source | Protocol | Location | Typical Latency |
| --- | --- | --- | --- |
| **NSE/BSE Direct** | UDP Multicast, dedicated cable, 6 streams | Exchange colocation | Baseline (fastest) |
| **Omnenest** | Omnenest RS protocol (proprietary) | Third-party DC | 3–8 ms slower than NSE Direct |

**How the merge works:**

Both NSE Direct and Omnenest feed into the **same Feed Handler**. The Feed Handler deduplicates by `(instrumentId, timestamp)` — whichever tick arrives first wins. Since NSE Direct is typically 3–8 ms faster, NSE ticks usually arrive first and are used. Omnenest ticks that arrive second are dropped as duplicates. But if NSE misses a UDP packet (packet loss), the Omnenest copy fills the gap automatically — and vice versa.

**Key properties of the merged architecture:**

- **All users get the best-of-both sources** — always the fastest, most complete data
- **No user is ever on a "slower" feed** — unlike an 85/15 traffic split
- **Omnenest is continuously validated** — every Omnenest tick is compared against its NSE counterpart. Divergence beyond threshold triggers an alert.
- **Automatic gap filling** — if either source drops a packet, the other fills in seamlessly
- **Zero-switchover failover** — if NSE dies entirely (heartbeat timeout), Omnenest is already active and delivering data. No switchover delay.
- **Input ~4K TPS** (2K from each source), **output ~2K TPS** after dedup

**Open point:** Verify whether `NSE_FNO_0` / `NSE_FNO_1` represent A/B redundancy (same instruments, two copies) or separate instrument sets (different instruments per stream). This affects dedup logic.

### Failover Detection

<Warning>
**Critical:** Exchanges only send ticks when values change. Not receiving a tick for a few seconds is normal — it means no trades happened. We **cannot** use "no ticks received" as the sole detection signal.
</Warning>

**Detection mechanisms (in order of reliability):**

| Signal | Method | Threshold |
| --- | --- | --- |
| **Heartbeat absence** | Exchanges send heartbeat packets (transaction code 99) periodically even when markets are quiet. If heartbeats stop → feed is dead. | 3 seconds if exchange heartbeats are available (open point: confirm heartbeat interval with exchange connectivity team). If heartbeats not available, fallback to cross-source divergence detection. |
| **Staleness indicator** | Within 1 second of no heartbeat, mark feed as UNCERTAIN and propagate staleness flag to Broadcast Service. Users see "Last updated: Xs ago" with amber (>3s) / red (>10s) coloring. | 1 second → UNCERTAIN flag |
| **Sequence gap** | UDP sequence numbers should be continuous. A sustained gap with no recovery → connection problem. | Gap > 100 with no fill for 5 seconds |
| **All-segment silence** | If ALL 6 streams go silent simultaneously during market hours (9:15–3:30) → infrastructure failure. Individual stream silence is normal. | All streams silent for 15 seconds during `OPEN` status |
| **Cross-source divergence** | If Omnenest is delivering but NSE Direct is not (or vice versa) → one source is down. | One source active, other silent for 3 seconds |

**Failover actions:**

```
NSE Direct feed drops (heartbeat lost):
    → Detect within 3 seconds (heartbeat timeout)
    → Omnenest is already active — zero switchover time
    → All ticks now come from Omnenest (no dedup partner)
    → Alert: P0 Slack "NSE Direct feed down, Omnenest sole source"
    → Mark feed quality as DEGRADED (single source, no cross-validation)
    → Attempt NSE reconnection with exponential backoff

Omnenest feed drops:
    → NSE Direct continues as sole source — no user impact
    → Alert: P0 Slack "Omnenest down, NSE Direct sole source"
    → Mark feed quality as DEGRADED

Both feeds drop:
    → CRITICAL alert — page on-call immediately
    → Serve stale data from Redis (mark as stale)
    → WebSocket sends stale flag to clients
    → Frontend shows "Delayed" indicator with red coloring
```

### External Circuit Breaker

A **circuit breaker sits between the exchange and our feed handler**. This is NOT our service — it's managed externally. When the exchange halts trading on an instrument (circuit limit hit), the halt message reaches us through this external component.
