<Info>
**Post-consolidation (TASK-33).** The Market Data Engine was consolidated from 15 independent services into **10 deployable services**. TimescaleDB has been fully removed. This page is the single source of truth for what each service does, what it depends on, and how it scales.
</Info>

## At a glance

- **10 deployable services** (consolidated from 15) + 2 shared modules
- **2 shared modules**: `shared/common` (enums, value types, security stubs), `shared/proto` (Protobuf contracts for Kafka + WebSocket)
- **Runtime stack**: Java 21, Spring Boot 3.2, Virtual Threads, Kafka 3.8, ClickHouse 0.6.5, Lettuce 6.4, Temporal 1.26
- **Build**: Gradle + Jib (Docker images)
- **Monitoring**: Micrometer + Prometheus (management port 9xxx), Spring Boot Actuator
- **Tracing**: Micrometer Tracing with Brave/Zipkin bridge

### What changed (TASK-33)

| Before (15 services) | After (10 services) | Merge type |
| --- | --- | --- |
| charts-writer + charts-historical-writer + charts-api | **charts-service** | Full merge вҖ” writer + historical + API in one service |
| premarket-scheduler + corporate-action-processor | **scheduler-service** | Merged into single Temporal worker with two task queues |
| incident-manager + recon-service | **ops-service** | Merged into autonomous operations hub |
| instrument-master + company-fundamentals | **instrument-master** | Absorption вҖ” fundamentals became a module inside instrument-master |

### instrument-master {#instrument-master}

<Accordion title="Full details">

**Port**: 8081 (mgmt 9081) | **Absorbs**: company-fundamentals | **Storage**: PostgreSQL, Redis C1

**What it does**: Instrument CRUD, full-text search, index membership, and company fundamentals (profile, shareholding, analysts, institutional activity). The canonical source for all ~90,000 instrument definitions across NSE + BSE.

**Key components**:
- Instrument CRUD and full-text search (symbol, exchange, segment, scrip type)
- Index membership management (NIFTY 50, SENSEX, sector indices)
- Company fundamentals module (absorbed from company-fundamentals service): profile, shareholding pattern, analyst recommendations, institutional activity (FII/DII)

**Kafka**:
- **Produces**: `mde-instrument-events` (instrument creates, updates, delistings)
- **Consumes**: `mde-instrument-events` (self-loop for cache invalidation across pods)

**Scaling**: Horizontal. Stateless reads from PostgreSQL + Redis cache. Each pod has a local Caffeine cache invalidated via Kafka. Typical: 2-3 replicas.

### feed-handler {#feed-handler}

<Accordion title="Full details">

**Port**: 8082 (mgmt 9082) | **Storage**: None (stateless, in-memory cache only)

**What it does**: Binary packet decoder for NSE/BSE multicast feeds. Receives UDP packets from 6 streams per source, performs LZO decompression, protobuf serialization, and cross-source deduplication. Publishes clean tick stream to Kafka.

**Key components**:
- Binary packet decoder (NSE and BSE native formats)
- LZO decompression for compressed exchange packets
- Source deduplication: `(instrumentId, timestamp)` key, first-arrival wins (NSE Direct typically 3-8ms faster than Omnenest)
- Market status state machine (PRE_OPEN, OPEN, CLOSE, POST_CLOSE)
- OI injection into MBP trade packets (F&O segments)
- Zero-LTP fallback for pre-open session (uses indicative price)
- Cross-minute re-send for deduplicated ticks (ensures downstream minute boundaries are correct)
- Previous-close enrichment for index ticks
- Post-close timing window handling (exchange sends settlement data after 3:30 PM)
- Protobuf serialization to Kafka

**Kafka**:
- **Produces**: `mde-ticks-nse-equity`, `mde-ticks-nse-derivative`, `mde-ticks-nse-index`, `mde-ticks-bse-equity`, `mde-ticks-bse-derivative`, `mde-ticks-bse-index`
- **Consumes**: `mde-instrument-events` (refresh local instrument cache on scrip changes)

**Scaling**: Single instance per source pair (NSE+Omnenest merged). UDP multicast requires dedicated NIC binding. Cannot horizontally scale individual stream processing вҖ” scale by adding source pairs or splitting segments to separate pods.

**Endpoints**: `/internal/admin/*` (feed enable/disable, stats, health)

</Accordion>

### publisher {#publisher}

<Accordion title="Full details">

**Port**: Management only (9082) вҖ” no HTTP API | **Storage**: Redis C1

**What it does**: Kafka tick consumer that writes real-time snapshots to Redis Cluster 1. Headless writer service with no REST endpoints. Writes 6 key types that form the live market data snapshot used by broadcast-service and other read-path services.

**Key components**:
- Kafka consumer for all 6 tick topics (pattern-based subscription)
- Redis snapshot writer: `FEED_PP` (full price packet), `OHLC`, `PC` (previous close), `LC_UC` (circuit limits from BSE MBP packet), `52WK` (52-week high/low), `DEPTH` (market depth L5)
- Lua CAS (compare-and-set) scripts for atomic 52-week high/low updates
- Market status Redis key (broadcast-service reads this for client notifications)
- BSE circuit limit extraction from MBP packet data

**Kafka**:
- **Produces**: Nothing
- **Consumes**: `mde-ticks-*` (all 6 tick topics)

**Redis keys written** (C1):
- `FEED_PP:{segment}:{instrumentToken}` вҖ” full price packet hash
- `OHLC:{segment}:{instrumentToken}` вҖ” OHLC for current day
- `PC:{segment}:{instrumentToken}` вҖ” previous close
- `LC_UC:{segment}:{instrumentToken}` вҖ” lower/upper circuit limits
- `52WK:{segment}:{instrumentToken}` вҖ” 52-week high/low (Lua CAS)
- `DEPTH:{segment}:{instrumentToken}` вҖ” market depth (top 5 bid/ask)
- `MARKET_STATUS` вҖ” current market phase

**Scaling**: Horizontal via Kafka consumer group partition assignment. Each pod processes a subset of partitions. Typical: 3-6 replicas depending on tick volume.

</Accordion>

### broadcast-service {#broadcast-service}

<Accordion title="Full details">

**Port**: 8087 (mgmt 9087) | **Storage**: Redis C1 (read-only)

**What it does**: WebSocket server that fans out real-time tick data to subscribed clients. Each pod consumes all Kafka tick topics with a unique consumer group (every pod gets every tick) and pushes matching ticks to connected WebSocket sessions.

**Key components**:
- WebSocket endpoint (`/ws/feed`) with Spring WebFlux
- Kafka consumer with unique group-id per pod (all pods receive all ticks)
- Cold-start snapshot from Redis C1 `FEED_PP` keys (on new subscription, client gets current state before live stream)
- Market status broadcasting to all connected clients
- Server-to-client heartbeat for idle connection detection
- Per-pod connection limit (configurable, default 100K)
- Inverted subscription index: maps `instrumentToken` to set of WebSocket sessions for O(1) fan-out lookup
- Protobuf serialization for wire efficiency

**Kafka**:
- **Produces**: Nothing
- **Consumes**: `mde-ticks-*` (all 6 tick topics, unique consumer group per pod)

**Scaling**: Horizontal. Each pod handles its own set of WebSocket connections. No shared state between pods. Sticky-session load balancer required (WebSocket affinity). Scale based on connection count. Typical: 5-10 replicas in production.

**Endpoints**: `WebSocket /ws/feed`, `/v3/api-docs`

</Accordion>

### charts-service {#charts-service}

<Accordion title="Full details">

**Port**: 8083 (mgmt 9083) | **Merges**: charts-writer + charts-historical-writer + charts-api | **Storage**: Redis C2, ClickHouse

**What it does**: The unified candle pipeline. Three internal thread pools handle the full lifecycle: (1) tick consumption and 1-minute candle aggregation, (2) ClickHouse batch persistence, (3) REST API for reading candles. Replaces three separate services from the original architecture.

**Key components**:

*Thread Pool 1 вҖ” Tick Consumer + Candle Aggregation*:
- Kafka tick consumer (pattern-based, all equity + derivative + index topics)
- In-memory `ConcurrentHashMap<BucketKey, BucketState>` keyed by `(token, intervalCode, bucketEpochSec)`
- On bucket-epoch rollover: closed 1m candle fanned out to Redis C2 + ClickHouse batch buffer + Kafka `mde-candles-1m`
- Partition-based horizontal scaling via `ConsumerRebalanceListener` (each pod owns a partition subset)

*Thread Pool 2 вҖ” ClickHouse Batch Writer*:
- In-memory batch accumulator, flushes on size threshold (default 5,000 rows) or time interval (default 60s)
- Batch INSERT into `mde.candles_1m` (ReplacingMergeTree, `updated_at` as version for idempotent replays)
- 4 materialized view rollups: `candles_5m`, `candles_15m`, `candles_1h`, `candles_1d` (ClickHouse-managed, not direct writes)
- Cascade rollup: updates running 5m/15m/1h buckets in Redis C2 only; higher intervals are ClickHouse MV responsibility

*Thread Pool 3 вҖ” REST API*:
- Serves chart data: Redis C2 for hot (current day), ClickHouse for historical
- Caffeine L1 cache per pod + Redis L2 cache-aside for API responses
- TradingView UDF-compatible endpoint (`POST /v1/price-charts`) for PML UI backward compatibility

**Kafka**:
- **Produces**: `mde-candles-1m` (closed 1m candles for market-movers-writer)
- **Consumes**: `mde-ticks-*` (all 6 tick topics)

**ClickHouse schema**:
- `mde.candles_1m` вҖ” ReplacingMergeTree, partition by `toYYYYMM(timestamp)`
- `mde.candles_5m_mv` вҖ” Materialized view rollup from `candles_1m`
- `mde.candles_15m_mv` вҖ” Materialized view rollup
- `mde.candles_1h_mv` вҖ” Materialized view rollup
- `mde.candles_1d_mv` вҖ” Materialized view rollup

**Scaling**: Horizontal via Kafka partition assignment for write path. Read path scales independently (stateless API with cache). Typical: 3-5 replicas.

**Endpoints**:

| Endpoint | Method | Notes |
| --- | --- | --- |
| `/v1/candles/{instrumentToken}` | GET | OHLCV range query. Params: `from`, `to`, `interval`. Auto-routes: Redis (today) or ClickHouse (historical). |
| `/v1/running-candles/{instrumentToken}` | GET | In-progress candle from Redis C2. Param: `interval` (default 1m). |
| `/v1/price-charts` | POST | TradingView UDF-compatible вҖ” PML UI backward compatibility. |
| `/v1/index` | GET | Index quotes |
| `/v3/api-docs` | GET | OpenAPI spec |

</Accordion>

### market-movers-writer {#market-movers-writer}

<Accordion title="Full details">

**Port**: 8084 (mgmt 9084) | **Storage**: Redis C2

**What it does**: Computes ranked market mover lists from closed 1-minute candles. Only processes equity instruments (~7K, not the full 90K universe). Writes Redis sorted sets for each mover category with automatic rank trimming.

**Key components**:
- Kafka consumer for `mde-candles-1m` topic (NOT raw ticks вҖ” uses closed candles for accurate % change)
- 8 mover categories: TOP_GAINERS, TOP_LOSERS, MOST_ACTIVE (by volume), VOLUME_SPIKE, BREAKOUT_HIGH, BREAKOUT_LOW, UPPER_CIRCUIT, LOWER_CIRCUIT
- Redis ZSET writes with `ZADD` + `ZREMRANGEBYRANK` (keep top N) + `EXPIRE`
- REST API with Caffeine cache for low-latency reads
- Equity-only filter (~7K instruments)

**Kafka**:
- **Produces**: Nothing
- **Consumes**: `mde-candles-1m` (from charts-service)

**Scaling**: Single instance sufficient (only ~7K instruments, one Kafka topic). Can scale to 2 replicas for HA.

**Endpoints**:

| Endpoint | Method | Notes |

| `/v1/movers` | GET | Ranked movers by category. Params: `category`, `exchange`, `limit`. |

### fno-data-writer {#fno-data-writer}

<Accordion title="Full details">

**Port**: 8085 (mgmt 9085) | **Storage**: Redis C2

**What it does**: Real-time F&O analytics engine. Consumes derivative ticks, computes Greeks, builds option chain aggregations, and writes enriched data to Redis for the FNO dashboard.

**Key components**:
- Kafka tick consumer for NSE + BSE derivative segments
- Greeks computation: Black-Scholes for equity options, Black-76 for index options
- IV solver: Newton-Raphson with Brent method fallback for convergence edge cases
- Max Pain calculation per underlying per expiry
- ATM Straddle computation (nearest-strike CE+PE combined premium)
- PCR calculation: both OI-based and Volume-based, per underlying
- Day-over-day change tracking (vs previous close OI/volume)
- BSE F&O support (same computation pipeline, exchange-agnostic)
- Redis chain writes: pipelined `HMSET` + `ZADD` + `EXPIRE` for option chain, Greeks, and analytics keys

**Kafka**:
- **Produces**: Nothing
- **Consumes**: `mde-ticks-nse-derivative`, `mde-ticks-bse-derivative`

**Scaling**: Horizontal via Kafka partition assignment. Each pod handles a subset of derivative instruments. Typical: 2-3 replicas.

**Endpoints**:

| Endpoint | Method | Notes |
| --- | --- | --- |
| `/v1/fno-chain` | GET/POST | Option chain with Greeks, IV, OI for a given underlying + expiry |
| `/v1/fno-chain/max-pain` | GET | Max Pain strike and P&L curve |
| `/v1/fno-chain/pcr` | GET | Put-Call Ratio (OI and Volume) |
| `/v1/fno-chain/atm-straddle` | GET | ATM Straddle premium and breakeven |
| `/v1/oi-buildup` | GET | OI buildup analytics |

### alert-engine {#alert-engine}

<Accordion title="Full details">

**Port**: 8091 (mgmt 9091) | **Storage**: Redis C2, PostgreSQL

**What it does**: User-facing alert management service. Users define price/volume conditions; the engine evaluates every incoming tick against active alerts and triggers notifications when conditions are met.

**Key components**:
- Alert condition types: PRICE_ABOVE, PRICE_BELOW, PERCENT_CHANGE, VOLUME_SPIKE, CIRCUIT_HIT, 52W_HIGH, 52W_LOW
- Redis inverted index: maps `instrumentToken` to set of active alert IDs for O(1) lookup per tick
- Kafka tick consumer for real-time evaluation
- BOD (beginning-of-day) reference data store for 52W and circuit conditions (loaded by scheduler-service)
- Reconciliation on restart: `@PostConstruct` rebuilds Redis index from PostgreSQL
- Rate limiting: 10 alert triggers per user per minute
- Max active cap: 50 alerts per user
- Kafka producer for trigger events (downstream notification service)
- PostgreSQL for alert definitions, trigger history, user preferences

**Kafka**:
- **Produces**: `mde-alert-triggers` (triggered alert events for notification delivery)
- **Consumes**: `mde-ticks-nse-equity`, `mde-ticks-bse-equity` (equity ticks for alert evaluation)

**Scaling**: Horizontal via Kafka partition assignment. Alert index is per-pod (rebuilt from PostgreSQL on startup). Typical: 2 replicas.

**Endpoints**:

| Endpoint | Method | Notes |
| --- | --- | --- |
| `/v1/alerts` | GET/POST | List and create alerts |
| `/v1/alerts/{alertId}` | GET/PUT/DELETE | Single alert CRUD |
| `/v1/alerts/triggered` | GET | Trigger history for a user |

</Accordion>

### scheduler-service {#scheduler-service}

<Accordion title="Full details">

**Port**: 8094 (mgmt 9094) | **Merges**: premarket-scheduler + corporate-action-processor | **Storage**: Redis C1, Redis C2, PostgreSQL

**What it does**: Temporal workflow worker operating on two task queues. Runs all scheduled jobs: beginning-of-day data preparation (06:00 IST) and corporate action processing (19:30 IST). Does not serve user-facing API traffic.

**Key components**:

*Task Queue: `premarket-scheduler-jobs` (BOD вҖ” 06:00 IST)*:
- Previous close loading: downloads NSE/BSE bhavcopy files, writes `PC` keys to Redis C1
- OHLC seed: initializes day's OHLC with previous close as open
- 52-week high/low refresh: recalculates from bhavcopy history, writes `52WK` keys
- Circuit limit loading: BSE circuit limits from DP (daily price) file, writes `LC_UC` keys
- Trading day stamp: sets `TRD_DAY` guard key in Redis C1
- Direct download from exchange URLs (NSE Equity bhavcopy, BSE bhavcopy, NSE F&O bhavcopy, F&O ban list)

*Task Queue: `corporate-action-jobs` (19:30 IST)*:
- T-1 pre-computation: downloads upcoming corporate action schedule
- Shadow ClickHouse table creation: copies affected candle partitions to shadow tables
- Price adjustment calculation: split ratios, bonus ratios, dividend adjustments
- Atomic partition swap: at midnight on ex-date, swaps shadow tables (with adjusted prices) into production tables
- PostgreSQL audit trail for every adjustment

**Kafka**:
- **Produces**: Nothing
- **Consumes**: Nothing (Temporal-driven, not Kafka-driven)

**Scaling**: Single instance per task queue (Temporal handles exactly-once execution). Scale to 2 for HA but only one active worker per task queue.

</Accordion>

### ops-service {#ops-service}

<Accordion title="Full details">

**Port**: 8088 (mgmt 9088) | **Merges**: incident-manager + recon-service | **Storage**: PostgreSQL, ClickHouse

**What it does**: Autonomous operations hub. Combines end-of-day reconciliation, proactive incident detection, and self-healing capabilities. Market-hours aware (adjusts behavior for pre-market, market hours, post-close).

**Key components**:

*Recon module*:
- End-of-day OHLCV reconciliation Temporal workflows
- Compares MDE candle data (ClickHouse) against exchange bhavcopy (source of truth)
- Recon run history in PostgreSQL
- Discrepancy detection with configurable thresholds

*Incident module*:
- AlertManager webhook receiver for Prometheus alerts
- Incident lifecycle: OPEN, ACKNOWLEDGED, INVESTIGATING, RESOLVED, CLOSED
- Jira ticket creation for P0/P1 incidents
- Slack notification for all severity levels
- Incident timeline persistence in PostgreSQL

*Self-healing*:
- LLM-driven RCA (Root Cause Analysis) using Claude Sonnet 4.5
- Autonomous remediation for known failure patterns (feed reconnection, cache rebuild, stale data flush)
- Market-hours awareness: suppresses non-critical alerts during off-hours, escalates during trading hours
- Health checks across all MDE services

**Kafka**:
- **Produces**: `mde-feed-anomalies` (detected anomalies for downstream alerting/audit)
- **Consumes**: Nothing directly (pulls data from other services via REST + storage queries)

**Scaling**: Single instance (ops coordination must be centralized). HA via warm standby.

**Endpoints**:

**Key data flow relationships**:
- **feed-handler** is the sole data ingress point. All downstream services consume from its Kafka tick topics.
- **publisher** is the sole writer to Redis C1 snapshots. No other service writes to C1 during market hours (scheduler-service writes BOD data before market open).
- **charts-service** produces `mde-candles-1m` which is the sole input for **market-movers-writer** (movers are computed from closed candles, not raw ticks).
- **broadcast-service** reads Redis C1 only for cold-start snapshots; live data comes directly from Kafka.
- **ops-service** has read access to all storage systems but writes only to its own PostgreSQL tables and Kafka anomaly topic.
