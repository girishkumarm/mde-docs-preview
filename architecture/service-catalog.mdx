## At a glance

- **14 Spring Boot services** — 13 runtime + 1 dev-only mock
- **2 shared modules**: `shared/common` (enums, value types, security stubs), `shared/proto` (Protobuf contracts for Kafka + WebSocket)
- **Runtime stack**: Java 21 + Spring Boot 3.2.11 + Kafka 3.8.0 + ClickHouse JDBC 0.6.5 + Lettuce 6.4.0 + Temporal 1.26.2
- **Dual-write (bake-off) period**: `charts-writer` + `charts-historical-writer` (legacy v1 candle pipeline) still run alongside `candle-writer` (v2). Neither is deleted from `settings.gradle` yet — see the rollback runbook for details.

## TimescaleDB reality check

Despite the v2 narrative, **three services still depend on TimescaleDB** in the current code:

| Service | Usage | When can it go? |
| --- | --- | --- |
| `charts-writer` (legacy v1) | Primary candle store — writes 1m hypertable | When v2 bake-off passes and the writer is scaled to 0 |
| `charts-historical-writer` | (does NOT use TimescaleDB — writes to ClickHouse) | Legacy consumer kept for rollback only |
| `recon-service` | Reads legacy 1m for reconciliation against ClickHouse | When the legacy data window is no longer in the recon lookback |
| `instrument-master` | Read-only proxy at `jdbc:postgresql://localhost:5433/timeseries` | After chart-read migration + proxy consumers re-pointed |

**Until those three are cleared, TimescaleDB is a live dependency.**

## Service details

### instrument-master {#instrument-master}

**App class**: `InstrumentMasterApplication` · **Port**: 8081 (mgmt 9081)
**Purpose**: Scrip master CRUD, search, company info, corporate-action event publishing.
**Kafka**: producer on `mde-instrument-master-events`, `mde-corporate-actions` (via `CorporateActionsEventPublisher`); consumer on `mde-instrument-master-events` (self-loop / invalidation).
**Storage**: PostgreSQL (canonical), Redis Feed (hot lookups), S3 (scrip snapshots), **TimescaleDB** (read-only proxy for legacy chart reads).

| Endpoint | Method | Notes |
| --- | --- | --- |
| `/v1/instruments` | GET | List / filter (symbol, exchange, segment, scripType) |
| `/v1/instruments/{instrumentToken}` | GET | Single instrument |
| `/v1/companies` | GET/POST | Company info |
| `/v1/search` | GET | Full-text search |
| `/internal/ca-sync` | GET/POST | **Unauthenticated** CA webhook |
| `/internal/admin/*` | | **Unauthenticated** operator admin |

### feed-handler {#feed-handler}

**App class**: `FeedHandlerApplication` · **Port**: 8082 (mgmt 9082)
**Purpose**: NSE + BSE multicast ingestion (6 UDP streams → decode → cross-source dedup → Kafka).
**Kafka produces** (from `application.yml`):

```
nse-equity:     pml-marketfeed-nse-equity-ticks
bse-equity:     pml-marketfeed-bse-equity-ticks
nse-derivative: pml-marketfeed-nse-derivative-ticks
bse-derivative: pml-marketfeed-bse-derivative-ticks
nse-index:      pml-marketfeed-nse-index-ticks
bse-index:      pml-marketfeed-bse-index-ticks
```

**Kafka consumes**: `mde-instrument-master-events` via `InstrumentInvalidationListener` (refresh local cache on scrip changes).
**Endpoints**: `/internal/admin/*` (feed enable/disable, stats).

### publisher {#publisher}

**App class**: `PublisherApplication` · **Port**: 8083 (mgmt 9083)
**Purpose**: Tick consumer → writes hot snapshots to Redis Feed + Lua atomic 52-week H/L + OHLC compare-and-set.
**Kafka consumes**:
- `pml-marketfeed-.*-ticks` pattern (all 6 tick topics)
- `mde-corporate-actions` (for cascading CA effects into Redis snapshots)

Redis keys written: `FEED_PP_{segment}:{pmlId}`, `MGR_*_OHLC:{pmlId}`, `MGR_*_52_WK_HL:{pmlId}`, `TRD_DAY`.

### market-movers-writer {#market-movers-writer}

### charts-writer {#charts-writer}

<Warning>
**Legacy v1 — to be removed after bake-off.** This service is still on `main` for rollback. New deployments should NOT include it. It is the writer that produces `mde-candles-1m`.
</Warning>

**App class**: `ChartsWriterApplication` · **Port**: 8086 (mgmt 9086)
**Purpose**: v1 candle pipeline — consumes ticks, aggregates, **writes TimescaleDB**, **publishes `mde-candles-1m`**.
**Kafka**: consumer `pml-marketfeed-.*-ticks` pattern, producer `mde-candles-1m`.
**Storage**: TimescaleDB (primary v1 candle store).

### candle-writer ⭐ {#candle-writer}

**App class**: `CandleWriterApplication` · **Port**: 8086 (mgmt 9086)
**Purpose**: Current (v2) dual-sink candle builder. Replaces `charts-writer` + `charts-historical-writer`.

**Internal pipeline** (verified against `services/candle-writer/src/main/java/.../`):

1. **`TickListener`** — `@KafkaListener(topicPattern = "pml-marketfeed-.*-ticks", groupId = "mde-candle-writer")`. Parses Protobuf `Tick`, drops non-TRADE packets, acks offset after in-memory apply.
2. **`CandleAggregator`** — `ConcurrentHashMap<BucketKey, BucketState>` keyed by `(token, intervalCode, bucketEpochSec)`. Each tick calls `BucketState.applyTick(ltp, qty, tsMicros)`. On bucket-epoch rollover for `(token, 1m)` the bucket is removed + handed to `onBucketRollover(...)`.
3. **Fan-out on rollover**:
   - `RedisCandleSink` → `CANDLE:1m:{token}` hash (Redis Feed)
   - `ClickHouseCandleSink.accumulate(row)` → in-memory batch
   - `cascadeRollup` → updates running 5m/15m/1h buckets **in Redis only**; higher intervals in ClickHouse are built by materialized views (not direct writes)
4. **`ClickHouseCandleSink.flush()`** fires on whichever comes first:
   - `buffer.size() >= mde.candle-writer.clickhouse-batch-size` (default **5000**)
   - `@Scheduled(fixedRateString = "${mde.candle-writer.clickhouse-flush-ms:60000}")` every **60 s**
5. Batch INSERT into `mde.candles_1m` (ReplacingMergeTree, `updated_at` as version → idempotent replays).

**candle-writer does NOT publish to Kafka.** It is pure consumer → Redis + ClickHouse. `mde-candles-1m` belongs to the v1 path (charts-writer → charts-historical-writer).

**Shipped commits**: `22b566c` (merge), `35560c0` (CH schema), `bb3f8ff` (API `source=` switch), `a9863de` (Redis auto-config fix).

### broadcast-service {#broadcast-service}

**App class**: `BroadcastServiceApplication` · **Port**: 8087 (mgmt 9087)
**Purpose**: WebSocket fan-out (`/ws/feed`) to authenticated sessions.
**Kafka consumes**: `pml-marketfeed-.*-ticks` pattern with **unique group-id per pod** (every pod receives every tick; see [Kafka vs Redis](/decisions/kafka-vs-redis)).
**Endpoints**: `WebSocket /ws/feed` + OpenAPI at `/v3/api-docs`.

### recon-service {#recon-service}

**App class**: `ReconServiceApplication` · **Port**: 8089 (mgmt 9089)
**Purpose**: End-of-day OHLCV reconciliation Temporal workflow.
**Kafka producer**: `mde-recon-reports`.
**Storage**: PostgreSQL (run history), ClickHouse (new source), **TimescaleDB (legacy source of 1m candles, still read for now)**.
**Endpoint**: `GET /v1/recon`.

**Storage**: PostgreSQL (incident registry).

| Endpoint | Method | Notes |

### jira-mock {#jira-mock}

### charts-api {#charts-api}

**App class**: `ChartsApiApplication` · **Port**: 8086 (mgmt 9086)
**Purpose**: Public read API for charts.

| Endpoint | Method | Notes |
| --- | --- | --- |
| `GET /v1/candles/{instrumentToken}` | GET | OHLCV range. Query: `from`, `to`, `interval`, `source={redis\|clickhouse\|auto}`. `auto` = Redis for current day, ClickHouse otherwise. |
| `GET /v1/running-candles/{token}` | GET | In-progress candle (Redis only). Query: `interval` (default 5s). |
| `POST /v1/price-charts` | POST | TradingView UDF-compatible — keeps PML UI working with zero config change (commit `bb3f8ff`). |
| `GET /v1/index` | GET | Index quotes |
| `/v3/api-docs`, `/swagger-ui.html` | GET | OpenAPI |

**Caches**: Caffeine L1 per pod + Redis API cache-aside (`candle:1m:{token}`).
**Redis auto-config disabled** (commit `a9863de`) — two qualified `RedisConnectionFactory` beans (Feed + API) collide with Spring Boot defaults.
