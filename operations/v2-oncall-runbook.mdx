5. **mde-charts-historical-writer (9188)** — (a) Is the Temporal schedule `charts-historical-backfill` active? (b) Is the S3 archival bucket writable? (c) ClickHouse batch insert latency normal? (V2_1 migration in `cf966ec`)
6. **mde-scheduler (9089)** — (a) Is Temporal server reachable at `172.18.0.11:7233`? (b) Are the cron schedules registered (`temporal schedule list`)? (c) Workflow history not truncated?
7. **mde-instrument-master (9092)** — (a) Is the exchange EOD contract file fetched? (b) PG `instrument_master_raw` has today's partition? (c) Did Kafka publish `mde.instrument.refresh`?
8. **mde-replay-service (9095)** — (a) Is the Kafka replay topic authorised? (b) Is S3 tick archive mounted? (c) Replay worker pool saturated?
9. **mde-quote-api (9091)** — (a) Is Redis `QUOTE:*` responsive? (b) Is the feed-handler snapshot stream delivering updates? (c) Circuit breaker open on downstream?

## 3. Alerts + response

Each of the 5 alerts below is defined in Prometheus rules shipped with v2 (source: Confluence 697008500 §4.4). No other v2-specific alerts exist yet — if you are paged on something else, it is a v1 alert (legacy charts-writer, symbol-svc) and unrelated.

### 3.1 `MdeCandleWriterRedisLagHigh`

**PromQL**: `redis_flush_lag_seconds > 5 for 2m`

**Plain English**: The candle-writer accepts ticks from Kafka and flushes aggregated 1-minute candles into Redis every ~1s. Lag > 5s means Redis flushes are backing up, so running-candle reads from charts-api are stale.

**Triage**:

1. Check candle-writer pod CPU + GC — `kubectl top pod -l app=mde-candle-writer`. If CPU maxed, Kafka throughput exceeds the writer's capacity.
2. Check Redis cluster: `redis-cli -h $REDIS_HOST CLUSTER INFO` and `INFO stats`. Slow commands or evictions will stall flushes.
3. Check Kafka lag on group `mde-candle-writer-v2`: `kafka-consumer-groups.sh --bootstrap-server $KAFKA_BROKERS --group mde-candle-writer-v2 --describe`.

**Mitigation**:

- If writer saturated: `kubectl scale deploy/mde-candle-writer --replicas=+2`.
- If Redis unhealthy: failover to read-replica writes via `MDE_CANDLE_REDIS_FAILOVER=true` and page Redis on-call.
- If not recovering in 10 min: execute rollback in Section 5.

---

### 3.2 `MdeCandleWriterClickHouseLagHigh`

**PromQL**: `clickhouse_flush_lag_seconds > 120 for 5m`

**Plain English**: Candle-writer also persists 1m candles to ClickHouse in 60s batches for the historical store. Lag > 2min means batches are stuck or CH is rejecting inserts. Charts-api historical reads will be stale.

**Triage**:

1. Check ClickHouse ingest latency: `clickhouse-client --host $CH_HOST -q "SELECT max(event_time) FROM candles_1m"` — compare to `now()`.
2. Check CH MergeTree pending parts: `SELECT table, count() FROM system.parts WHERE active AND table='candles_1m' GROUP BY table`. If > 300, merges are lagging.
3. Tail candle-writer logs for CH timeout / batch-reject errors: `kubectl logs -l app=mde-candle-writer --tail=200 | grep -i clickhouse`.

**Mitigation**:

- Increase batch flush: `MDE_CANDLE_CH_BATCH_INTERVAL_MS=30000` (more frequent, smaller batches).
- If CH is the bottleneck: page data-platform on-call; candles stay in Redis so live charts are unaffected — this is a historical-read issue only.
- Temporarily force charts-api to Redis: `MDE_CHARTS_API_DEFAULT_SOURCE=redis` (see Section 5).

---

### 3.3 `MdeRunningCandlesStale`

**PromQL**: `time() - last_redis_flush_ts > 10 for 1m`

**Plain English**: No Redis flush has happened in the last 10s during market hours. Candle-writer is either crashed, not consuming Kafka, or Redis is unreachable. Customers see frozen live charts.

**Triage**:

1. Is candle-writer pod `Running`? `kubectl get pod -l app=mde-candle-writer -o wide`. If `CrashLoopBackOff`, grab logs: `kubectl logs -l app=mde-candle-writer --previous --tail=100`.
2. Kafka consumer group lag and liveness: `kafka-consumer-groups.sh --bootstrap-server $KAFKA_BROKERS --group mde-candle-writer-v2 --describe` — check `CONSUMER-ID` column is populated.
3. Redis reachable from the pod? `kubectl exec deploy/mde-candle-writer -- redis-cli -h $REDIS_HOST PING`.

**Mitigation**:

- Restart: `kubectl rollout restart deploy/mde-candle-writer` — usually recovers within 30s.
- If markets are open and no recovery in 3min: execute rollback in Section 5 (revive v1 charts-writer).

---

### 3.4 `MdeRedisCandleMemoryHigh`

**PromQL**: `redis_memory_used_bytes / redis_memory_max_bytes > 0.80 for 10m`

**Plain English**: Redis is > 80% full. TTL on `CANDLE:*` keys is 48h; if memory creeps up it's likely (a) TTLs not being applied, (b) symbol universe grew, or (c) a bad deploy double-wrote candles.

**Triage**:

1. Check key count trend: `redis-cli -h $REDIS_HOST --scan --pattern 'CANDLE:*' | wc -l` vs yesterday's baseline (expect ~symbol_count × intervals × 2 days).
2. Sample TTLs: `redis-cli -h $REDIS_HOST --scan --pattern 'CANDLE:*' | head -10 | xargs -I{} redis-cli -h $REDIS_HOST TTL {}` — any `-1` means no TTL set (bug).
3. Check Redis maxmemory policy: `redis-cli -h $REDIS_HOST CONFIG GET maxmemory-policy` — should be `volatile-lru`.

**Mitigation**:

- If TTLs missing: patch candle-writer (likely a recent regression) and backfill: `redis-cli -h $REDIS_HOST --scan --pattern 'CANDLE:*' | xargs -I{} redis-cli -h $REDIS_HOST EXPIRE {} 172800`.
- If symbol universe grew and it's legitimate: request Redis memory bump from platform-ops.
- If > 90%: emergency evict stale keys by pattern; do not let Redis OOM (writer will crash).

---

### 3.5 `MdeChartsApiSourceErrorSpike`

**PromQL**: `rate(mde_charts_api_source_errors_total[1m]) > 5`

**Plain English**: Charts-api is throwing errors when reading from its configured source (Redis or ClickHouse). Customers see chart load failures.

**Triage**:

1. Which source? Check `MDE_CHARTS_API_DEFAULT_SOURCE` env on the pod and grep logs for `source=redis|clickhouse error=`.
2. Is the source itself up? Run `redis-cli PING` / `clickhouse-client -q "SELECT 1"`.
3. Is it a specific endpoint? `kubectl logs -l app=mde-charts-api --tail=500 | grep ERROR | awk '{print $NF}' | sort | uniq -c` — group errors by URI.

**Mitigation**:

- If Redis is flaky: `MDE_CHARTS_API_DEFAULT_SOURCE=clickhouse` and `kubectl rollout restart deploy/mde-charts-api`.
- If ClickHouse is flaky: `MDE_CHARTS_API_DEFAULT_SOURCE=redis` (limits reads to last 48h but keeps customers served).
- If both degraded: page data-platform + Redis on-call, then execute Section 5 rollback to v1.

## 4. Common commands

### Redis

```bash
# Probe candle keys
redis-cli -h $REDIS_HOST --scan --pattern 'CANDLE:*' | head -20

# Inspect a sample hash (running candle)
redis-cli -h $REDIS_HOST HGETALL 'CANDLE:NSE_EQ:RELIANCE:1m:running'

# Check TTL
redis-cli -h $REDIS_HOST TTL 'CANDLE:NSE_EQ:RELIANCE:1m:2025-04-20T09:15'

# Memory snapshot
redis-cli -h $REDIS_HOST INFO memory | grep -E 'used_memory_human|maxmemory_human'
```

### ClickHouse

```bash
# Row count for today
clickhouse-client --host $CH_HOST -q \
  "SELECT count() FROM candles_1m WHERE event_date = today()"

# Materialized-view lag (seconds behind now)
clickhouse-client --host $CH_HOST -q \
  "SELECT now() - max(event_time) AS lag_s FROM candles_1m"

# Inspect pending parts
clickhouse-client --host $CH_HOST -q \
  "SELECT table, count() FROM system.parts WHERE active GROUP BY table ORDER BY 2 DESC"
```

### Kafka

```bash
# Consumer-group lag for candle-writer
kafka-consumer-groups.sh --bootstrap-server $KAFKA_BROKERS \
  --group mde-candle-writer-v2 --describe

# All MDE consumer groups
kafka-consumer-groups.sh --bootstrap-server $KAFKA_BROKERS --list | grep ^mde-

# Peek at a topic
kafka-console-consumer.sh --bootstrap-server $KAFKA_BROKERS \
  --topic mde.ticks.nse-eq --max-messages 5 --from-beginning
```

### Temporal

```bash
# List schedules (local dev)
temporal schedule list -n default --address 172.18.0.11:7233

kubectl -n temporal port-forward svc/temporal-frontend 7233:7233 &

```

## 5. Rollback recipe

v1 writers (`charts-writer`, `charts-historical-writer-v1`) were intentionally left on `main` branch for exactly this purpose. v2 can be disabled without a redeploy.

### Tier 1 — soft switch (seconds)

Force charts-api to read from the known-good source:

```bash
# If Redis is misbehaving — force ClickHouse reads
kubectl set env deploy/mde-charts-api MDE_CHARTS_API_DEFAULT_SOURCE=clickhouse
kubectl rollout status deploy/mde-charts-api

# If ClickHouse is misbehaving — force Redis reads
kubectl set env deploy/mde-charts-api MDE_CHARTS_API_DEFAULT_SOURCE=redis
kubectl rollout status deploy/mde-charts-api
```

### Tier 2 — disable v2 writers, revive v1 (5–10 min)

```bash
# 1. Stop v2 candle-writer (stops double-writing)
kubectl scale deploy/mde-candle-writer --replicas=0

# 2. Revive v1 writers (still on main image)
kubectl scale deploy/mde-charts-writer-v1 --replicas=3
kubectl scale deploy/mde-charts-historical-writer-v1 --replicas=2

# 3. Wait for v1 writers to catch up Kafka offsets
kafka-consumer-groups.sh --bootstrap-server $KAFKA_BROKERS \
  --group mde-charts-writer-v1 --describe

# 4. Point charts-api at v1 data path
kubectl set env deploy/mde-charts-api MDE_CHARTS_API_DATA_PATH=v1
kubectl rollout restart deploy/mde-charts-api
```

### Tier 3 — ArgoCD rollback (last resort)

1. Open ArgoCD → app `mde-candle-writer` → **History and Rollback**.
2. Pick the previous successful sync (image SHA from last known-good deploy). v2 baseline SHAs are in the shipped-SHAs callout at the top of this page.
3. Click **Rollback** → **Sync**.
4. Repeat for `mde-charts-api` if needed.
5. Notify `#mde-oncall` with Jira incident ID and the restored SHA.
