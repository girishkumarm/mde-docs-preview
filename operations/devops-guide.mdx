# MDE — DevOps Operating Guide (Staging)

> **v3 — consolidated (2026-04-22)** — MDE consolidated from 15 to 10 deployable services. TimescaleDB completely removed. ClickHouse is the sole time-series store (ReplacingMergeTree + 4 MV rollups). Two Redis clusters: C1 (real-time snapshots, RDB persistence) and C2 (candles + movers + FNO + alerts, AOF persistence). All services built with Jib (no Dockerfile), base image Eclipse Temurin 21-jre-alpine.
>
> **Service consolidation merges:**
> - `charts-writer` + `charts-historical-writer` + `charts-api` merged into `charts-service`
> - `recon-service` + `incident-manager` merged into `ops-service`
> - `premarket-scheduler` absorbed into `scheduler-service`
> - `company-fundamentals` absorbed into `instrument-master`
>
> DevOps can use this guide as the provisioning target for staging. Production deploy still gated by readiness checklist (Confluence 698056921).

**Scope**: Staging environment on EKS `ap-south-1`. Minimum viable sizing, single-AZ where safe.
**Last updated**: 2026-04-22
**Owner**: [girish3.kumar@paytm.com](mailto:girish3.kumar@paytm.com)

## 3. Database strategies

### 3.1 PostgreSQL (RDS)

- Instance: RDS PostgreSQL 15, `db.r6g.large`.
- Databases: `instrument_master` (instruments + fundamentals), `alert_engine`, `scheduler_service` (premarket + corporate actions), `ops_service` (incidents + recon), `temporal` -- one role per DB.
- Flyway migrations run in-service on boot.
- Backups: automated daily snapshots, 7-day retention.
- Max connections: 100.

### 3.2 Kafka (MSK)

- MSK cluster: 3 brokers, `m5.large`.
- 8 topics to pre-create:

| Topic | Partitions | RF | Retention | Cleanup |
| --- | --- | --- | --- | --- |
| `pml-marketfeed-nse-equity-ticks` | 6 | 3 | 7 days | delete |
| `pml-marketfeed-nse-fno-ticks` | 12 | 3 | 7 days | delete |
| `pml-marketfeed-nse-index-ticks` | 3 | 3 | 7 days | delete |
| `pml-marketfeed-bse-equity-ticks` | 3 | 3 | 7 days | delete |
| `mde-candles-1m` | 6 | 3 | 30 days | delete |
| `mde-instrument-master-events` | 3 | 3 | 7 days | compact |
| `mde-alert-events` | 3 | 3 | 7 days | delete |
| `mde-status-events` | 3 | 3 | 7 days | compact |

### 3.3 ClickHouse

- Deployment: ClickHouse Keeper (3-node cluster) or managed ClickHouse Cloud.
- Storage: EBS gp3, 500GB initial.
- Database: `mde`.
- Tables:
  - `candles_1m` -- ReplacingMergeTree, partitioned by `toYYYYMMDD(timestamp)`.
  - 4 materialized view rollup tables: `candles_5m`, `candles_15m`, `candles_1h`, `candles_1d`.
- Materialized views auto-populate the rollup tables from `candles_1m`.
- Retention: `candles_1m` = 2 years, rollups (`5m`, `15m`, `1h`, `1d`) = 5 years.
- Backup: daily S3 backup via `clickhouse-backup` to `mde-staging-backups/clickhouse/` (14-day retention).

### 3.4 Redis Cluster 1 (C1) — Real-time Snapshots

- Instance: 3 masters, 3 replicas, `cache.r6g.large`, maxmemory 8GB.
- `maxmemory-policy volatile-lru`.
- Persistence: RDB snapshots every 5 min.
- Key patterns:

| Pattern | TTL | Description |
| --- | --- | --- |
| `FEED_PP_{exchange}_{segment}:{pmlId}` | 8 h | Feed price packet |
| `OHLC_{exchange}_{segment}:{pmlId}` | 8 h | OHLC data |
| `PC_{exchange}_{segment}:{pmlId}` | 8 h | Previous close |
| `LC_UC_{exchange}_{segment}:{pmlId}` | 8 h | Circuit limits |
| `52WK_{exchange}_{segment}:{pmlId}` | 24 h | 52-week high/low |
| `DEPTH_{exchange}_{segment}:{pmlId}` | 8 h | Market depth |
| `MKT_STATUS:{exchange}:{segment}` | 8 h | Market status |

- Daily snapshot to S3 (ElastiCache automatic, 7-day retention).

### 3.5 Redis Cluster 2 (C2) — Candles + Movers + FNO + Alerts

- Instance: 3 masters, 3 replicas, `cache.r6g.xlarge`, maxmemory 16GB.
- `maxmemory-policy noeviction` — writers trim ZSETs explicitly.
- Persistence: AOF (`appendfsync everysec`) — critical for candle durability.
- **Why AOF on C2**: 1-minute candles are the hot data store and must survive restarts without data loss. RDB snapshots (used on C1) can lose up to 5 minutes of data. AOF with everysec provides at most 1 second of data loss.
- Key patterns:

| Pattern | TTL | Description |
| --- | --- | --- |
| `candle:1m:{token}:{epoch_min}` | 3 h | 1-minute candle hash |
| `candle:idx:{token}` | 3 h | Candle ZSET index |
| `MOVERS:{category}:{exchange}:{segment}` | 8 h | Market movers ZSET |
| `FNO:{type}:{exchange}:{expiry}` | 8 h | FNO analytics |
| `ALERT:IDX:{userId}` | no TTL | Alert Redis index |
| `BOD:GUARD:{date}` | 24 h | BOD preload guard key |

### 3.6 Temporal

- Schedules: `im-daily-sync`, `bod-preload-daily`, `recon-post-market-daily`, `corporate-action-precompute`.
- Persistence: `temporal` DB on the shared RDS instance.
- Namespace: `default`.
- Task queues: `scheduler-queue` (BOD/premarket), `ops-queue` (recon/incidents).

## 6. Observability

### 6.1 Prometheus

- Scrape interval 15s, retention 7d, 30 GB PVC.
- All 10 services expose metrics via Micrometer on management port (9xxx).
- Metric naming convention: `mde_{service}_{metric}` (e.g., `mde_feed_handler_packets_received_total`, `mde_charts_service_clickhouse_insert_latency`).

### 6.2 Grafana Dashboards (8 dashboards)

| # | Dashboard | Key Panels |
| --- | --- | --- |
| 1 | **MDE Overview** (single pane of glass) | 10 service health status, key TPS gauges, data flow diagram, error rate heatmap |
| 2 | **Feed Handler & Publisher** | Packets/sec by exchange, decode errors, LZO decompression latency, Redis C1 write latency, dedup ratio |
| 3 | **Charts Pipeline** | Candles/min aggregated, ClickHouse batch insert rate, API query latency (p50/p95/p99), cache hit ratio |
| 4 | **Market Movers & FNO** | Signals/sec, ZSET writes/sec, Greeks compute duration, IV solver iterations, max pain calculation time |
| 5 | **Broadcast Service** | WebSocket active connections, messages/sec, slow consumer evictions, Kafka consumer lag, HPA replica count |
| 6 | **Alerts & Operations** | Active alerts gauge, alerts triggered/min, recon mismatches, auto-heal actions taken, incident MTTR |
| 7 | **Infrastructure** | Kafka broker health + consumer lag, Redis C1/C2 memory + ops/sec, ClickHouse query latency + merge rate, PostgreSQL connections + slow queries, JVM heap + GC metrics |
| 8 | **Business KPIs** | Live instrument count, data freshness (last tick age), BOD completeness percentage, market hours timeline, candle coverage percentage |

### 6.3 Distributed Tracing

- Micrometer Tracing + Brave/Zipkin propagation across all services.
- Kafka message headers carry `traceId` for log correlation.
- Trace context propagated through Redis pub/sub and Temporal workflows.

### 6.4 Structured Logging

- Format: JSON via Logstash Logback encoder.
- MDC fields: `service`, `traceId`, `spanId`, `instrumentToken`.
- Separate marker-based appenders for key packet types (TRADE, INDEX, OHLC, MBP, MKT_STATUS).
- Loki: 7-day retention, 20 GB PVC.

### 6.5 Alerting

- AlertManager routes: Slack `#mde-staging-alerts` for warnings; webhook -> ops-service for critical.
