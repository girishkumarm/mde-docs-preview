<Info>
**Two Redis clusters** (Decision 2). **Cluster 1 (C1)** = port **6380** — real-time snapshots, hot-path writes from the tick pipeline. **Cluster 2 (C2)** = port **6381** — candles, movers, FNO, alerts. Both are Redis 7.4 in local compose; ElastiCache in staging/prod.
</Info>

## Cluster 1 (C1) — Real-time Snapshots (port 6380)

Written primarily by `publisher` and `scheduler-service`; read by `broadcast-service`, `feed-handler`, `charts-service`, `alert-engine`, and `market-movers-writer`.

| Key pattern | Type | Writer | Readers | TTL | Description |
| --- | --- | --- | --- | --- | --- |
| `FEED_PP:{token}` | Hash | publisher | broadcast-service, feed-handler (zero-LTP fallback) | None | Last tick snapshot: ltp, open, high, low, close, volume, oi, chg, chgPct, bid/ask depth |
| `OHLC:{token}` | Hash | publisher | charts-service (seed), broadcast-service | None | Intraday OHLC: open, high, low, close, volume |
| `PC:{token}` | Hash | scheduler-service (BOD) | publisher, alert-engine, market-movers-writer | None | Previous close: pc, prevOpen, prevHigh, prevLow, prevVol |
| `LC_UC:{token}` | Hash | scheduler-service (BOD) + publisher (BSE intraday) | alert-engine, broadcast-service | None | Circuit limits: uc (upper), lc (lower) |
| `52WK:{token}` | Hash | scheduler-service (BOD) + publisher (Lua CAS) | alert-engine | None | 52-week range: high, low, highDate, lowDate |
| `DEPTH:{token}` | Hash | publisher | broadcast-service | None | Market depth: 5 bid/ask levels with price, qty, orders |
| `MKT_STATUS:{exchange}` | String/Hash | publisher | broadcast-service, ops-service | None | Market status: status, session, updatedAt |

### Atomic operations (Lua)

`publisher` uses Lua scripts for compare-and-set on hot-path updates. Scripts live in `services/publisher/src/main/resources/lua/`.

- **52-week H/L**: `SCRIPT.EVAL` with `token`, `price`, current hi/lo — atomic check + replace
- **OHLC compare-and-set**: maintains O/H/L/C consistency under concurrent ticks

Server-side execution guarantees single-roundtrip atomicity.

## Cluster 2 (C2) — Candles, Movers, FNO, Alerts (port 6381)

Written by `charts-service`, `market-movers-writer`, `fno-data-writer`, `alert-engine`, and `scheduler-service`. C2 has AOF enabled for durability.

| Key pattern | Type | Writer | Readers | TTL | Description |
| --- | --- | --- | --- | --- | --- |
| `candle:1m:{token}:{epochMinute}` | Hash | charts-service | charts-service (API) | 2–3 days | 1m candle: open, high, low, close, volume, oi, trades |
| `candle:idx:{token}` | ZSET | charts-service | charts-service (API) | 2–3 days | Candle time index: member=epochMinute, score=epochMinute |
| `MOVERS:{category}:{index}` | ZSET | market-movers-writer | market-movers-writer (API) | 1 day | Top movers: member=token, score=pctChange/volume |
| `FNO:CHAIN:{underlying}:{expiry}` | Hash | fno-data-writer | fno-data-writer (API) | 1 day | Option chain: field=strikeType (e.g., "22500CE"), value=JSON |
| `FNO:OI_RANK:{underlying}:{expiry}` | ZSET | fno-data-writer | fno-data-writer (API) | 1 day | OI ranking: member=strike, score=OI |
| `FNO:MAX_PAIN:{underlying}:{expiry}` | Hash | fno-data-writer | fno-data-writer (API) | 1 day | Max pain: strike, totalPain, updatedAt |
| `FNO:ATM_STRADDLE:{underlying}:{expiry}` | Hash | fno-data-writer | fno-data-writer (API) | 1 day | ATM straddle: strike, cePrice, pePrice, premium, ceIV, peIV |
| `FNO:PCR:{underlying}:{expiry}` | Hash | fno-data-writer | fno-data-writer (API) | 1 day | Put-call ratio: pcrOI, pcrVol, totalCallOI, totalPutOI |
| `FNO:BAN:{token}` | Hash | scheduler-service | fno-data-writer, broadcast-service | 1 day | Ban list: banned, reason, date |
| `ALERT:IDX:{token}` | Set | alert-engine | alert-engine | None | Alert index: set of alertIds for token |
| `BOD:GUARD:{activity}:{date}` | String | scheduler-service | scheduler-service | 2 days | Idempotency guard: prevents re-running BOD activity |

### Three-layer candle durability (Decision 7)

Hot 1m candles live in Redis C2 with AOF persistence and 2–3 day TTL. The three durability layers are:

1. **Redis C2** — hot path, sub-ms reads for live chart queries
2. **ClickHouse** — charts-service historical writer thread flushes closed candles to ClickHouse within minutes
3. **Kafka `mde-candles-1m` topic** — retention-based backup; can replay to rebuild both Redis and ClickHouse

## Connection pools

- **Lettuce 6.4.0** for all services (reactive + blocking modes).
- `spring.data.redis.lettuce.pool.max-active: 16` per pod.
- Command timeout: 2 s. Resilience4j wraps every call with a circuit breaker (Decision 12).

## Why two clusters?

One bad query from an API consumer should not slow down tick ingestion. The clusters isolate:

- **Write amplification** on C1 — every tick fans out to multiple hash updates from publisher.
- **Query bursts** on C2 — e.g., a user loading a big option chain, 30-day chart, or movers leaderboard.

See **[Decision 2](/architecture/diagrams#d2-redis-split)** for the full rationale.

## Running locally

```bash
make up

# C1 — Real-time Snapshots
redis-cli -h localhost -p 6380

# C2 — Candles, Movers, FNO, Alerts
redis-cli -h localhost -p 6381

# full reset
make wipe && make up
```

## Observability

- Prometheus + `redis_exporter` scrapes both clusters; Grafana dashboard `infra/grafana/redis.json` shows memory, eviction rate, command latency.
- Per Decision 12, any Redis timeout falls back to:
  1. Stale cache value if present
  2. Synchronous ClickHouse read (charts-service only)
  3. Empty response with `X-Fallback-Layer` header for observability
