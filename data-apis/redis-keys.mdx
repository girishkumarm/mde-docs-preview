<Info>
**Two Redis clusters** (Decision 2). **Feed (Cluster 1)** = port **6380** — hot-path writes from the tick pipeline. **API (Cluster 2)** = port **6381** — read-heavy cache for REST APIs. Both are Redis 7.4 in local compose; ElastiCache in staging/prod. Patterns below were re-verified against the code.
</Info>

## Cluster 1 — Feed (port 6380)

Hot-path writes. Most writes come from `publisher` and `candle-writer`; reads come from `broadcast-service`, `fno-data-writer`, `charts-api`, and the movers/api services (for cold-start snapshots).

| Key pattern | Type | Writers | Readers | TTL | Notes |
| --- | --- | --- | --- | --- | --- |
| `FEED_PP_{segment}:{pmlId}` | STRING (Protobuf bytes) | publisher | broadcast-service, fno-data-writer, charts-api (cold-start snapshot) | — | Serialized tick snapshot per exchange segment + PML ID |
| `MGR_{exchange}_{segment}_OHLC:{pmlId}` | HASH | publisher | — | — | Day OHLC snapshot maintained via Lua compare-and-set |
| `MGR_{exchange}_{segment}_52_WK_HL:{pmlId}` | HASH | publisher, premarket-scheduler | market-movers-writer | — | 52-week high/low, Lua atomic replace |
| `MGR_{exchange}_{segment}_PC:{pmlId}` | STRING | premarket-scheduler | market-movers-writer | — | Previous close (loaded at pre-market) |
| `MGR_{exchange}_{segment}_LC_UC:{pmlId}` | HASH | instrument-master | market-movers-writer | — | Circuit limits (lower / upper) |
| `CANDLE:{interval}:{token}` | HASH | candle-writer (v2), charts-writer *(legacy)* | charts-api (`source=redis`), broadcast-service | — | Closed interval candle state |
| `CANDLE:RUN:{interval}:{token}` | HASH | candle-writer | charts-api `/v1/running-candles` | **120 s** | Running (in-progress) 5s/15s/30s/current-1m bucket |
| `TRD_DAY` | STRING (boolean flag) | publisher, premarket-scheduler | all services (via `@TradeDay` / `ConditionalSchedulingAspect`) | — | Is today a trading day? Gates every pre-market scheduled job |

## Cluster 2 — API (port 6381)

Read-heavy. Written by writers in bulk on rollover; read by REST APIs.

| Key pattern | Type | Writers | Readers | TTL | Notes |
| --- | --- | --- | --- | --- | --- |
| `candle:1m:{token}` | STRING (JSON) | charts-api *(cache-aside)* | charts-api query path | **60 s** intraday, **30 d** historical | Cache of ClickHouse query results |
| `fno:chain:{underlying}:{expiry}:{strike}:{CE\|PE}` | HASH | fno-data-writer | fno-data-writer `/v1/fno-chain` | — | Aggregated Greeks, IV, OI, volume |
| `movers:{segment}:{window}` | ZSET | market-movers-writer | market-movers-writer `/v1/movers` | — | Leaderboard zsets per segment + window (1m / 5m / 1h / day) |

## Atomic operations (Lua)

`publisher` uses Lua scripts for compare-and-set on two hot-path updates. Scripts live in `services/publisher/src/main/resources/lua/`.

- **52-week H/L**: `SCRIPT.EVAL` with `token`, `price`, current hi/lo — atomic check + replace
- **OHLC compare-and-set**: maintains O/H/L/C consistency under concurrent ticks

Server-side execution guarantees single-roundtrip atomicity.

## Connection pools

- **Lettuce 6.4.0** for all services (reactive + blocking modes).
- `spring.data.redis.lettuce.pool.max-active: 16` per pod.
- Command timeout: 2 s. Resilience4j wraps every call with a circuit breaker (Decision 12).

## Why two clusters?

One bad query from an API consumer shouldn't slow down tick ingestion. The clusters isolate:

- **Write amplification** on Feed — every tick fans out to multiple hash updates.
- **Query bursts** on API — e.g., a user loading a big option chain or 30-day chart.

See **[Decision 2](/architecture/diagrams#d2-redis-split)** for the full rationale.

## Running locally

```bash
make up

# Feed cluster
redis-cli -h localhost -p 6380

# API cluster
redis-cli -h localhost -p 6381

# full reset
make wipe && make up
```

## Observability

- Prometheus + `redis_exporter` scrapes both clusters; Grafana dashboard `infra/grafana/redis.json` shows memory, eviction rate, command latency.
- Per Decision 12, any Redis timeout falls back to:
  1. Stale cache value if present
  2. Synchronous ClickHouse read (charts-api only)
  3. Empty response with `X-Fallback-Layer` header for observability
