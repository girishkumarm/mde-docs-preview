## Temporal Workflows (scheduler-service)

All workflows run inside `scheduler-service`. The Temporal worker connects to the self-hosted Temporal server and polls the task queues below.

| Workflow | Task Queue | Schedule | Description |
| --- | --- | --- | --- |
| `BodPreloadWorkflow` | `premarket-scheduler-jobs` | 06:00 IST | 5 activities: prevClose, OHLC seed, 52-week calc, circuit limits, stamp trading day |
| `FnoBodWorkflow` | `premarket-scheduler-jobs` | 06:30 IST | F&O bhavcopy parse, OI reference data, ban list |
| `PreComputeAdjustmentWorkflow` | `corporate-action-jobs` | 19:30 IST | T-1 corporate action pre-computation: read history, compute adjusted prices, write shadow tables |
| `ApplyAdjustmentWorkflow` | `corporate-action-jobs` | 00:01 IST (on ex-date) | Atomic partition swap: shadow to main in ClickHouse, update Redis BOD keys |
| `VerifyAdjustmentWorkflow` | `corporate-action-jobs` | 06:00 IST (on ex-date) | Post-flip verification: sample candles, validate adjustment, auto-rollback if bad |
| `ReconWorkflow` | `ops-jobs` | 16:00 IST | Cross-validate Redis vs ClickHouse vs Kafka offsets |

### Idempotency

Every BOD workflow uses the `BOD:GUARD:{activity}:{date}` Redis key (TTL 2 days) to prevent duplicate execution. If the guard key exists, the activity short-circuits with a logged skip.

### Creating a new workflow

```java
// 1. Define the workflow interface
@WorkflowInterface
public interface MyCustomWorkflow {
    @WorkflowMethod
    void execute(MyInput input);
}

// 2. Implement the workflow
public class MyCustomWorkflowImpl implements MyCustomWorkflow {
    private final MyActivity activity = Workflow.newActivityStub(MyActivity.class,
        ActivityOptions.newBuilder()
            .setStartToCloseTimeout(Duration.ofMinutes(5))
            .setRetryOptions(RetryOptions.newBuilder().setMaximumAttempts(3).build())
            .build());

    @Override
    public void execute(MyInput input) {
        activity.step1(input);
        activity.step2(input);
    }
}

// 3. Register in the worker
worker.registerWorkflowImplementationTypes(MyCustomWorkflowImpl.class);
worker.registerActivitiesImplementations(new MyActivityImpl());

// 4. Schedule via cron (or trigger ad-hoc via Temporal dashboard)
WorkflowOptions options = WorkflowOptions.newBuilder()
    .setTaskQueue("my-team-queue")
    .setCronSchedule("0 9 * * *")  // daily 9 AM
    .build();
```

## Architectural Decisions

This section tracks the defining decisions behind the current design.

### Decision 1 — Protobuf for tick serialization

- Raw tick data uses **Protocol Buffers** (`Tick.proto`) for bandwidth efficiency at 50K+ ticks/sec.
- Compact binary format reduces Kafka partition sizes and network bandwidth by ~60% vs JSON.
- Schema evolution via proto field numbering — new fields are backward-compatible.

### Decision 2 — Two Redis clusters for workload isolation

- **Cluster 1 (C1)**: Real-time snapshots — hot-path writes from publisher, reads by broadcast-service and feed-handler.
- **Cluster 2 (C2)**: Candles, movers, FNO derivatives, alerts — read-heavy, write-on-rollover pattern.
- Isolation prevents API query bursts (option chain, 30-day charts) from impacting tick ingestion latency.

### Decision 3 — Kafka for inter-service messaging

- All inter-service data flow goes through Kafka topics (ticks, candles, alerts, market status).
- `acks=all` + idempotent producer guarantees at-least-once delivery end-to-end.
- Partition-by-token ensures strict per-instrument ordering.

### Decision 4 — Temporal for workflow orchestration

- All scheduled jobs (BOD preload, corporate actions, recon) run as Temporal workflows.
- Built-in retry, timeout, visibility, and manual intervention via Temporal dashboard.
- Idempotent activities with Redis guard keys prevent duplicate execution.

### Decision 5 — Virtual threads for all services

- Java 21+ virtual threads (`-Djdk.virtualThreadScheduler`) for all 10 services.
- Eliminates thread-pool tuning — each Kafka consumer, Redis call, and HTTP handler gets a lightweight virtual thread.
- Measured 3x throughput improvement on publisher service under load testing.

### Decision 6 — ClickHouse for historical candles

- **Replaces TimescaleDB.** ClickHouse with `ReplacingMergeTree` engine and 4 materialized views for interval rollups (5m, 15m, 1h, 1d).
- Continuous aggregation from `candles_1m` base table eliminates manual rollup services.
- Column-oriented storage gives 10x compression ratio on OHLCV data and sub-second query performance on multi-year ranges.
- `1w` and `1M` intervals computed from `candles_1d` materialized view.

### Decision 7 — Redis for hot 1m candles (three-layer durability)

- Hot 1m candles live in **Redis C2 with AOF persistence** and 2–3 day TTL.
- Three durability layers: Redis C2 (hot reads) -> ClickHouse (historical writer flushes within minutes) -> Kafka `mde-candles-1m` topic (retention-based backup, replay to rebuild).
- Intraday chart queries hit Redis first (sub-ms), falling back to ClickHouse only for data older than TTL.

### Decision 8 — Black-76 for index options

- Index options use **Black-76** pricing model (futures-settled), separate from **Black-Scholes-Merton (BSM)** for equity options (spot-settled).
- IV calculation in `fno-data-writer` selects model based on underlying type at runtime.
- Prevents systematic IV errors that occur when applying BSM to cash-settled index options.

### Decision 9 — Market movers from 1m candles, not raw ticks

- `market-movers-writer` consumes `mde-candles-1m` (~120 events/min) instead of raw tick topics (~50K events/sec).
- Reduces CPU and memory by ~400x while providing the same end-user experience (movers update every minute).
- Leaderboard ZSETs in Redis C2 are updated on each candle close, not on every tick.

### Decision 10 — Corporate action pre-computation at T-1

- Corporate actions (splits, bonus, consolidation) are pre-computed at T-1 evening via `PreComputeAdjustmentWorkflow`.
- Shadow tables in ClickHouse hold adjusted candle data; `ApplyAdjustmentWorkflow` performs atomic partition swap at 00:01 IST on ex-date.
- `VerifyAdjustmentWorkflow` at 06:00 IST samples candles and auto-rolls back if verification fails.
- Eliminates real-time adjustment complexity during market hours.

### Decision 11 — Autonomous ops hub with market-hours awareness

- `ops-service` provides centralized incident management, recon orchestration, and health monitoring.
- **Auto-remediation only outside market hours** (POST_CLOSE, CLOSED) — during OPEN and PRE_OPEN, ops-service escalates to humans with pre-assembled context.
- Remediation actions: `RESTART_POD`, `REBUILD_CACHE`, `BACKFILL_CANDLES`, `SCALE_UP`, `CLEAR_DEAD_CONNECTIONS`.
- Market-hours awareness prevents automated actions from causing data disruption during live trading.

### Decision 12 — Zero-error user experience via graceful degradation

- **Users should NEVER see an error.** This is a hard product rule, not a goal.
- **Layered fallback for every user-facing data path:** Primary (live) -> Cached (Redis C2) -> Stale (ClickHouse) -> Final (skeleton/empty UI). Each downgrade is silent; staleness shown as a subtle freshness indicator, never as an error.
- **Stale-while-revalidate by default.** APIs return within 50 ms with whatever they have; background tasks refresh upstream.
- **Circuit breakers on every downstream call.** Resilience4j-wrapped, per-service thresholds. Open circuits short-circuit to fallback.
- **Convert all 5xx to 200 OK with `data: null, stale: true`.** Frontend treats null data as a loading state, never as an error.
