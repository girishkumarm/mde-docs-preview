## Tick topics (from exchange feeds)

**Producer**: `feed-handler`. Ticks are partitioned by `instrumentToken` hash so all ticks for a symbol land on the same partition (strict per-symbol ordering). Serialization: **Protobuf** (`Tick.proto`).

| Topic | Partitions | Producer | Consumers | Notes |
| --- | --- | --- | --- | --- |
| `mde-ticks-nse-equity` | 12 | feed-handler | publisher, charts-service, alert-engine, broadcast-service | NSE equity segment |
| `mde-ticks-bse-equity` | 6 | feed-handler | publisher, charts-service, broadcast-service | BSE equity segment |
| `mde-ticks-nse-derivative` | 24 | feed-handler | publisher, charts-service, fno-data-writer, broadcast-service | NSE F&O — highest partition count for options volume |
| `mde-ticks-bse-derivative` | 6 | feed-handler | publisher, charts-service, fno-data-writer, broadcast-service | BSE F&O |
| `mde-ticks-nse-index` | 3 | feed-handler | publisher, broadcast-service | NSE indices |
| `mde-ticks-bse-index` | 3 | feed-handler | publisher, broadcast-service | BSE indices |

## Candle topics

**Producer**: `charts-service`. Serialization: **Protobuf** (`Candle`).

| Topic | Partitions | Producer | Consumers | Notes |
| --- | --- | --- | --- | --- |
| `mde-candles-1m` | 12 | charts-service | market-movers-writer | Closed 1m candles. market-movers-writer consumes candles NOT raw ticks (~120/min vs 50K/sec — see Decision 9). charts-service flushes closed candles to ClickHouse internally — it does NOT consume its own topic. |

## Delivery guarantees

Per architecture Decision 3 (`acks=all` + Idempotent Producer):

```yaml
spring.kafka.producer:
  acks: all
  enable.idempotence: true
  retries: Integer.MAX_VALUE
  max.in.flight.requests.per.connection: 5
  compression.type: lz4
```

Consumers use `enable.auto.commit: false` and manual offset commit after successful processing. At-least-once guaranteed end-to-end.

## Consumer groups

One group per service, with pod identity when required:

| Service | Group ID | Scope |
| --- | --- | --- |
| broadcast-service | `broadcast-service-${POD_NAME}` | **Unique per pod** — every pod gets every tick for WebSocket fan-out |
| publisher | `mde-publisher` | Shared across publisher pods |
| charts-service | `mde-charts-service` | Shared |
| alert-engine | `mde-alert-engine` | Shared |
| fno-data-writer | `mde-fno-data-writer` | Shared |
| market-movers-writer | `mde-market-movers-writer` | Shared |
| instrument-master | `mde-instrument-master` | Shared (self-loop for cache invalidation) |

<Warning>
**broadcast-service uses UNIQUE consumer group per pod** (`broadcast-service-${POD_NAME}`) so each pod receives ALL messages. A shared group would starve pods and break WebSocket fan-out. All other services use standard consumer groups where partitions are distributed across pods.
</Warning>

## Key design notes

- **Tick topics use Protobuf serialization** (`Tick.proto`) for bandwidth efficiency at 50K+ ticks/sec
- **Candle topics use Protobuf serialization** (`Candle`) — same binary efficiency as ticks for the hot candle pipeline
- **market-movers-writer consumes `mde-candles-1m`** NOT raw tick topics (Decision 9 — ~120 events/min vs 50K/sec)
- **alert-engine only consumes equity ticks** (`mde-ticks-nse-equity`, `mde-ticks-bse-equity`) — alerts are not evaluated against derivative or index ticks
- **charts-service flushes ClickHouse internally** — the historical writer thread is part of charts-service itself, not a Kafka consumer of its own topic

## Local development

`docker-compose.yml` provisions Kafka 3.8.0 (Confluent 7.5.0 image) on port `9094`. The `kafka-init` sidecar creates all topics above at startup. Reset:

```bash
make down && make wipe && make up
```

## Observability

- Each service's `/actuator/prometheus` binds Kafka client metrics (producer send rate, consumer lag, ISR changes).
- Grafana dashboards under `infra/grafana/` chart per-topic consumer lag.
- AlertManager (`infra/alertmanager/`) pages on broadcast-service lag > 500 ms or any consumer stalled > 30 s.
