## Delivery guarantees

Per architecture Decision 4 (**`acks=all` + Idempotent Producer**):

```yaml
spring.kafka.producer:
  acks: all
  enable.idempotence: true
  retries: Integer.MAX_VALUE
  max.in.flight.requests.per.connection: 5
  compression.type: lz4
```

Consumers use `enable.auto.commit: false` and manual offset commit after successful processing, so at-least-once is guaranteed end-to-end.

## Consumer groups

One consumer group per **service**, with instance id in the group name:

- `broadcast-service-${HOSTNAME}` — **unique per pod** (Decision 1 — every broadcast pod receives every tick)
- `candle-writer-default` — shared across candle-writer pods
- `fno-data-writer-default`
- `market-movers-writer-default`
- `charts-historical-writer-default`
- `recon-service-default`

Unique-per-pod group IDs for broadcast-service are essential — a shared group would starve pods and break WebSocket fan-out. See **[Kafka vs Redis](/decisions/kafka-vs-redis)** for the full reasoning.

## Local development

`docker-compose.yml` provisions Kafka 3.8.0 (Confluent 7.5.0 image) on port `9094`. The `kafka-init` sidecar creates all topics above at startup with these defaults. To recreate:

```bash
make down
make wipe
make fresh
```

## Observability

- Prometheus scrapes each service's `/actuator/prometheus` endpoint; Micrometer binds Kafka client metrics automatically.
- Grafana dashboards under `infra/grafana/` chart consumer lag, produce rate, and error rate per topic.
- AlertManager rules in `infra/alertmanager/` trigger P1 when broadcast-service lag > 500 ms or any consumer stalls for > 30 s.
